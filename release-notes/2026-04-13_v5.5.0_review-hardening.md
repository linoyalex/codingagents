## What's New

### Review Layer Hardening (ISS-024, ISS-014, ISS-033)

The framework previously allowed reviewers to trust developer framing without verifying against source specifications. This created blind spots: unverified test claims, hidden assumptions left unchallenged, and no spec traceability in handoffs.

v5.5.0 introduces three complementary layers to harden the review phase:

1. **Reviewer Independence** — explicit methodology for reading PRD first, treating claims as hypotheses to falsify, and tracing schema fields through the validate/transform chain
2. **Adversarial Stance** — gate reviewers now explicitly adopt an adversarial-but-constructive stance: challenge hidden assumptions, verify bypass paths, check stale state, verify trust boundaries
3. **Source Spec Verification** — reviewers load the source specification (PRD or ticket) before reading the diff, and separate-context enforcement prevents implicit trust inheritance from authoring phases

Together these prevent: (1) reviewer trust of unverified developer summary, (2) hidden assumptions going unchallenged, (3) stale state reaching production, and (4) same-agent continuity allowing implicit trust leakage across phase boundaries.

#### Key Changes

**Reviewer Independence methodology added to code-review skill:**
- New section guides reviewers: read PRD before handoff, treat developer claims as hypotheses to falsify, grep adjacent symbols, verify test coverage by reading fixtures, and cross-read PRD vs implementation
- Practical example: when reviewing a parse/validate/transform chain, trace one test case from input → schema → consumer to verify the field makes it through the full pipeline
- Skill remains within size budget (152 lines)

**Adversarial stance enforced in gate roles:**
- Both ROLE_CODE_REVIEWER and ROLE_SECURITY now include explicit adversarial-but-constructive requirements
- Reviewers are prompted to ask: Are there guard failures? Is this state stale? Can this bypass a trust boundary?
- Prompted to flag contradictory artifacts (when PRD says X but implementation does Y)

**Separate context requirement for gate phases:**
- Phase 4 (security gate) and Phase 6 (code review) now require separate context from authoring phases — not just a fresh session
- Enforced via `produced_by` check in role headers: review must be launched in a different agent session, with no carried-over handoff framing beyond the machine-readable fields
- Reviewers must re-derive coverage expectations from the source spec independently, preventing trust inheritance
- Review document headers now include: "Reviewed in separate context from authoring phase" (not just "fresh session")

**Pipeline phase tagging in CLAUDE.md:**
- Phases 1–3 and 5 marked as (authoring) — agents building/implementing
- Phases 4 and 6 marked as (gate/review) — agents auditing/verifying
- Makes the authoring/adversarial distinction visible to agents and eliminates the assumption of trust continuity

**source_spec field required in handoff.json:**
- New mandatory field in handoff schema points reviewers to the source specification
- For features: PRD path (e.g., `docs/features/review-hardening/prd.md`)
- For bugfixes: ticket file (e.g., `docs/issues/tickets/ISS-024.md`) or GitHub issue URL
- Validated by checkpoint.js with three-level defense: schema pattern check, path traversal guard (no `..` segments), and file-existence check

**Source-spec-first prompt injection:**
- review.md and security-gate.md commands now generate prompts that load the source specification first
- Prompt structure: "First read `<source_spec_path>`. Then verify the diff matches the PRD, not the developer summary."
- Reviewers cannot proceed without a resolvable source spec; checkpoint.js halts with explicit error if source_spec is missing or unresolvable

**Separate-context verification tests:**
- New regression suite verifies: (a) adversarial stance enforced in both gate roles, (b) source_spec required in schema, (c) commands reject missing source_spec, (d) review commands require separate context, (e) gate reviewers are read-only (no src/ writes), (f) pipeline phases tagged as authoring vs gate/review

### Bug Fixes

**detectPhase() now recognizes .js/.mjs test files:**
- Helper no longer misclassifies TypeScript/JavaScript test files in tests/node/ as implementation files
- Improves phase detection accuracy for polyglot test suites

## Known Limitations

1. **Separate context depends on agent discipline.** The `produced_by` check is layered mitigation, not a hard gate. If a future agent omits `produced_by`, the role check silently passes. This is intentional design (defense-in-depth) but relies on agent adherence to role requirements.

2. **Source spec validation depends on file existence.** A source_spec path that points to a valid artifact in the wrong location (e.g., a ticket moved after handoff) will fail validation. Agents must use canonical paths and update handoff.json if source locations change.

3. **Reviewer Independence is a methodology, not enforced.** Contract tests verify the methodology is present in the skill; enforcement depends on reviewer discipline to follow the PRD-first methodology.

## Migration & Next Steps

- **For existing target projects:** Upgrade between feature cycles when possible. v5.5.0 introduces a new required handoff field and new gate role requirements; in-flight phases may need handoff updates before re-review.
- **For future code reviews:** Reviewers will now follow Reviewer Independence methodology, load source_spec before diff, and operate in separate context from authoring phases.
- **For future bugfixes:** Handoffs for bugfixes must point source_spec to ticket file (precedence: ticket file > GitHub issue URL > other declared source). Commands will validate the file exists.
