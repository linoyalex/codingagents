## What's New

### Code Review Skill Hardening (ISS-039)

The code-review skill previously lacked four systemic methodology checks that allowed findings to escape during the review-hardening RCA: downstream schema impact tracing, source/installed drift detection, test suite execution, and finding reproduction. This release closes all four gaps and adds symmetric gate enforcement to prevent gate commands from diverging.

#### Key Changes

- **Schema impact tracing (AC1)** — the review methodology now instructs reviewers to grep for all producers and consumers of any schema changed in the diff, tracing fields through the schema → validate → transform chain.

- **Source/installed drift check (AC2)** — Quick Automated Checks now include a command comparing each touched source file (`commands/`, `skills/`, `hooks/`) against its installed copy (`.claude/commands/`, `.claude/skills/`, `.claude/helpers/`) and flagging any divergence. A pre-diff caveat instructs verifying against the committed version, not the working tree.

- **Test suite execution (AC3)** — reviewers now run the project's test suite as part of the review, not just grep patterns. The reviewer determines the appropriate test command from the project's configuration and runs suites covering files touched by the diff.

- **Reproduction requirement (AC4)** — BLOCKING and HIGH findings must be reproduced with actual commands before final severity assignment. Unverifiable findings are marked "unverified" and cannot receive BLOCKING severity without escalation.

- **Symmetric gate enforcement (AC6)** — reviewers must confirm that every gate check (`produced_by`, `source_spec`, `separate context`) exists in both `commands/review.md` and `commands/security-gate.md`. If one gate has a check the other lacks, a HIGH finding is raised. This prevents the gate commands from silently diverging.

- **Structural anchor tests (AC5)** — 64 contract, integration, and E2E tests verify all methodology sections using heading/label anchors, not phrase-binding.
