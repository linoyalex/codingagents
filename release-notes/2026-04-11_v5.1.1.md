## What's New

### Feature Resolution Safety Across All Pipeline Phases (ISS-009)

All pipeline commands (phases 2-7: architect, test-design, security-gate, review) now enforce strict feature resolution with hard-fail semantics instead of lenient fallback behavior.

#### What Changed

**Before:** Phases 2-7 used a weak "read handoff and warn" pattern where malformed arguments, feature mismatches, or stale handoffs would print a warning but proceed anyway — risking silent failures or wrong feature context.

**After:** All phases now route through `resolve-feature.js`, which:
- Parses CLI arguments strictly and rejects unknown flags
- Detects feature mismatches between CLI args and handoff
- Fails hard with non-zero exit on any safety violation (invalid args, stale handoff, no fallback available)
- Only silently falls back to handoff when args are completely absent (the safe case)

#### New Helper: resolve-feature.js

A new 203-line helper in `hooks/resolve-feature.js` (also installed to `.claude/helpers/resolve-feature.js`) provides safe argument classification:
- Parses `--feature-slug <name>`, `--phase <N>`, `--args <args-string>` with strict validation
- Returns either the resolved feature target or a detailed failure reason
- Powers all phase commands via simple wrapper calls

#### Synchronization Guarantee

Both helper files are now protected by deterministic sync tests:
- `hooks/checkpoint.js` and `.claude/helpers/checkpoint.js` verified byte-identical
- `hooks/resolve-feature.js` and `.claude/helpers/resolve-feature.js` verified byte-identical
- Tests use strict `assert.equal(source, installed)` to prevent drift

#### Installer Fixes

- **init.sh** now copies `resolve-feature.js` to target projects on first install
- **upgrade.sh** now includes `resolve-feature.js` in the core installer bundle
- Prior releases had a gap: the helper was wired into commands but not deployed to projects

## Known Limitations (Pre-existing, Not Introduced in This Release)

- **Feature slug naming mismatch:** The project's feature slugs use uppercase prefixes (e.g., `ISS-009-resolve-feature-coverage`), but `FEATURE_SLUG_RE` regex only accepts lowercase. This forces callers to rely on handoff.json fallback in practice. Tracked as a separate issue.
- **upgrade.sh idempotency gap (LOW risk):** resolve-feature.js copy is guarded by `if [ "$CORE_NEEDS_UPGRADE" = true ]`, so projects with recent core versions that re-run upgrade.sh won't receive the file if core is already up to date. Low risk since new installs always get it, but noted for future idempotency improvements.
