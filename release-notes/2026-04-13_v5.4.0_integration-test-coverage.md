## What's New

### Three-Level Test Coverage Requirement (ISS-022)

The framework previously lacked explicit guidance on integration tests — Phase 3 (test design) defined unit and E2E tests, but the middle tier (integration tests that call production entry points) was undefined. This created a gap: test designs could lack integration coverage without explicit failure, and QA agents had no clear bar for what "integration test" means.

v5.4.0 closes this gap by establishing a three-level coverage requirement with precise definitions for each tier.

## Upgrade Warning

Upgrading agents to `v5.4.0` in the middle of an active feature cycle is strongly discouraged.

This release adds a new Phase 3 gating factor:

- integration-test output is now a required artifact
- Phase 3 verification now checks for visible-effect assertions against the production entry point

That means older in-flight test-design outputs can fail after upgrade even if they were valid under the previous release.

Recommended recovery path:

1. finish the current feature before upgrading, if possible
2. if already upgraded, run `/status` first
3. regenerate Phase 3 artifacts from the last stable phase instead of forcing older outputs through the new gate

#### Key Changes

- **Three-level coverage structure added to TDD skill:** Unit tests (call the module directly), integration tests (call the production entry point and assert visible effect in output), and E2E tests (exercise full workflows). Naming conventions clarified: `[feature].integration.test.*` for integration tests.

- **test-design command output slot wired:** The test-design command now includes `tests/integration/$ARGUMENTS.integration.test.ts` as a required output artifact. Previously, the skill mentioned integration tests but the command had no slot for them.

- **Blocking integration test verification:** Phase 3 verification now requires that at least one test imports the production entry point AND contains an assertion on visible output. Import-only shells or utility mocks do not satisfy this — the check is deterministic and blocking, not advisory.

- **Fixture validation against production schema required:** Before writing fixture data, QA agents must read and confirm the fixture values match the production schema/type/enum exactly. This prevents test-to-prod divergence.

- **Degenerate input coverage rule:** When validation constraints are widened (e.g., email field now accepts empty or whitespace), the TDD skill requires test coverage for the degenerate values. Minimum required set: empty string, whitespace-only, max-length boundary. Additional values depend on the type being widened.

- **Architecture dependency explicitly documented:** Integration tests require understanding call chains. The TDD skill now requires architecture docs to include a "Call Chain" or "Integration Points" section. If missing, the QA agent adds an `// ARCH GAP` comment to the test file and flags it in the handoff for Phase 6 review. Phase 3 may proceed, but the gap is explicitly recorded.

- **PIPELINE_GUIDE.md Phase 3 deliverables updated:** Integration tests are now explicitly listed alongside unit and E2E tests as a Phase 3 output requirement.

## Known Limitations

1. **Integration test definition depends on clear entry points.** If a feature has multiple or no obvious production entry points, QA agent must document the chosen entry point and rationale in the test file header. This is guidance, not a gate.

2. **Architecture dependency gap-handling is recorded, not blocked.** If the arch doc lacks a Call Chain section, Phase 3 proceeds but records the gap for Phase 6 review. Future work (ISS-023) will make call chains a Phase 2 deliverable, eliminating this gap at the source.

3. **Fixture validation is a best-practice rule, not enforced.** Contract tests confirm the rule is present in the skill; enforcement depends on QA agent discipline.

---

## Migration & Next Steps

- **For existing target projects:** plan upgrades between feature cycles when possible. `v5.4.0` changes the Phase 3 contract and can invalidate in-flight test-design artifacts created under earlier releases.
- **For future Phase 3 test design:** QA agents will now follow the three-level coverage structure, integration entry point requirement, fixture validation, and degenerate input rules.
- **For future Phase 2 architecture:** Architects are encouraged to include Call Chain or Integration Points sections to provide QA agents with the context needed for integration test design (ISS-023 tracks making this explicit).
