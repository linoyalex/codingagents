## What's New

### Command↔Skill Wiring Verification (ISS-036)

The framework previously had no automated way to verify that when a skill declares "produce artifact X," the command invoking that skill actually tells the agent *where* and *how* to create X. This gap produced the most damaging class of escaped defect: the skill says "produce integration tests," all tests pass on the skill's text, but the command never mentions integration tests — the feature is "green" but not shipped.

v5.5.0 closes this gap with a test-time wiring verification library (`lib/wiring-check.js`) that cross-references skill artifact declarations against command output instructions.

#### Key Changes

- **4-stage wiring algorithm:** Discovery (parse `## Skill References` tables from commands) → Registry parse (parse `## Required Artifacts` tables from skills) → Wiring check (verify command's Output section references both the artifact's naming pattern and target path) → Negative fixture (dedicated mock fixture proves gap detection works).

- **Two new conventions established:** Commands must declare skill dependencies in a `## Skill References` table (fail-closed: commands with skill-loading prose but no table fail the check). Skills that require named artifacts must declare them in a `## Required Artifacts` table with 4 columns: `Artifact | Pattern | Path | Condition`.

- **Phase 5 and Phase 3 verification steps:** `commands/implement.md` and `commands/test-design.md` now include `## Artifact Wiring Verification` sections instructing developers and QA agents to confirm all skill artifacts have output slots before committing.

- **Blank-cell parser hardening:** Blank Pattern or Path cells now throw descriptive errors instead of silently bypassing the wiring check via `String.includes('')` always returning true. Blank Source path cells in Skill References throw instead of producing raw EISDIR errors.

- **All 7 pipeline commands now have `## Skill References` tables:** architect, document, implement, review, security-gate, specify, and test-design.

## Process Learnings

| Phase | Insight |
|-------|---------|
| Phase 5 | Extracting shared library (`lib/wiring-check.js`) from test code was necessary to avoid Node's recursive `node --test` subprocess guard — integration tests that spawned `node --test` as a child process failed silently |
| Phase 6 | Phrase-binding source-text greps in tests (scanning JS source for keyword strings) are the most brittle form of test coupling — replace with behavioral assertions that import and call the functions directly |
| Phase 5 | Blank table cells parsed via `split('\|').filter(Boolean)` silently collapse columns; use `slice(1, -1)` to preserve blank interior cells, then validate required cells are non-empty |

## Known Limitations

1. **Fail-closed heuristic depends on explicit skill paths.** The discovery stage scans for `skills/` or `.claude/skills/` prose to detect commands that load skills. Commands that load skills via indirect mechanisms (e.g., helper scripts) would bypass this check. Current commands all use explicit paths.

2. **Wiring check is test-time only.** No runtime enforcement prevents a command from being used without wiring verification. This is by design for v1; runtime enforcement can layer on later without architectural change.

3. **Single artifact entry in TDD skill.** Currently only integration tests are declared in the Required Artifacts table. As more artifact types are added to skills, the wiring checker will automatically cover them.
