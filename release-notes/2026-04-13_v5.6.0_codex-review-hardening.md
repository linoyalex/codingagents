## What's New

### Codex Review Method Hardening (ISS-027)

Recent Codex re-reviews of ISS-009 exposed three systemic gaps in the review method: install-path blindness (reviewer missed that a new dependency wasn't copied by the installer), test-truthfulness gaps (a test name claimed sync verification but only checked one property), and incomplete edge-case enumeration (malformed inputs surfaced across multiple re-reviews instead of one).

v5.5.0 closes these gaps by adding four conditional review rules to `codex/reviewers/review-code.md`, each with an explicit trigger condition so reviewers skip inapplicable checklists.

#### Key Changes

- **Install-path tracing rule (AC1):** When a diff introduces a new file in `skills/`, `commands/`, or `hooks/`, the reviewer must verify `init.sh` and `upgrade.sh` operationalize it. The rule is mechanism-agnostic — literal copies, directory copies, loops, manifests, and helper functions all satisfy it.

- **Test-truthfulness verification rule (AC2):** When a diff includes test files, the reviewer must compare each test name to its assertion body. Misleading names (e.g., "sync check" that only asserts one property) are flagged as MAJOR.

- **Parser/validator edge-case checklist (AC3):** When a diff modifies a parser or validator, the reviewer must enumerate malformed-input shapes (empty, boundary, wrong types, missing/extra fields) and verify direct test coverage for each important shape.

- **Unchanged-file scope expansion rule (AC4):** When a changed file is operationalized by an unchanged file (installer, generator, copy script), the unchanged file is in scope. Scope expansion must be deliberate and documented.

- **Process docs updated (AC5):** `docs/memory/codex-rules.md` updated with a "Review Method Rules" section. `docs/memory/review-process.md` deduplicated — now defers to `codex-rules.md` as the canonical source.

- **Structural anchor tests (AC6):** 19 tests protect the four new rules from silent regression using heading-level regex anchors and keyword presence — no phrase-binding.

- **Mechanism-agnostic installer coverage contract test (AC7):** 24 tests verify every source file reaches the target project via `init.sh` and `upgrade.sh`. The `isCoveredByScript()` function accepts any valid installer mechanism. False-positive guards filter comment-only references (`activeLines()`) and echo/log lines (`isInertLine()`). Escape-hatch abuse prevented by exclusion cap (5 per script) and phantom-exclusion detection.

## Known Limitations

1. **`isInertLine` blacklist is intentionally narrow.** It filters `echo`, `printf`, `log`, `print` but not `test -f`, `ls`, or variable assignments. Broader blacklisting risks rejecting valid installer mechanisms. Current `init.sh`/`upgrade.sh` don't use these patterns for path references.

2. **Installer contract test is string-based, not execution-based.** It verifies path presence in active script lines, not that the installer actually copies files. A manifest entry mentioning a path satisfies the contract even if the manifest reader has a bug. This is a known tradeoff — execution-based testing would require a clean-room install environment.

3. **ISS-044 filed for systematic fix.** The adversarial contract testing gap (comment-only matches, unbounded escape hatches) was fixed ad-hoc in this feature but needs systematic support in `/test-design` and TDD skill.
