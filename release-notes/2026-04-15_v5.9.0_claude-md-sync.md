## What's New

### CLAUDE.md Section-Level Sync (ISS-008)

Framework consumers can now keep their project's root `CLAUDE.md` current with the framework's reference conventions using `--sync-claude-md` on `init.sh` or `upgrade.sh`. Instead of overwriting the entire file, the sync operates at section level using managed HTML comment markers, preserving all user-added content outside the markers.

#### Key Changes

- **Opt-in sync via `--sync-claude-md`** — both `init.sh` and `upgrade.sh` accept the new flag. Without it, behavior is unchanged (backwards compatible).

- **Fail-closed allowlist** — only 3 sections sync downstream: `code-conventions-must-follow`, `architecture-notes`, `known-gotchas`. New bullets in `docs/CLAUDE.md` do NOT sync until a maintainer explicitly adds them to the allowlist in `lib/sync-claude-md.sh`.

- **Managed markers** — synced content is wrapped in `<!-- managed:start:<section-id> -->` / `<!-- managed:end:<section-id> -->` pairs. Content inside markers is framework-owned (replaced on sync). Content outside markers is user-owned (never touched).

- **Legacy migration** — projects with pre-sync `CLAUDE.md` files (no markers) are migrated automatically: the sync locates sections by heading, strips template scaffolding and template-identical lines, inserts markers with framework content, and preserves genuine user content below the closing marker.

- **Pre-sync backup** — before modifying an existing `CLAUDE.md`, the sync saves a copy to `CLAUDE.md.pre-sync` with restore instructions. No-op syncs skip backup creation. If the backup cannot be created, sync aborts before touching the original file.

- **Interactive prompt redesign** — `init.sh` without `--sync-claude-md` now offers `(o)verwrite / (e)xit` when an existing `CLAUDE.md` is found. Exit halts the script immediately (preventing partial installs). Non-interactive environments keep the existing file with a reminder.
