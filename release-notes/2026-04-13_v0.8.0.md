## What's New

### Skill Size Convention & Progressive Disclosure (ISS-013)

The framework previously enforced a strict ~100-line skill budget that conflated instructional prose with templates, examples, and boilerplate. This led to two problems: (1) skills that violated the budget in practice because necessary detail couldn't be excluded from the count, and (2) no mechanism to handle skills that needed rich contextual guidance without becoming bloated.

v0.8.0 replaces this with a revised budget and a progressive disclosure pattern that separates core instruction from context-specific detail.

#### Key Changes

- **Skill size budget revised:** Inline skills now allow ~150 lines instructional prose (templates, tables, and examples excluded from line count) with a 250-line total threshold. This more accurately reflects the cost of the content and eliminates artificial pressure to trim necessary guidance.

- **Progressive disclosure pattern:** Skills exceeding 250 lines can now split into a compact SKILL.md (≤120 prose lines covering the core job-to-be-done) plus sibling reference files at `skills/<name>/<reference>.md`. This allows deep, phase-specific or context-specific guidance to coexist with scannable core material.

- **Stop conditions footer rule:** Pipeline-gating skills (verification-gate, security-audit, tdd, code-review) must now end with a `**STOP CONDITIONS (end of file):**` marker explaining hard constraints. This ensures reviewers who skim don't miss critical gates.

- **verification-gate pilot:** The verification-gate skill was converted to progressive-disclosure format as a proof-of-concept. Instead of simply reorganizing content, the pilot demonstrated signal-positive trimming: boilerplate was removed, phase-specific examples were moved to reference files, and the core SKILL.md now covers the essential job-to-be-done without overhead.

- **Skill migration audit:** A dynamic audit of all existing skills in `skills/` categorizes them by compliance tier: Compliant (≤250 lines), Needs Trimming (251–300 lines), Needs Splitting (>300 lines). The report is committed to `docs/memory/skill-migration-audit.md` and includes total counts.

#### Progressive Disclosure Pattern

The pattern is as follows:

```
skills/skill-name/
├── SKILL.md          # ≤120 lines instructional prose covering core job
└── phase-3.md        # Phase-specific reference (examples, checklists, templates)
└── phase-5.md        # Phase-specific reference
└── worked-example.md # Detailed walkthrough for complex concepts
```

From SKILL.md, reference files are linked using: `[See reference: skills/skill-name/phase-3.md]`

The **verification-gate** skill demonstrates this pattern. Its core SKILL.md explains what verification-gate is and why it matters. Per-phase guidance (Phase 3 test design, Phase 5 implementation, Phase 6 review) lives in sibling reference files, allowing readers to navigate to the context they need without wading through irrelevant phases.

## Known Limitations

1. **Progressive disclosure is opt-in.** Only verification-gate was pilot-converted; other skills remain inline. Conversion of existing skills is tracked as future work (unblocks ISS-022, ISS-024, ISS-025, ISS-033).

2. **countProseLines edge case:** The prose-line counter treats `---` (markdown separator) as prose. This is a false-positive only when used outside fenced code blocks. Real skills rarely trigger this; unit tests validate the boundary.

3. **AC7 smoke test residual risk:** The phase 3 smoke test is a shell script, not unit-tested. Contract + E2E suites validate the same workflows with stronger observability.

4. **Link format convention:** The pattern requires `.claude/skills/<name>/<reference>.md` links. This matches installed skill paths in target projects but differs from source repository structure (`skills/...`). Clarified in docs/CLAUDE.md "Known Gotchas."

## Migration Path for Existing Skills

The audit report (`docs/memory/skill-migration-audit.md`) identifies which skills need action:

- **Compliant (≤250 lines):** No action required. Inline skills remain valid indefinitely.
- **Needs Trimming (251–300 lines):** Can either trim to ≤250 or adopt progressive disclosure (if reusable reference material exists).
- **Needs Splitting (>300 lines):** Must either trim or split using the progressive disclosure pattern.

No forced migration is required. Future tickets (ISS-022, ISS-024, ISS-025, ISS-033) will address specific skills as new content is added.
