## What's New

### Core Skills Reliability Refresh (ISS-010)

The four core skills — `prd-writing`, `architecture-decision`, `tdd`, and `verification-gate` — have been comprehensively refreshed to improve agent reliability and reduce ambiguous guidance.

#### Key Improvements

**1. Stop Conditions & Failure Handling**
Each refreshed skill now opens with explicit guidance on when to halt, escalate, or recover instead of improvising:
- Ambiguous or stale inputs (feature spec incomplete, prior phase artifact missing)
- Malformed state (handoff.json invalid, test skip detected)
- Conflicting signals (test fails for wrong reason, requirements contradict architecture)

Example: `tdd` skill now requires stopping if RED fails for an unintended reason — preventing GREEN code from being written on top of false-positive test failures.

**2. Architecture Decision Robustness**
`architecture-decision` now captures:
- **Decision Confidence** — how confident in this choice (low/medium/high) and what would change the decision
- **Revisit Triggers** — conditions that should prompt re-evaluation (e.g., if token limits hit 70%, revisit caching strategy)
- **Rollback / Fallback** — what to do if the decision proves wrong in production
- **Trust Boundary** — which external systems/dependencies this decision depends on

This provides future maintainers with the reasoning context needed to make good decisions about whether to keep or replace the choice.

**3. Test-Driven Development with Case Coverage**
Before writing GREEN code, agents now identify three concrete test cases from the spec:
- **Happy path** — the intended use case
- **Edge case** — boundary conditions (empty, null, max size)
- **Misuse or abuse case** (when relevant) — how the feature could be misused or attacked

Example provided: For a password-reset endpoint, happy case is a valid token resetting a password; edge case is a token expiring at a second boundary; misuse case is a token for User A being used to reset User B's password.

**4. Feature-Scoped Verification Checks**
`verification-gate` now prefers narrow, deterministic checks over broad project-wide commands. Phase-specific examples now include:
- Node.js stack: `node --test tests/node/<feature>*.test.js`
- npm/TypeScript stack: `npm test -- tests/contracts/<feature>.test.ts tests/e2e/<feature>.spec.ts`
- Python stack: `pytest tests/<feature>/ -v`

All examples include "adapt to your stack" comments to prevent hardcoding toolchain assumptions.

**5. Signal Density Improvements**
Despite adding robust new concepts, core skills were consolidated from 553 lines to 409 lines (26% reduction) through:
- Removing templated preambles; keeping per-skill "Top Rules" sections concise
- Keeping structural anchors (heading names, template fields) instead of binding guidance to exact wording
- Focusing on highest-leverage rules at the top of each skill
