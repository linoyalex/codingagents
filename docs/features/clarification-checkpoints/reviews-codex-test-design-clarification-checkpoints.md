## Coverage Map Notes
- AC0b has reasonable structural coverage: the suite does explicitly look for contradiction-check language.
- AC4 is covered at a basic structural level: there are direct tests for a review checkpoint in `commands/architect.md`.
- AC6 and AC8 are weakly covered because ordering is used as a proxy for “no hidden auto-advance.”
- AC1, AC2, and AC3 are only partially covered because the suite does not bind the concrete trigger matrix or the explicit “Dependencies assumptions and proceed” behavior.
- AC0, AC0a, and AC0c are only partially covered because the tests do not assert the specific fidelity failure modes called out in the PRD.
- AC5 and AC7 are weakly covered because the suite does not verify approval-gated revision cycles or distinct waiting/completion signals.

## Recommendations
- Replace the pure ordering assertions for AC6/AC8 with negative-pattern checks that fail on unconditional finalize/commit/handoff instructions after a checkpoint unless they are explicitly approval-gated.
- Add trigger-level assertions for AC1 and AC3 that bind the named clarification triggers and the exact “record unanswered questions as Dependencies assumptions and proceed” behavior.
- Strengthen AC0a and AC0c with section-scoped assertions for canonical `docs/CLAUDE.md` precedence and the “enumerate or ask the user which ones apply” branch.
- Add architect-flow assertions for explicit approval gating and “still in review” state signaling, not just generic `feedback` or `revision` words.
- Align the verification path in handoff/CI so the reviewed `tests/contracts/` and `tests/e2e/` artifacts are actually run.
