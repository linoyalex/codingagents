# Architecture Review: dogfood-pipeline

## Findings
- [BLOCKING] [AC6 / AD2] The scoping strategy is not actually enforceable in this branch. The architecture says `restore-context.js` "already loads both files" and that phase prompts must include `For framework development, use docs/CLAUDE.md as primary context.` to keep agents on framework-dev instructions (`docs/features/dogfood-pipeline/architecture.md:25-31`), but the SessionStart hook only restores handoff/archive context and never reads either CLAUDE file (`hooks/restore-context.js:50-99`, `hooks/restore-context.js:131-152`). The architect and document commands also still instruct agents to read generic `CLAUDE.md`, not `docs/CLAUDE.md` (`commands/architect.md:21-23`, `commands/document.md:21-23`). Because the branch has no hard boundary between the consumer template and framework-development instructions, AC6 cannot be validated reliably and phase 2/7 runs can succeed while using the wrong governing prompt.
- [BLOCKING] [AC4 / Failure Modes] The design treats invalid handoff recovery as covered, but the only next-session recovery path silently skips malformed handoffs. The PRD requires that when the next phase starts, `restore-context.js` logs an error and notifies the agent (`docs/features/dogfood-pipeline/prd.md:15`, `docs/features/dogfood-pipeline/prd.md:29`), while the architecture says invalid handoffs are handled because `checkpoint.js` blocks the previous phase (`docs/features/dogfood-pipeline/architecture.md:79-85`). In reality, `restore-context.js` returns `null` on malformed JSON or missing minimum fields and emits no diagnostic (`hooks/restore-context.js:53-58`, `hooks/restore-context.js:96-98`). Since `hooks/settings.json:14-21` wires that script as the sole SessionStart hook, a corrupted `.claude/handoff.json` between phases can leave the next agent with no warning and no restored context. I reproduced this with a malformed handoff: the hook exited `0` with zero stdout and zero stderr.
- [MAJOR] [AD4 / AC9] Token-budget reporting is under-specified and not anchored to a single source of truth. The architecture says dogfood should use `docs/CLAUDE.md` as primary context (`docs/features/dogfood-pipeline/architecture.md:25-31`) but also says phase variance should be compared against the "CLAUDE.md budget table" via an unspecified shell one-liner (`docs/features/dogfood-pipeline/architecture.md:44-46`, `docs/features/dogfood-pipeline/architecture.md:92`). The only budget table in this branch is in the root template CLAUDE (`CLAUDE.md:276-286`), while the existing reporting helper reads root `CLAUDE.md` and `PIPELINE.md`, not `docs/CLAUDE.md` (`codex/report-usage.sh:35-109`). Tests currently verify only that the reporting script gets installed and is executable (`tests/test-install-scripts.sh:133-142`), not that the dogfood flow can generate `docs/features/dogfood-pipeline/token-report.md` or flag >20% variance correctly. As written, AC9 is reproducible only by tribal knowledge and is easy to compute against the wrong budget source.

## Open Questions
- Should AC4 and AC6 be satisfied by explicit framework changes in this branch, or is the dogfood run intentionally meant to document those failures instead of preventing them?
- Which file is the authoritative budget source for AC9 during dogfooding: root `CLAUDE.md`, `PIPELINE_GUIDE.md`, or a new dogfood-local budget table?
- If the existing `codex/report-usage.sh` is acceptable, why does AD4 reject a shared reporting tool in favor of an ad hoc shell one-liner?

## Recommendation
Rework before implementation. Merge recommendation: REQUEST CHANGES.
