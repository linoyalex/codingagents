# ISS-037: Make review artifacts additive and expose the latest review state clearly

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `review-workflow`, `artifacts`, `traceability`, `claude`, `codex`
- **Created:** 2026-04-13
- **Reporter:** Linoy
- **Feature area:** Review artifacts, reviewer prompts, review workflow, documentation

## Description

Review artifacts are currently easy to overwrite or regenerate in place. That makes the
review loop harder to operate because both humans and agents have to scan the full file to
figure out:

- what the latest review said
- whether the latest verdict is approve vs request changes
- which findings belong to the newest pass versus an older pass
- which review state should guide the next phase

There is also a freshness problem during iterative review: an agent can carry forward stale
context from an earlier pass and act as if it re-read the artifact, even after the
on-disk review file has changed. The workflow needs a deterministic "fresh copy" handshake
so later phases can prove they are using the latest review state rather than cached memory.

We should make review artifacts for both Claude and Codex additive by default. New review
passes should be preserved rather than replacing prior rounds, and the newest review should
be clearly identifiable at the top of the file.

Within each preserved review pass, the recommendation/verdict should appear up front so a
reader does not need to scroll through findings before learning whether the review approves,
requests changes, or needs discussion.

This should apply across the main review artifact families, including:

- `review.md`
- `review-codex-code-<feature>.md`
- `review-codex-prd-<feature>.md`
- `review-codex-architecture-<feature>.md`
- `review-codex-tests-<feature>.md`

## Acceptance Criteria

- [ ] **AC1 (Additive review convention):** Review workflows for both Claude and Codex are
      updated so each new review pass is added as a preserved entry instead of replacing
      the prior review content.
- [ ] **AC2 (Newest review first):** The latest review entry appears first in the file,
      with older review rounds retained below it in reverse-chronological order.
- [ ] **AC2a (Recommendation first within each pass):** Each preserved review pass places
      its recommendation/verdict near the top of that pass in a standardized, easy-to-scan
      location rather than only at the end of the entry.
- [ ] **AC3 (Latest-state locator):** Each review artifact exposes a clear, standardized
      latest-review summary near the top of the file so an agent can identify the current
      verdict without scanning historical entries.
- [ ] **AC4 (Latest-state fields):** The latest-review summary includes at minimum:
  - review timestamp
  - reviewer identity/system (`claude` or `codex`)
  - review type (code, PRD, architecture, test design, etc.)
  - latest verdict/recommendation
  - pointer to the newest full review entry if the summary is separate
- [ ] **AC4a (Freshness token):** Each review artifact exposes a machine-readable freshness
      token for the latest pass — at minimum the current `**Generated:**` timestamp, and
      optionally a stronger token such as a content hash — so downstream agents can prove
      they re-read the current on-disk copy.
- [ ] **AC5 (Prompt/workflow updates):** Relevant Claude and Codex reviewer instructions are
      updated so they append/prepend a new review entry using the standardized format
      instead of regenerating the entire artifact in place.
- [ ] **AC6 (Agent discoverability):** Commands, prompts, or docs define how later agents
      should locate the latest review quickly, distinguish it from historical entries, and
      prove they re-read the current on-disk copy before acting on it.
- [ ] **AC6a (Review freshness handshake):** When a later phase consumes a review artifact,
      the workflow requires the agent to echo back the latest freshness token from the file
      it just read. If the token in context does not match the file on disk, the agent must
      re-open the artifact and use the newer version.
- [ ] **AC7 (Timestamp compatibility):** The additive format works cleanly with the
      generated-timestamp convention so it is unambiguous which timestamp belongs to the
      latest review versus prior rounds.
- [ ] **AC7a (Handoff compatibility):** If the framework chooses to carry review freshness
      metadata in `.claude/handoff.json`, the schema and restore-context flow support
      feature-local review pointers (path + freshness token) without breaking existing
      handoff consumers.
- [ ] **AC8 (Regression protection):** Deterministic tests or contract checks verify that:
  - new reviews preserve older entries
  - newest review appears first
  - each preserved review pass puts the recommendation/verdict up front
  - the latest-review summary remains parseable/predictable
  - the freshness token remains parseable/predictable
  - downstream prompts require a fresh-read echo before acting on review state
  - both Claude and Codex review artifact families are covered
- [ ] **AC9a (Round delineation for new findings):** When a reviewer adds new findings to a file that already contains resolution notes from the implementer, the new findings must be in a clearly separated section (e.g., `## Round 2 Findings`) so the implementer can distinguish new issues from resolved ones. Evidence: ISS-036 Codex added blank-cell parser findings to the same file where Claude had added "ALL FINDINGS RESOLVED" notes for 3 earlier issues — Claude re-verified the old findings 7 times without noticing the new content.
- [ ] **AC10 (Cross-phase review caveat forwarding):** When a review artifact for phase N
      contains a caveat, "partial" verdict, or unresolved open question on any AC, that
      caveat must be visible to later-phase reviewers. Implementation options (architect
      decides): (a) carry `review_caveats` in handoff.json so Phase 6 reviewer sees them
      via restore-context, (b) instruct Phase 6 reviewer to scan for prior-phase review
      artifacts (`review-claude-*-<feature>.md`, `review-codex-*-<feature>.md`) and extract
      open caveats before forming their own assessment, or (c) aggregate caveats in the
      latest-state summary (AC3). The Phase 6 reviewer must verify each forwarded caveat
      independently — forwarding provides visibility, not trust inheritance.
      Evidence: qa-test-quality Phase 3 review flagged AC12 as "partial" (FINDING-2, LOW)
      with the correct diagnosis ("only checks for skip markers, not full regression").
      Phase 6 reviewer ran in fresh context per separate-context convention, had no
      visibility into the Phase 3 caveat, and marked AC12 as PASS. The finding was lost
      at the phase boundary.
- [ ] **AC9 (Process docs):** Shared workflow docs explain:
  - when to append a new review entry
  - how to interpret the latest-review summary
  - how this interacts with any separate `review-history.md` or traceability artifacts

## Notes

### Why this is needed

- Operators often need the latest actionable review state, not a full archaeology pass.
- Agents need a deterministic place to look for the current verdict.
- Agents also need a deterministic way to prove they read the current file from disk rather
  than relying on stale carried context.
- Even within a single review pass, the recommendation should be visible immediately instead
  of being buried after detailed findings.
- Preserving prior review rounds improves traceability without requiring a separate file for
  every review pass.

### Evidence for AC10 (qa-test-quality cross-phase finding attrition)

During the qa-test-quality feature (Batch 2.75), Claude's Phase 3 test-design review
correctly flagged AC12 as "partial" — the skip-marker scan only covered 2 of 3 test files
and could not enforce full regression, only guard against self-skipping. The AC Coverage
Matrix marked it `Partial (see FINDING-2)`.

The Phase 6 code review ran in a fresh context (correct per separate-context convention)
and had no visibility into the Phase 3 caveat. It formed its own assessment, saw the AC12
test existed and passed, and marked AC12 as PASS. Codex later caught the same gap as a
MAJOR finding.

The separate-context convention is correct — reviewers must not carry authoring-phase
context. But it creates a finding-forwarding gap: caveats from earlier reviews are
invisible to later reviewers. AC10 closes this gap by making prior-phase caveats
discoverable without violating reviewer independence.

### Relationship to other tickets

- **ISS-006** adds `review-history.md` to record what changed in response to feedback.
  ISS-037 complements it by preserving the review rounds themselves and making the latest
  review easy to identify in the artifact.
- **ISS-012** becomes easier to operate when the latest Codex/Claude verdict is visible at
  the top of the file and later phases can verify they consumed the latest on-disk copy.
- **ISS-026** adds timestamps to generated artifacts. ISS-037 builds on that by supporting
  multiple timestamped review rounds in a single artifact.

### Likely implementation shape

- Near-term workflow rule: later phases re-open the review file from disk and echo the
  latest `**Generated:**` line before acting on its findings.
- Durable workflow option: extend `.claude/handoff.json` with feature-local review metadata
  (for example review path + freshness token) so restore-context can surface the latest
  review state explicitly.
- Stronger integrity option: store a content hash alongside the timestamp when simple
  freshness tokens prove too weak for multi-round review flows.
