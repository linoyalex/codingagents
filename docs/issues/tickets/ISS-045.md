# ISS-045: /test-design must instruct QA to adversarially test contract robustness, not just contract satisfaction

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `testing`
- **Created:** 2026-04-13
- **Reporter:** Codex review feedback on ISS-027 architecture
- **Feature area:** `commands/test-design.md`, `skills/tdd/SKILL.md`

## Description

During codex-review-hardening (ISS-027), the QA agent wrote an installer coverage contract test that checked path-presence in `init.sh`/`upgrade.sh`. Codex review found two HIGH-severity gaps:

1. **False-pass on comment lines:** The contract checked raw string presence, meaning a path in a `# comment` line would satisfy the assertion. The contract tested *declaration*, not *active coverage*.
2. **Unbounded exclusion escape hatch:** The exclusion array had no cap or ownership policy, meaning future contributors could silence the safety invariant instead of fixing installer drift.

Both gaps share a root cause: **the QA agent tested whether the contract was satisfied, but not whether the contract could be trivially evaded.** This is distinct from ISS-043 (symmetric coverage), which is about testing all members of a set. This ticket is about adversarially testing the robustness of the contract mechanism itself.

## Acceptance Criteria

- [ ] **AC1:** `commands/test-design.md` includes an instruction: "For contract tests (tests that assert a safety invariant), also test that the contract cannot be trivially satisfied — e.g., commented-out code, escape hatches that grow unbounded, string matches that hit dead code."
- [ ] **AC2:** The TDD skill's "What to Test First" list includes: "Contract robustness — can the safety invariant be trivially evaded?"
- [ ] **AC3:** A contract test verifies the adversarial-contract instruction exists in `commands/test-design.md`.

## Notes

- Root cause: QA accepted the architecture doc's "path-presence" contract literally without testing edge cases of the matching mechanism
- The developer would have implemented to pass (TDD discipline), so the false-pass risk would have shipped
- ISS-043 covers symmetric testing; this ticket covers adversarial contract testing — complementary, not duplicative
- Fixed ad-hoc in ISS-027 Phase 3 rework (activeLines filter, exclusion cap); this ticket makes it systematic
