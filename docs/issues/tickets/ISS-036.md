# ISS-036: Add command↔skill wiring verification to prevent artifact-slot drift

- **Type:** Feature
- **Priority:** P1 — High
- **Label:** `framework`, `pipeline-quality`, `testing`, `reliability`
- **Created:** 2026-04-13
- **Reporter:** Linoy (via cross-review pattern analysis)
- **Feature area:** `commands/*.md`, `skills/*/SKILL.md`, contract tests, Phase 5 verification

## Description

When a skill introduces a new artifact type (e.g., "integration tests must be produced"), the corresponding command must also be updated to tell the agent *where to create that artifact* and *what to name it*. In practice, the developer agent updates the skill content and writes tests that validate the skill's text — but never checks that the command operationalizes the new artifact.

This gap has produced the most damaging class of escaped defect: the skill says "produce X," the tests verify the skill says "produce X," all tests pass, but the command never tells the agent to actually create X. The feature is "green" but not shipped.

### Evidence

- **integration-test-coverage (BLOCKING):** TDD skill requires `[feature].integration.test.*` files, but `commands/test-design.md` only provides output slots for `tests/contracts/` and `tests/e2e/`. An agent following the command exactly would never create the integration test.
- **artifact-timestamps:** Fitness functions verified 4 source commands mention `**Generated:**` but missed Codex reviewer prompts and installed command copies — the wiring was incomplete.
- **ISS-009:** New `resolve-feature.js` wired into commands but not into `init.sh` — a different axis of the same wiring gap.

### Root cause

No phase in the pipeline verifies that **every artifact type required by a skill has a corresponding output instruction in the command that invokes it**. The developer reads the architecture doc and failing tests, implements skill changes, tests pass on content anchors, and the command is never checked.

## Acceptance Criteria

- [ ] **AC1 (Wiring contract test):** Add a contract test that, for each command that loads a skill, verifies the command provides output instructions for every artifact type the skill requires. At minimum: if `skills/tdd/SKILL.md` names `integration.test.*` as a required output, `commands/test-design.md` must reference that naming convention and provide a concrete output path.
- [ ] **AC2 (Phase 5 verification step):** Add to `commands/implement.md` (or the TDD skill's Phase 5 checklist): "For every new artifact type introduced in a skill, verify the corresponding command includes an output slot (file path, naming convention) for that artifact."
- [ ] **AC3 (Phase 3 verification step):** Add to `commands/test-design.md` verification: "Confirm the command provides output instructions for every test level required by the TDD skill (unit, integration, E2E)."
- [ ] **AC4 (Skill artifact registry):** Skills that require agents to produce named artifacts must list them in a discoverable format (e.g., a `## Required Artifacts` section or frontmatter field) so contract tests can cross-reference against commands.
- [ ] **AC5 (No regression):** Existing tests pass. The wiring test catches the integration-test-coverage gap when run against the pre-fix state.
