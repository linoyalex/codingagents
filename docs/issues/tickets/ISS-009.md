# ISS-009: `/implement` can proceed after mangled command arguments by silently falling back to handoff feature

- **Type:** Bug
- **Priority:** P1 — High
- **Label:** `pipeline`, `workflow`, `commands`, `safety`
- **Created:** 2026-04-06
- **Reporter:** Linoy
- **Feature area:** Slash command argument handling, handoff validation, Phase 5 implementation

## Description

Calling `/implement` produced the message:

`Handoff valid: feature preset-storage-parity, phase 4 → phase 5. Note: command args got mangled but feature slug is preset-storage-parity. Proceeding.`

This suggests the command invocation arguments were parsed incorrectly, but the workflow
continued anyway by trusting the feature slug from `.claude/handoff.json`.

That fallback may be useful for recovery in some cases, but silently proceeding after
argument corruption is unsafe. If the handoff is stale or references a different feature,
the implementation phase could run against the wrong target while only emitting an
informational note.

## Acceptance Criteria

- [ ] Reproduce the `/implement` argument-mangling path in a deterministic test
- [ ] When command arguments are malformed, missing, or mangled, the workflow does not
      silently proceed on an ambiguous target
- [ ] The system has one clearly defined authority for feature resolution:
  - either explicit command args are authoritative and mismatches hard-fail
  - or `handoff.json` is authoritative and conflicting args are rejected with a clear message
- [ ] If fallback-to-handoff behavior is retained, it is explicit, documented, and only
      allowed when the command input is unambiguously empty rather than corrupted
- [ ] Error output clearly tells the user what happened and how to recover
- [ ] Regression coverage protects against implementing the wrong feature because of
      stale handoff state or mangled slash-command arguments

## Notes

Relevant current contract:
- `commands/implement.md` says the phase should read `.claude/handoff.json` first and
  warn if it references a different feature or unexpected phase
- the observed runtime behavior appears to go further by auto-recovering from bad args
  and proceeding

This should be treated as a safety bug, not just a UX issue, because it affects which
feature the pipeline may operate on.
