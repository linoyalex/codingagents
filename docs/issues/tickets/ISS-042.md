# ISS-042: /implement command must instruct developer to verify handoff known_risks

- **Type:** Feature
- **Priority:** P1 — High
- **Label:** `framework`, `pipeline`
- **Created:** 2026-04-13
- **Reporter:** developer agent (RCA from review-hardening Phase 5 rework)
- **Feature area:** `commands/implement.md`, `skills/tdd/SKILL.md`

## Description

During review-hardening Phase 5, the security audit flagged path traversal on `source_spec` as a HIGH finding and the handoff carried it in `known_risks`. The developer read the architecture doc and test files (per `/implement` instructions) but never checked `known_risks` from the handoff. The path traversal guard was not implemented until Phase 6 review caught it.

The `/implement` command says "Read ONLY: architecture.md + failing test files." It does not instruct the developer to read `known_risks` from the incoming handoff and verify each is addressed before committing GREEN.

## Acceptance Criteria

- [ ] **AC1:** `commands/implement.md` includes an instruction between RED and GREEN: "Read `known_risks` from `.claude/handoff.json`. For each risk, verify your implementation addresses it or document why it is deferred."
- [ ] **AC2:** The TDD skill's GREEN phase checklist includes: "Verify handoff known_risks are addressed."
- [ ] **AC3:** A contract test verifies the `known_risks` instruction exists in `commands/implement.md` using structural anchors.

## Notes

- Root cause: the handoff carries security findings as `known_risks` but no phase instructs the developer to act on them
- The security audit's findings reach the developer only through handoff text, not through a structured enforcement point
- This is a process gap, not a code gap — the fix is prose in the command template
