# ISS-044: Prevent scope expansion during post-review artifact rework

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `workflow`, `pipeline`, `review-quality`
- **Created:** 2026-04-13
- **Reporter:** Linoy
- **Feature area:** `commands/specify.md`, `commands/architect.md`, `codex/reviewers/review-prd.md`

## Description

When an agent reworks a pipeline artifact (PRD, architecture doc) to address review findings, it currently re-runs the full generation process with unconstrained freedom to add new material. This creates a scope-expansion vector: the agent fixes the 5 findings from the review but also invents 6 new ACs that aren't in the source ticket.

This was observed in the clarification-checkpoints feature (ISS-029):
- Rev1 of the PRD faithfully transcribed ISS-029's AC0–AC8
- Codex review found 5 legitimate gaps
- Rev2 addressed the Codex findings **and** added AC9–AC14 (tier routing) — a feature that does not exist in ISS-029
- The drift was only caught when a human re-triggered `/specify`

Neither the authoring command nor the Codex PRD reviewer detected the scope expansion because:
1. No command instruction says "when reworking after review, only address the findings"
2. The Codex PRD reviewer (`review-prd.md`) does not compare PRD ACs against the source ticket
3. No structural check detects AC count or scope changes between revisions

This ticket adds rework scope-lock instructions and ticket-fidelity checking to the review loop.

## Acceptance Criteria

- [ ] **AC1 (Rework scope-lock in `/specify`):**
  Given `/specify` is re-run to address review findings,
  When the agent rewrites the PRD,
  Then it must only address the specific findings from the review — new ACs or scope expansions require explicit user approval before being added.

- [ ] **AC2 (Rework scope-lock in `/architect`):**
  Given `/architect` is re-run to address review findings,
  When the agent rewrites the architecture document,
  Then it must only address the specific findings — new sections, decisions, or scope expansions require explicit user approval.

- [ ] **AC3 (Codex PRD reviewer ticket fidelity):**
  Given `codex/reviewers/review-prd.md` reviews a PRD that has a source ticket reference,
  When the reviewer evaluates the PRD,
  Then it must read the source ticket and verify that every PRD AC traces to a ticket AC. ACs that do not trace to the ticket must be flagged as potential scope creep.

- [ ] **AC4 (AC count change detection):**
  Given a PRD is being reworked (not authored for the first time),
  When the agent produces a revised PRD,
  Then it must compare the AC count and AC IDs against the previous revision and explicitly call out any additions or removals with justification.

- [ ] **AC5 (Verification):**
  Given the rework scope-lock behavior is implemented,
  When structural checks run,
  Then they verify: (a) rework scope-lock language exists in `/specify` and `/architect` command files, and (b) the Codex PRD reviewer includes a ticket-fidelity check instruction.

- [ ] **AC6 (Root-cause synthesis before rework):**
  Given the implementer receives review findings to address,
  When they begin reworking,
  Then the rework instructions (in `/implement` or equivalent command) must require: write a one-sentence root cause for each finding before implementing the fix. If the root cause is "X is missing" (additive), verify that adding X resolves the reviewer's stated concern. If the root cause is "the approach to Y is wrong" (structural), the fix must change the approach, not patch it.
  Evidence: qa-test-quality round 1 Codex finding said "AC12 only scans two files... does not enforce the acceptance criterion." Implementer read "two files" and added a third, rather than recognizing the root cause was "skip-marker scanning is the wrong enforcement mechanism for regression detection." A root-cause synthesis step would have surfaced the mismatch before the shallow fix was implemented.

- **ISS-033** adds source-spec verification at Phase 6 review time. ISS-044 adds it to the Codex PRD review loop, which runs earlier and independently.
- **ISS-027** hardens `review-code.md` (Codex code review). ISS-044 hardens `review-prd.md` (Codex PRD review) — different reviewer, same principle.

### Evidence

- clarification-checkpoints rev2 added 6 ACs (tier routing) not in ISS-029 while addressing 5 Codex findings
- Pattern likely recurs in any feature where Codex review triggers a rework cycle

### Evidence for AC6 (qa-test-quality round 2 RCA)

Two rework-quality failures in a single review cycle, both caused by shallow fix resolution:

- **Symptom-fix on AC12:** Codex said "AC12 only scans two files... does not enforce the acceptance criterion." The implementer read "two files" and added a third file to the scan list. Round 2 Codex re-flagged the same finding — the enforcement mechanism (skip markers) was unchanged. Root cause was never articulated before fixing.
- **Untested adversarial case on AC11:** Codex said "table checks don't verify non-empty Test approach column." The implementer added a `filter(Boolean)` assertion that passed because production data has no empty cells. The fix was never tested against the scenario it was meant to guard against (empty cell), and `filter(Boolean)` silently collapses empty cells, making the assertion point to the wrong column.

AC1-AC2 would not have caught these — scope didn't expand. The fixes were too shallow, not too broad. AC6 addresses this complementary failure mode.

### Branch note — `feature/ISS-001-invariants-audit`

The invariants-audit branch reinforced that post-review rework needs tighter coupling to
the original findings, not just a generic "revise the artifact" loop. A useful narrow
workflow slice emerged:

- rework commands should read the existing review artifact first
- when they address a finding, they should write a short response note back into that
  review artifact describing what was changed
- re-reviewers should inspect those response notes before deciding whether the finding is
  resolved

This is complementary to ISS-044 rather than a replacement for it. ISS-044 still owns the
scope-lock and root-cause discipline that prevents rework from drifting or staying
shallow. Inline response notes mainly improve traceability and reviewer continuity during
the rework loop.
