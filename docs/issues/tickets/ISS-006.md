# ISS-006: Add `review-history.md` to capture cross-review rework in feature artifacts

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `review-workflow`, `artifacts`, `audit-trail`
- **Created:** 2026-04-06
- **Reporter:** Linoy
- **Feature area:** Feature artifacts, review loop, process traceability

## Description

The current pipeline records review output, but it does not provide a standard,
feature-local audit trail for what changed after review feedback. During rework, it is
hard to answer basic questions quickly:

- which reviewer requested the change
- when the feedback was addressed
- which review artifact or comment drove the update
- what was updated in response

This makes the review loop harder to follow, especially when a feature receives
multiple review passes or feedback from different review types. We should standardize on
`docs/features/<feature>/review-history.md` as the canonical place to record review-driven
updates across the feature lifecycle without adding a separate tracking system.

## Acceptance Criteria

- [ ] Add a standard `docs/features/<feature>/review-history.md` artifact and define its
      required format
- [ ] Each review-driven update entry includes:
  - ISO 8601 timestamp
  - reviewer identity
  - review type
  - pointer to the source feedback artifact or review comment
  - short summary of updates made in response
- [ ] The convention supports multiple review rounds for the same feature without
      overwriting earlier entries
- [ ] `review-history.md` works across all supported review sources, including:
  - Claude review artifacts
  - Codex review artifacts
  - architecture reviews
  - code reviews
  - test-design reviews
  - security reviews
- [ ] Reviewer and/or phase prompts are updated so the workflow explicitly records
      review follow-up updates in `review-history.md`
- [ ] At least one deterministic test or contract check covers the new review-history
      convention
- [ ] Documentation explains when to append a new entry versus editing an existing one
- [ ] Documentation explains which review outcomes require an entry:
  - new blocking or major feedback
  - follow-up updates that address prior review findings
  - subsequent review passes that confirm or reopen concerns

## Notes

This should stay lightweight. The goal is traceability inside the feature directory, not
a heavyweight approval system.

Motivating example:
- an architecture review identifies blocking feedback
- a code review later adds follow-up concerns
- the author updates `prd.md`, `architecture.md`, `test-design.md`, or related artifacts
- `review-history.md` records when each update happened, which reviewer/review type drove
  it, what feedback it answered, and what changed

Suggested entry shape:
- `Timestamp: 2026-04-06T14:30:00Z`
- `Reviewer: codex`
- `Review type: architecture`
- `Feedback: docs/features/<feature>/review-codex-architecture-<feature>.md`
- `Updates made: Clarified AC4 failure handling; aligned CLAUDE scoping instructions`

Open implementation choice:
- whether entries should be appended only by the authoring phase that makes the fix, or
  also by later reviewers when they verify the fix
