# ISS-026: Add generated timestamps to feature artifacts

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `artifacts`, `traceability`, `documentation`
- **Created:** 2026-04-11
- **Reporter:** Linoy
- **Feature area:** Pipeline-generated feature artifacts (`prd.md`, `architecture.md`, `security-audit.md`, `review.md`, Codex review docs, and related templates/commands)

## Description

Feature artifacts are generated across multiple phases, but most of them do not include an explicit generated timestamp inside the document itself. That makes it harder to answer basic traceability questions such as:

- when this artifact was last produced
- whether a review predates the current branch state
- whether two review documents were written in the same cycle or on different reruns
- whether an artifact was copied forward or regenerated

This ticket adds a consistent timestamp convention to pipeline-generated feature artifacts so readers can quickly understand freshness and sequencing without relying only on git history.

## Acceptance Criteria

- [ ] **AC1 (Timestamp convention):** A single document-level timestamp convention is defined for generated feature artifacts, including exact field name and format (ISO 8601).
- [ ] **AC2 (Core artifact coverage):** The pipeline instructions/templates are updated so newly generated `prd.md`, `architecture.md`, `security-audit.md`, `review.md`, and `review-codex-*.md` files include the timestamp field.
- [ ] **AC3 (Fresh generation behavior):** When an artifact is regenerated or rewritten in a later phase/review cycle, its timestamp is updated rather than preserved from an earlier run.
- [ ] **AC4 (Documentation):** The convention is documented in the relevant pipeline or artifact-writing guidance so future contributors know which artifacts require timestamps.
- [ ] **AC5 (Verification):** At least one regression check verifies that the generated artifact instructions/templates require the timestamp field.
- [ ] **AC6 (No regression):** Existing tests continue to pass after the prompt/template updates.

## Notes

### Why this matters

- Review artifacts are often read out of sequence during re-review, and timestamps make stale feedback much easier to spot.
- Feature folders increasingly contain multiple review documents (`review.md`, `review-codex-*`, `review-claude-*`), so readers need an in-document freshness signal.
- Git history helps, but a visible timestamp inside the artifact lowers the cost of understanding the pipeline state during audits and handoffs.

### Candidate scope

- `commands/specify.md`
- `commands/architect.md`
- `commands/security-gate.md`
- `commands/review.md`
- Codex reviewer prompt/templates that write feature review files
