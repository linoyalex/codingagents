# ISS-046: Introduce shared project configuration and pipeline profiles for portable multi-project use

- **Type:** Architecture
- **Priority:** P2 — Medium
- **Label:** `framework`, `portability`, `configurability`, `workflow`, `architecture`
- **Created:** 2026-04-14
- **Reporter:** Linoy
- **Feature area:** Shared project config, command/skill resolution, pipeline profiles, portability

## Description

`codingagents` currently works best in repositories that adopt the framework's own
conventions wholesale: `docs/features/`, `docs/issues/`, `CHANGELOG.md`,
`release-notes/`, `.claude/handoff.json`, `main...HEAD`, and a Claude-first runtime
layout. That is acceptable as an opinionated default, but it becomes a portability gap
when another project wants to use the framework without mirroring this repo exactly.

We should keep the framework opinionated about process quality while separating
**framework defaults** from **project-specific settings**. Projects need a supported way to
configure paths, outputs, ticket sources, review strictness, and workflow shape without
forking commands and skills.

This ticket defines the architectural foundation for that flexibility. The target outcome
is not "every project can do anything." The target outcome is:

- the framework keeps a strong default development process
- projects can override the settings that are truly project-specific
- commands, skills, hooks, and install/upgrade flows read one shared config model instead
  of inventing ad hoc knobs in many places
- later tickets such as backlog-system support and multi-agent routing can build on the
  same foundation

## Acceptance Criteria

- [ ] **AC1 (Shared config model):** Define a machine-readable project configuration model
      that separates opinionated framework defaults from project-level overrides. The
      design must define precedence clearly enough that commands, skills, hooks, and
      install/upgrade flows resolve the same final configuration.

- [ ] **AC2 (Backward-compatible defaults):** If a project provides no explicit config,
      the framework continues to behave like the current `codingagents` defaults. The new
      architecture must not require every existing project to migrate immediately.

- [ ] **AC3 (Project-specific paths):** The config model includes the project-specific path
      and location settings that are currently hardcoded or scattered, including at
      minimum:
  - feature artifact root
  - ticket/backlog root
  - release-note/changelog outputs
  - project guidance files such as `CLAUDE.md`
  - review diff base / default branch

- [ ] **AC4 (Optional functions):** The config model defines how projects enable, disable,
      or narrow optional functions such as documentation outputs, release-note updates, or
      other non-core workflow surfaces. The architecture must also define which functions
      are invariant safety requirements that cannot be disabled casually.

- [ ] **AC5 (Review strictness):** The config model supports bounded review/gate
      strictness settings with a clear explanation of what changes across the supported
      levels and what remains mandatory regardless of project preference.

- [ ] **AC6 (Pipeline profiles):** The architecture supports named work-type or pipeline
      profiles such as `bugfix`, `small-ticket`, `standard-feature`, and `epic`, with a
      documented way to change required artifacts, documentation expectations, and review
      intensity without turning the pipeline into an unbounded custom workflow engine.

- [ ] **AC7 (Ticket-system adapters):** Define the extension point for different
      ticket/backlog systems, including the current markdown-file convention and external
      systems such as GitHub Issues, Linear, or a project-owned custom pointer. This does
      not have to implement every adapter immediately, but the core config contract must
      make later adapters coherent.

- [ ] **AC8 (Provider-neutral foundation):** The architecture leaves room for first-class
      multi-agent/provider routing without assuming one vendor's runtime layout is the
      framework definition. This ticket should make ISS-038 easier to implement, not force
      ISS-038 to invent a second config model.

- [ ] **AC9 (Resolved-config consumers):** The design identifies which commands, skills,
      hooks, and install/upgrade flows must read resolved project config instead of
      hardcoded conventions, and it groups that migration into coherent implementation
      slices.

- [ ] **AC10 (Migration plan):** Document how existing repos adopt the new config model
      incrementally, including how defaults preserve current behavior and how projects move
      from "no config" to "project-tuned config" safely.

- [ ] **AC11 (Regression protection):** Define deterministic checks that will protect the
      config shape, precedence rules, and core command consumers once this architecture is
      implemented.

- [ ] **AC12 (Docs update):** Top-level docs explain the intended philosophy: opinionated
      defaults are good, but project-specific settings belong in shared config rather than
      hardcoded path assumptions or one-off command edits.

## Notes

### Why this is separate from ISS-038

- **ISS-038** is about making Codex, Gemini, and future LLMs first-class coding agents.
- **ISS-046** is the broader project-portability foundation that defines where shared
  project settings live.

ISS-038 should build on this ticket's config model rather than inventing a second routing
format just for agents.

### Why this is separate from ISS-034

- **ISS-034** is about backlog-system flexibility specifically.
- **ISS-046** defines the shared configuration foundation that backlog adapters, review
  strictness, pipeline profiles, and future agent routing should all use.

### Design principles

- Keep the framework opinionated by default.
- Only make project-specific concerns configurable.
- Prefer one shared config model over many command-local switches.
- Bound customization so reliability invariants do not disappear behind arbitrary per-repo
  scripting.
