# ISS-031: Extend `/document` to update README and other project documentation artifacts

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `commands`, `documentation`, `pipeline`
- **Created:** 2026-04-11
- **Reporter:** Linoy
- **Feature area:** `commands/document.md`, `skills/release-docs/SKILL.md`

## Description

The `/document` command (Phase 7) currently updates three artifacts after a feature merge:

1. `CHANGELOG.md` — one bullet per AC
2. `release-notes/` — a new release note file
3. `CLAUDE.md` (or `docs/CLAUDE.md`) — conventions and timestamp

It does **not** update the project `README.md` or any other user-facing documentation. For a framework like codingagents, this means the README drifts from reality after every feature cycle — new commands, changed behavior, added skills, or revised workflow steps are not reflected until someone manually updates it.

This ticket extends `/document` to optionally update `README.md` and other configurable documentation artifacts as part of Phase 7, so that user-facing docs stay current without a separate manual pass.

## Acceptance Criteria

- [ ] **AC1 (README update):** `/document` checks if the feature introduced or changed user-facing behavior (new commands, changed workflow, added skills, modified install steps). If so, it updates the relevant sections of `README.md`.
- [ ] **AC2 (Scoped reads):** The documentation-specialist reads only the feature's `prd.md`, existing `README.md`, and the diff summary — not `src/`. The update is derived from the spec, not the implementation.
- [ ] **AC3 (Additive only):** README updates are additive or corrective — the agent does not rewrite sections unrelated to the feature. Changes are limited to sections that reference concepts touched by the feature.
- [ ] **AC4 (Configurable artifact list):** The skill or command supports a list of documentation artifacts to update (defaulting to `README.md`). Projects can configure additional files (e.g., `HOW_TO_USE.md`, `PIPELINE_GUIDE.md`, `API.md`) via a convention in `CLAUDE.md` or the command itself.
- [ ] **AC5 (Skip when not needed):** If the feature does not affect user-facing behavior (e.g., internal refactors, test-only changes), the agent skips the README update and logs why.
- [ ] **AC6 (Verification):** Phase 7 verification includes a check that updated doc artifacts are consistent with the feature's ACs.
- [ ] **AC7 (No regression):** Existing Phase 7 behavior (CHANGELOG, release notes, CLAUDE.md) continues to work unchanged. The README update is an addition, not a replacement.

## Notes

### Current gap

The codingagents README documents repo structure, setup, commands, skills, and workflow. After landing ISS-009 (which wired 4 new commands through `resolve-feature.js`), the README was not updated to reflect this. The same will happen for every future feature unless `/document` is extended.

### Design considerations

- The documentation-specialist runs on Haiku. README updates may need Sonnet for accuracy if the changes are complex. Consider a model escalation rule: Haiku for CHANGELOG/release-notes, Sonnet for README rewrites.
- `PIPELINE_GUIDE.md` and `HOW_TO_USE.md` are also user-facing docs that drift. The configurable artifact list (AC4) should cover these.
- The agent should not re-read `src/` — it should derive README changes from the PRD and the diff summary, keeping token cost low.
