# ISS-043: /test-design must instruct QA to test symmetric requirements across all enumerated components

- **Type:** Feature
- **Priority:** P1 — High
- **Label:** `framework`, `testing`
- **Created:** 2026-04-13
- **Reporter:** developer agent (RCA from review-hardening Phase 5 rework)
- **Feature area:** `commands/test-design.md`, `skills/tdd/SKILL.md`

## Description

During review-hardening, the architecture doc enumerated two gate roles (ROLE_CODE_REVIEWER.md, ROLE_SECURITY.md) and two gate commands (review.md, security-gate.md) that should receive identical hardening (adversarial stance, separate context, produced_by check, read-only). The QA agent wrote tests for the code-reviewer side but not consistently for the security side — `produced_by` check was tested only in `review.md`, and `separate context` was tested only in `ROLE_CODE_REVIEWER.md`.

The developer then implemented to pass the tests, reproducing the asymmetry in production. Phase 6 review caught the gap.

## Acceptance Criteria

- [ ] **AC1:** `commands/test-design.md` includes an instruction: "When the architecture enumerates multiple components receiving the same treatment (e.g., both gate roles get adversarial stance), write tests for ALL enumerated components, not just one representative."
- [ ] **AC2:** The TDD skill's "What to Test First" list includes: "Symmetric requirements across all enumerated components."
- [ ] **AC3:** A contract test verifies the symmetric-testing instruction exists in `commands/test-design.md`.
- [ ] **AC4 (Behavioral binding):** `commands/test-design.md` includes an instruction: "Tests must bind to the specific behavior the PRD requires — not just assert that a keyword exists in the file. A test that passes when the command mentions 'clarification' but drops the actual trigger classes is too weak."
- [ ] **AC5 (Negative-pattern testing):** `commands/test-design.md` includes an instruction: "When an AC specifies a 'must not' property (e.g., 'must not silently finalize'), write a negative-pattern assertion that fails if the forbidden pattern is present — do not rely on ordering alone as a proxy for conditionality."

## Notes

- Root cause: QA tested one representative of each symmetric pair, not all members
- The developer implemented to pass tests (TDD discipline) which propagated the asymmetry
- This is a Phase 3 quality gap — fixing it at test design time prevents asymmetric implementations
- Related to ISS-039 AC6 (reviewer-side symmetric enforcement check)

### Evidence from ISS-039 test design review (code-review-skill-hardening)

Codex review of ISS-039 Phase 3 found three MEDIUM gaps, two of which map to this ticket:
- **Behavioral binding (AC4):** AC2 tests verified drift-check mechanics (path mapping, diff command) but not the *consequence* — that the reviewer must raise a finding on divergence. A methodology saying "compare files" without saying "raise a HIGH finding on mismatch" would pass. Test fixed: added assertion for finding/flag/HIGH/divergence in the drift-check procedure.
- **Behavioral binding (AC4):** AC4 tests checked that `reproduction.md` mentions "escalation" but didn't verify the negative rule (BLOCKING must NOT be assigned to unverified findings) appeared in the same policy section. Could pass on a document mentioning escalation in unrelated context. Test fixed: added contiguous policy slice assertion requiring unverified + BLOCKING prohibition + escalation together.
- **Multi-behavior coverage (AC1):** AC3 enumerates 4 behaviors (discovery, execution, scoping, error handling) — QA tested 3 of 4, missing the multi-suite selection rule. Test fixed: added assertion requiring language about scoping suites to files touched by the diff.

### Evidence for AC4–AC5 (clarification-checkpoints test design review)

Codex review of ISS-029 test design found two additional Phase 3 quality gaps beyond symmetry:
- **Behavioral binding (AC4):** Tests for AC1/AC2/AC3 matched single keywords (`/clarification/i`, `/trigger/i`, `/material/i`) instead of binding to the specific trigger classes and "record unanswered in Dependencies and proceed" behavior the PRD requires. A command that mentions clarification generally but drops the actual triggers would pass the suite.
- **Negative-pattern testing (AC5):** AC6/AC8 specify "must not silently finalize after checkpoint" — a negative property. QA tested this via ordering (checkpoint pos < commit pos), which is necessary but insufficient. A command with an unconditional "continue anyway" path after the checkpoint would still pass. The test needed a negative assertion: no unconditional finalize/commit between checkpoint and approval gate.
