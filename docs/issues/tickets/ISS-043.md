# ISS-043: /test-design must instruct QA to test symmetric requirements across all enumerated components

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `testing`
- **Created:** 2026-04-13
- **Reporter:** developer agent (RCA from review-hardening Phase 5 rework)
- **Feature area:** `commands/test-design.md`, `skills/tdd/SKILL.md`

## Description

During review-hardening, the architecture doc enumerated two gate roles (ROLE_CODE_REVIEWER.md, ROLE_SECURITY.md) and two gate commands (review.md, security-gate.md) that should receive identical hardening (adversarial stance, separate context, produced_by check, read-only). The QA agent wrote tests for the code-reviewer side but not consistently for the security side — `produced_by` check was tested only in `review.md`, and `separate context` was tested only in `ROLE_CODE_REVIEWER.md`.

The developer then implemented to pass the tests, reproducing the asymmetry in production. Phase 6 review caught the gap.

## Acceptance Criteria

- [ ] **AC1:** `commands/test-design.md` includes an instruction: "When the architecture enumerates multiple components receiving the same treatment (e.g., both gate roles get adversarial stance), write tests for ALL enumerated components, not just one representative."
- [ ] **AC2:** The TDD skill's "What to Test First" list includes: "Symmetric requirements across all enumerated components."
- [ ] **AC3:** A contract test verifies the symmetric-testing instruction exists in `commands/test-design.md`.

## Notes

- Root cause: QA tested one representative of each symmetric pair, not all members
- The developer implemented to pass tests (TDD discipline) which propagated the asymmetry
- This is a Phase 3 quality gap — fixing it at test design time prevents asymmetric implementations
- Related to ISS-039 AC6 (reviewer-side symmetric enforcement check)
