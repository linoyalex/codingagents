# ISS-054: Formalize ADR practice — architect must read, write, and promote architectural decisions

- **Type:** Architecture
- **Priority:** P2 — Medium
- **Label:** `framework`, `architecture`, `process`
- **Created:** 2026-04-15
- **Reporter:** Linoy
- **Feature area:** `skills/architecture-decision/SKILL.md`, `commands/architect.md`, `docs/decisions/`

## Description

The project has an ADR Index in CLAUDE.md but it has only one entry (dogfood proposal). Architectural decisions are made in per-feature `architecture.md` files that no future session reads. Reusable patterns (e.g., "combined sibling with purpose boundary, split at ~80 lines") are lost after the feature pipeline completes.

Two levels of architectural guidance need formalization:

1. **Project-level ADRs** — decisions specific to this project that have cross-feature implications. These should be formal ADR files in `docs/decisions/` referenced from the CLAUDE.md ADR Index. The architect agent should read them before making new decisions to avoid re-debating settled questions.

2. **Global framework conventions** — patterns that apply to any codingagents project. These belong in `docs/CLAUDE.md` Code Conventions and propagate to consumer projects via `--sync-claude-md`.

## Acceptance Criteria

- [ ] **AC1 (Read existing ADRs):** The architect skill (`skills/architecture-decision/SKILL.md`) instructs the architect to read the ADR Index in CLAUDE.md and review relevant existing ADRs before making new decisions.

- [ ] **AC2 (Promote cross-feature decisions):** The architect skill instructs: when a feature architecture decision has cross-feature implications (it would change how other features are designed), extract it as a formal ADR in `docs/decisions/ADR-NNN.md` and add it to the CLAUDE.md ADR Index.

- [ ] **AC3 (ADR backfill):** Populate the ADR Index with decisions currently buried in feature architecture docs or Known Gotchas. Candidates include: gate symmetry (ISS-039), progressive disclosure (ISS-013), structural anchors (ISS-010), source/installed sync (ISS-009).

- [ ] **AC4 (Promotion criteria):** The architect skill documents clear criteria for when a decision should be promoted to a formal ADR vs left in the feature architecture doc. Minimum: "If a future architect working on a different feature would need to know this decision, it's an ADR."

- [ ] **AC5 (No regression):** Existing tests continue to pass. The ADR Index in CLAUDE.md does not exceed the ~250 line budget (entries are one-line pointers to ADR files).

## Notes

### Root cause

During QA test quality hardening (ISS-043/045/049), the architecture session made the decision "one combined sibling with purpose boundary, split at ~80 lines." This pattern is reusable but would have been lost in `docs/features/qa-test-quality/architecture.md` — a file no future session reads. The purpose boundary was promoted to a global convention in `docs/CLAUDE.md` as an immediate fix, but the broader ADR workflow gap remained.
