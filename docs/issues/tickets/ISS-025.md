# ISS-025: Add adversarial self-review checkpoint to Phase 5 verification

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `skills`, `testing`, `verification`
- **Created:** 2026-04-11
- **Reporter:** Linoy (via global memory audit of cross-project learnings)
- **Feature area:** `skills/verification-gate/SKILL.md`, `skills/tdd/SKILL.md`

## Description

After implementing a feature, the developer currently runs verification checks (tests pass, lint clean, handoff written) but does not systematically challenge their own work. Tests that pass on the first try can create false confidence — they prove fixture-implementation consistency, not implementation-production consistency.

This ticket adds an adversarial self-review step to the Phase 5 verification gate and the TDD GREEN step.

## Acceptance Criteria

- [ ] **AC1 (TDD GREEN checkpoint):** `skills/tdd/SKILL.md` adds to the GREEN step: "Before committing GREEN, walk each AC and ask: if a malicious tester wanted to fail this AC while keeping my tests green, what would they do? If an answer exists, that test is missing."
- [ ] **AC2 (Fixture validation in Phase 5):** `skills/verification-gate/SKILL.md` adds to Phase 5 verification: "Validate that test fixtures use production-correct types/values, not stand-in shapes."
- [ ] **AC3 (First-try suspicion):** The TDD skill adds: "Be slightly suspicious when tests pass on the first try. Tests passing means fixture-implementation consistency, not implementation-production consistency. If a critical AC's test went green immediately, double-check that the test exercises the path the AC describes."
- [ ] **AC4 (Test the architecture's known limitations):** When the architecture doc discloses a known limitation, residual risk, or weak enforcement mechanism, the Phase 3 QA agent must write a test that verifies the mitigation exists AND the limitation is disclosed. Add this instruction to `skills/tdd/SKILL.md` or `commands/test-design.md`. Evidence: review-hardening AC6 disclosed same-agent-different-role as a known limitation with layered mitigations, but Phase 3 tests only checked for keyword "separate context" — the limitation was never falsified.
- [ ] **AC5 (Decompose enumerated ACs):** When an AC lists ordered alternatives, precedence rules, or multiple valid values, the QA agent must write one test per enumerated item plus one test for the ordering. Add this instruction to `skills/tdd/SKILL.md` under "What to Test First" or "Coverage Rules". Evidence: review-hardening AC15 stated "Precedence: ticket file > GitHub issue URL > other declared source" but Phase 3 tests only covered the ticket path — URL and ordering were untested.
- [ ] **AC6 (Assertion specificity check):** Add to the Phase 3 verification checklist (`skills/verification-gate/phase-3-test-design.md`): "For tests asserting document structure, verify the assertion would fail if the document contained the right keywords in the wrong location or with wrong mappings." Evidence: review-hardening AC7 tests checked for "authoring" and "gate/review" keywords anywhere in the pipeline section but did not verify the specific phase-to-tag mapping, allowing a mislabeled pipeline table to pass.
- [ ] **AC7 (Phase 2 adversarial self-review):** `commands/architect.md` adds to the Phase 2 verification checklist: "For every skip/pass-by-default path in the design, ask: what happens if the metadata is missing, stale, or malformed? If the answer is 'silently pass,' switch to fail-closed or document why fail-open is acceptable." Evidence: ISS-036 architect committed fail-open discovery (skip commands without `## Skill References`) and full-file substring matching (incidental mentions satisfy check) — both caught by Codex, not by self-review.
- [ ] **AC8 (No regression):** Existing tests continue to pass. Skills stay within size budget.

## Notes

### Evidence

- Tests passed on first try because fixtures used wrong data types — internally consistent but not production-correct. 15/15 tests passed but Codex found a BLOCKING bug (closet-designer preset-storage-parity).
- Self-review in the same session as implementation produced false confidence — the developer was invested in their own code being right (closet-designer consistent-design-labels).

### Evidence for AC4–AC6 (review-hardening Phase 3 Codex review)

Three test-quality gaps found by Codex review of review-hardening test shells:
- **Known-limitation oracle (AC4):** Architecture disclosed same-agent-different-role as a known limitation with residual risk, but QA agent never tested whether mitigations exist or limitation is disclosed — only tested keyword "separate context". Root cause: no instruction to use arch doc's known limitations as test oracles.
- **Enumerated AC decomposition (AC5):** AC15 listed three precedence alternatives (ticket > URL > other). QA agent only tested the first (ticket path). Root cause: "one test per AC" rule doesn't instruct decomposing ACs with enumerated branches.
- **Assertion specificity (AC6):** AC7 tests checked for keyword presence ("authoring", "gate/review") but didn't verify the specific phase-to-tag mapping. A mislabeled pipeline table would pass. Root cause: TDD skill warns against phrase-binding but doesn't guide how specific structural anchors should be.
