# ISS-028: Add ticket-aware feature selection and backlog state commands

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `workflow`, `pipeline`, `documentation`
- **Created:** 2026-04-11
- **Reporter:** Linoy
- **Feature area:** Command UX, feature selection, backlog state transitions, pipeline resumability

## Description

Typing the full feature slug for every phase command is tedious, especially for long issue-driven feature names like `iss-009-resolve-feature-coverage`. The current pipeline optimizes for explicitness, but the ergonomics are clunky when a user already knows the backlog ticket they want to work on.

This ticket adds a ticket-aware `/feature` command that accepts a backlog ticket ID such as `ISS-009`, proposes a feature slug suffix based on the ticket title, lets the user confirm or rename the final feature slug before starting work, and updates backlog state so the ticket moves into active work automatically.

It also adds a corresponding ticket-resolution command so backlog state can be closed cleanly when the feature cycle is done.

Example desired flow:

1. User runs `/feature ISS-009`
2. The system reads `docs/issues/tickets/ISS-009.md`
3. It proposes a slug such as `iss-009-resolve-feature-coverage`
4. The user can accept the proposal or rename the suffix before continuing
5. The selected feature becomes the active feature context for later commands
6. The ticket is moved from `docs/issues/backlog.md` to `docs/issues/in-progress.md`

This should reduce repeated slug typing without reintroducing ambiguous natural-language parsing into phase commands.

## Acceptance Criteria

- [ ] **AC1 (Ticket ID input):** `/feature ISS-00N` reads the matching backlog ticket file and derives a proposed feature slug from the ticket title.
- [ ] **AC2 (Slug proposal):** The proposed slug preserves the ticket prefix and appends a kebab-case title-derived suffix, e.g. `ISS-009` → `iss-009-resolve-feature-coverage`.
- [ ] **AC3 (Rename confirmation):** Before the feature context is committed, the command presents the proposed slug and allows the user to rename or accept it.
- [ ] **AC4 (Backlog -> in-progress transition):** When `/feature ISS-00N` is confirmed, the ticket is removed from `docs/issues/backlog.md` and added to `docs/issues/in-progress.md` with the chosen branch/feature reference.
- [ ] **AC5 (Explicit context):** The selected slug is stored in an explicit machine-readable place so later commands can use it safely without guessing.
- [ ] **AC6 (Conflict handling):** If the proposed slug already exists, or if the ticket already maps to an existing feature directory, the command surfaces that clearly and lets the user choose whether to reuse or rename.
- [ ] **AC7 (Resolve command):** A dedicated command is added to resolve/close the active or specified ticket, moving it from `docs/issues/in-progress.md` to `docs/issues/closed.md` with a required resolution summary.
- [ ] **AC8 (Phase compatibility):** The design preserves the current strictness of phases 2-7: later phase commands should still use explicit resolved context rather than fuzzy natural-language parsing.
- [ ] **AC9 (Verification):** Deterministic tests cover ticket lookup, slug proposal, rename handling, backlog transition, resolution/close flow, and at least one conflict case.

## Notes

### Why this matters

- Users often think in terms of backlog ticket IDs first, not full feature slugs.
- The current workflow is accurate but repetitive for long slug names.
- Starting work and updating backlog state are currently separate manual chores.
- This provides a more ergonomic entry point without weakening the fail-closed `resolve-feature` model introduced for ISS-009.

### Relationship to other tickets

- ISS-009 tightened phase command argument handling. ISS-028 is the ergonomic counterpart: make feature selection easier up front instead of relaxing later phase command parsing.
- ISS-026 (artifact timestamps) and ISS-027 (Codex review method hardening) are process-quality improvements; ISS-028 is workflow ergonomics for human operators.
