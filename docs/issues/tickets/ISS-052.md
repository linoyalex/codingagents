# ISS-052: Add branch management skill and release finalization command

- **Type:** Feature
- **Priority:** P1 — High
- **Label:** `framework`, `workflow`, `branch-management`, `release`
- **Created:** 2026-04-15
- **Reporter:** Linoy
- **Feature area:** `skills/branch-management/SKILL.md`, `commands/specify.md`, `commands/document.md`, pipeline commands

## Description

The pipeline has no branch lifecycle management. `/specify` commits directly to whatever branch the operator is on — if that's main, the PRD lands on main. No phase command verifies it's on the correct branch. Phase 7 (`/document`) runs post-merge on main, meaning documentation changes bypass the feature branch and go unreviewed.

This was exposed during Batch 2.75 when the QA test quality PRD was committed to main because no command created or checked out a feature branch.

This ticket introduces two capabilities:

1. **Branch management skill** (`skills/branch-management/SKILL.md`) — branch naming convention, creation, and phase-branch verification. Consumed by `/specify` (create) and all phase commands (verify).

2. **Release finalization command** — a `/merge` or `/release` command (separate skill or extension of ISS-051) that merges the feature branch to main and promotes `[Unreleased]` changelog entries to a versioned release. This separates the mechanical merge workflow from release planning (ISS-051).

Additionally, Phase 7 moves from post-merge (on main) to pre-merge (on the feature branch), making the branch a complete unit of work. Phase 7 writes under `## [Unreleased]` instead of a versioned heading; the release finalization command promotes it at merge time.

## Acceptance Criteria

### Branch management skill

- [ ] **AC1 (Skill exists):** `skills/branch-management/SKILL.md` defines the branch naming convention (`feature/<slug>`), branch creation procedure, and a phase-branch verification matrix.

- [ ] **AC2 (Phase 1 creates branch):** `/specify` (via the branch management skill) creates and checks out `feature/<slug>` before its first commit. If the branch already exists, it checks out the existing branch. If the operator is already on the correct feature branch, it proceeds without error.

- [ ] **AC3 (Authoring phases verify branch):** Phase commands 2-7 (`/architect`, `/test-design`, `/security-gate`, `/implement`, `/review`, `/document`) verify they are on a feature branch (not main) before proceeding. If on main, the command warns and stops.

- [ ] **AC4 (Status reports branch):** `/status` reports the current branch and flags mismatches between the branch name and the active feature from handoff.json.

### Phase 7 on feature branch

- [ ] **AC5 (Phase 7 pre-merge):** `/document` runs on the feature branch, not on main after merge. All documentation artifacts (CHANGELOG entry, release note, CLAUDE.md updates) are part of the feature branch.

- [ ] **AC6 (Unreleased convention):** Phase 7 writes changelog entries under `## [Unreleased]` instead of a versioned heading (`## [5.x.0] — date`). The `[Unreleased]` section is always present in CHANGELOG.md.

### Release finalization command

- [ ] **AC7 (Merge command exists):** A `/merge` or `/release` command (in a release management skill or the branch management skill) performs: (1) verify all 7 phases complete via handoff, (2) checkout main, (3) merge feature branch with `--no-ff`, (4) promote `[Unreleased]` to `[version] — date` in CHANGELOG.md, (5) add fresh `## [Unreleased]` section, (6) commit and push.

- [ ] **AC8 (Version source):** The version number for promotion comes from the RELEASE.md roadmap or is provided as a command argument. The merge date is the current date.

- [ ] **AC9 (Release note finalization):** If the release note filename contains a placeholder or `UNRELEASED`, it is renamed to include the version and date.

### Pipeline documentation

- [ ] **AC10 (Pipeline docs updated):** CLAUDE.md pipeline sequence, README workflow section, and QUICKSTART.md are updated to reflect: (1) all phases on feature branch, (2) merge as a post-pipeline step, (3) Phase 7 writes `[Unreleased]`.

### Tests

- [ ] **AC11 (Contract tests):** Contract tests verify: (a) branch management skill has creation and verification sections, (b) `/specify` references the branch management skill, (c) phase commands include branch verification, (d) `/document` does NOT instruct to checkout main.

- [ ] **AC12 (No regression):** Existing tests continue to pass.

## Notes

### Root cause

During Batch 2.75 (ISS-043/045/049), `/specify` committed the PRD directly to main. Three commits had to be replayed onto a feature branch via `git reset --hard` and branch creation. The global CLAUDE.md says "All changes must be committed to a feature branch" but no command enforces this.

### Design decisions from discussion

- Branch management and release management are separate concerns — branch skill handles git lifecycle, release skill handles version promotion
- The `[Unreleased]` pattern follows [Keep a Changelog](https://keepachangelog.com/) convention and eliminates version/date guessing on the feature branch
- If two feature branches both add to `[Unreleased]`, the merge conflict is trivial (additive entries under the same heading)
- ISS-051 (release-manager) is about release *planning* (which version, which tickets, roadmap); this ticket is about the mechanical merge + promotion workflow

### Relationship to other tickets

- **ISS-051** — release planning and roadmapping. ISS-052 handles the mechanical merge/promote step; ISS-051 handles planning which version gets what. The two should be designed to complement each other, not overlap.
- **ISS-030** — semver policy. ISS-052 needs a version source (RELEASE.md roadmap or argument); ISS-030 formalizes how versions are chosen.
- **ISS-028** — ticket-aware feature selection. ISS-052's branch creation builds on the feature slug; ISS-028 could link tickets to branches.

### Scope considerations for the architect

- The branch management skill should be stack-agnostic (git is universal, but hosting-specific commands like `gh pr create` should be optional/documented, not required).
- The release finalization command could live in a new `skills/release-management/SKILL.md` or as a sibling reference in `skills/branch-management/`. The architect should decide based on skill size budget.
- The `/merge` command needs to handle the case where main has moved since the branch was created (rebase vs merge-commit policy).
