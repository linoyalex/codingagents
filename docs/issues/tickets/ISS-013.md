# ISS-013: Revise skill size convention and adopt progressive disclosure

- **Type:** Architecture
- **Priority:** P2 — Medium
- **Label:** `framework`, `skills`, `documentation`, `developer-experience`
- **Created:** 2026-04-07
- **Reporter:** Linoy (via Claude review of ISS-010)
- **Feature area:** `skills/`, `docs/CLAUDE.md` conventions

## Description

The current `docs/CLAUDE.md` convention "Skills stay under ~100 lines" is significantly stricter than Anthropic's official Claude Code skills guidance (≤500 lines per `SKILL.md`, with bundled resource files for detail) and is causing real friction:

1. **It is being violated in practice.** ISS-010's first implementation pushed all four core skills well past the 100-line cap (`prd-writing` 132, `architecture-decision` 170, `tdd` 150, `verification-gate` 240). The implementation review (Claude pass) cited this as a BLOCKING finding because the convention was the framework's own stated rule, but the fix is not "trim everything to 100" — some of that growth is legitimately useful template/example content.

2. **The convention conflates two very different kinds of lines.** Templates, tables, fitness-function examples, and worked examples have a high signal-per-line ratio because agents copy directly from them. Instructional prose has a low signal-per-line ratio because under context pressure agents skip preambles. A 240-line skill that is mostly templates is healthier than a 120-line skill that is mostly prose. Today's rule treats them the same.

3. **We are not using progressive disclosure.** Anthropic's published skill architecture loads metadata (~100 tokens) first, then loads full SKILL.md (<5k tokens) only when the skill is invoked, then loads bundled resources only as needed. We currently put everything inline in one file. Splitting `verification-gate` into a slim `SKILL.md` plus per-phase reference files would let the SKILL.md stay short *and* let the per-phase detail be richer.

4. **Stop conditions only appear at the top.** Recent prompt-engineering research (and Anthropic's own attention-pattern guidance) shows models attend most reliably to the **start and end** of a long prompt. ISS-010 added Stop Conditions sections at the top of refreshed skills but did not add a closing reminder. For high-stakes skills (verification-gate, tdd) the stop conditions should appear at both ends.

5. **Root and docs CLAUDE.md have drifted.** The root `CLAUDE.md` Memory & Instruction Governance table still says `~100 lines each` for skills, while `docs/CLAUDE.md` says `~120 lines`. There is no enforcement mechanism ensuring shared conventions stay consistent across both files. Phase 7 (DOCUMENT) updates "CLAUDE.md" but is ambiguous about which file. Any convention change in this ticket must land in both files, and a sync check should prevent future drift.

This ticket revises the convention, adopts progressive disclosure for large skills, updates the affected documentation, and ensures root/docs CLAUDE.md consistency. It does NOT retroactively give ISS-010 a free pass — ISS-010 must still demonstrate its own line additions are signal-positive.

## Acceptance Criteria

- [ ] **AC1 (Size budget):** `docs/CLAUDE.md` "Code Conventions → Must Follow" replaces "Skills stay under ~100 lines" with: *Skills should stay under ~150 lines of instructional prose. Templates, tables, code examples, and bundled reference files do not count toward the budget. If a skill exceeds 250 total lines, split it using progressive disclosure.*
- [ ] **AC2 (Progressive disclosure pattern):** Add a new "Skill Authoring → Progressive Disclosure" section to `docs/CLAUDE.md` (or to `skills/SKILL_AUTHORING.md` if `docs/CLAUDE.md` would grow too long) that documents the pattern: slim `SKILL.md` + `skills/<name>/<reference>.md` files + how the SKILL.md should reference them.
- [ ] **AC3 (Stop conditions footer rule):** Document that high-stakes skills (verification, security, TDD, anything that gates the pipeline) must repeat their stop conditions or no-go criteria at the **end** of the file, not only the top, to align with model attention patterns.
- [ ] **AC4 (Pilot conversion):** As proof-of-concept, refactor at least one over-budget skill (suggested: `verification-gate`) into the progressive-disclosure pattern. Slim `SKILL.md` ≤120 lines, per-phase verification detail moved to `skills/verification-gate/phase-<N>.md` reference files.
- [ ] **AC5 (Migration guidance):** The ticket or its PRD lists each existing skill against the new budget and notes whether it currently complies, needs trimming, or needs splitting. No mass rewrite — just the audit.
- [ ] **AC6 (Tests adapt):** Any contract test that asserts heading presence (e.g. `core-skill-contracts.test.js`) must continue to pass against both inline skills and progressive-disclosure skills, OR be updated to recognize the new pattern.
- [ ] **AC7 (No regression on agent behavior):** A spot-check run of at least one phase command using a converted skill confirms the pipeline still executes end-to-end without missing context.
- [ ] **AC8 (Convention is enforced):** Either a regression test or a documented pre-merge check verifies the new size budget against the `skills/` tree.
- [ ] **AC9 (Root/docs CLAUDE.md sync):** The skill size budget in root `CLAUDE.md` (Memory & Instruction Governance table, currently `~100 lines each`) must be updated to match the new convention in `docs/CLAUDE.md`. Any generic pipeline convention changed by this ticket must be consistent across both files. A pre-merge check or test should verify shared conventions do not drift.

## Findings From Codex's ISS-010 Implementation Review (driving this ticket)

These findings come from `docs/features/skill-reliability-refresh/review-claude-skill-reliability-refresh.md`. They are documented here so the size convention change is grounded in concrete evidence rather than abstract preference:

1. All four refreshed core skills exceeded the 100-line cap (prd-writing 132, architecture-decision 170, tdd 150, verification-gate 240).
2. The "Success Criteria / Allowed Inputs / Deliverables / Stop Conditions" 4-section header was added identically to every refreshed skill, contributing ~25 lines of templated boilerplate per skill with low per-skill signal.
3. The PRD's AC8 ("signal density without prompt sprawl") and the architect doc's AD2 ("prefer fewer, sharper instructions") were both violated by the implementation, and none of the three Codex sibling reviews flagged this — a sign that the current review prompts do not force AC-by-AC verification.
4. Anthropic's official skill guidance (research conducted 2026-04-07) recommends:
   - SKILL.md ≤500 lines, with bundled resources via progressive disclosure
   - Frontmatter: required `name` and `description`, optional `allowed-tools`
   - Naming: gerund form (`writing-prds` not `prd-writing`)
   - Stop conditions at both start and end of prompt for high-stakes skills

## Notes

### Why this is a separate ticket from ISS-010 rework

Bundling them would let the implementer use the convention change as cover for the bloat ("we raised the limit, so we comply now") without proving the new content is signal-positive. Keeping them separate forces ISS-010 to honestly answer "did this PR improve signal density?" against the *current* rule first. ISS-013 then provides legitimate breathing room for the next iteration.

### Web research conducted 2026-04-07

Sources reviewed:
- Anthropic Claude Code Skills documentation: skill structure, progressive disclosure, frontmatter spec
- Anthropic skill authoring best practices: naming, conciseness, security, testing
- Claude Code subagent design patterns: tool scoping, model selection, context preservation
- 2026 prompt engineering guidance on agent reliability, stop conditions, attention to start/end of context
- Glen Rhodes "Agent reliability comes from information architecture, not prompt quality" — most agent failures are context failures, not model failures
- Multi-agent safety guidance: "a collection of safe agents does not guarantee a safe collection of agents" — reinforces ISS-001's importance

### Adjacent improvements identified during research (file as separate tickets if pursued)

These are framework-level deltas surfaced by the research that should NOT be folded into ISS-013, to keep this ticket focused. Listed here so they are not lost:

- **ISS-018:** Audit and standardize skill `name` field — currently mixed gerund/noun form. Anthropic recommends gerunds.
- **ISS-019:** Add `allowed-tools` to skill frontmatter for skills that should never write to source. Enforces tool scoping at the skill layer, not just the agent layer.
- **ISS-016:** Document intentional divergence from Anthropic's "slash commands should be simple shortcuts" guidance. Our slash commands are full phase orchestrators.
- **ISS-017:** Add a "stop conditions reinforcement footer" to high-stakes skills, separate from the size revision in ISS-013.
- **ISS-001 reinforcement:** The Codex rubber-stamp pattern on ISS-010 is direct evidence for ISS-001 (invariants-audit skill). The current review layer does not catch cross-layer contradictions (PRD says X → implementation does opposite → reviewer says proceed). ISS-001 should explicitly include "for each AC, cite the evidence it is met or violated" as part of the review method.
