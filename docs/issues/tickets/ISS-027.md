# ISS-027: Harden Codex code review method for install-path, sync-drift, and test-truthfulness checks

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `review-quality`, `documentation`, `testing`
- **Created:** 2026-04-11
- **Reporter:** Linoy
- **Feature area:** `codex/reviewers/review-code.md`, Codex review workflow, review quality gates

## Description

Recent Codex re-reviews of ISS-009 exposed gaps in the current `review-code` method:

1. **Install/upgrade path checks were missed.** The review focused on changed helper code and tests, but did not follow the new dependency into `init.sh` / `upgrade.sh`, even though the diff introduced a new runtime dependency on `.claude/helpers/resolve-feature.js`.

2. **False-confidence sync tests were underweighted.** A test named like a source/installed sync check only verified one narrow property (`require.main`), not the full parity its name implied.

3. **Edge-case enumeration was incomplete.** The parser review caught some malformed input shapes, but only after multiple re-reviews surfaced additional cases. The review method should require a small malformed-input matrix for parsers and validators.

These are not product bugs in a single feature branch. They are review-method weaknesses in the Codex layer and should be fixed at the reviewer prompt/process level so future reviews catch these issues earlier and more consistently.

## Acceptance Criteria

- [ ] **AC1 (Installer/source-of-truth rule):** `codex/reviewers/review-code.md` explicitly requires reviewers to inspect installer, upgrade, generation, or source-of-truth files when a diff introduces a new helper/script/path dependency.
- [ ] **AC2 (Test-truthfulness rule):** The review method explicitly requires checking that a test’s assertion body proves what its name claims, especially for sync/parity/regression tests.
- [ ] **AC3 (Parser edge-case rule):** The review method adds a parser/validator checklist requiring reviewers to enumerate malformed-input shapes changed by the diff and verify direct coverage for each important shape.
- [ ] **AC4 (Unchanged-file scope guidance):** The prompt clarifies that unchanged files are in scope when they install, generate, copy, or otherwise operationalize a changed file.
- [ ] **AC5 (Process docs):** `docs/memory/codex-rules.md` or `docs/memory/review-process.md` is updated to reflect the stronger review expectations for Codex code reviews.
- [ ] **AC6 (Regression protection):** At least one deterministic test or fixture-backed check protects the updated review prompt/method from regressing silently.
- [ ] **AC7 (Installer coverage contract test):** Add a deterministic test that globs `skills/*/SKILL.md`, `commands/*.md`, and `hooks/*.js` in the source tree and asserts each has a corresponding copy line in `init.sh` (and optionally `upgrade.sh`). This prevents the recurring BLOCKING pattern where new files are added to the source tree but never copied to target projects (observed in ISS-009 with `resolve-feature.js` and in dogfood with `CHANGELOG.md`/`release-notes/`).

## Notes

### Evidence

- ISS-009 required multiple Codex re-reviews before install-path risk was identified.
- A narrow sync test around `checkpoint.js` created false confidence while meaningful drift remained between source and installed copies.
- Malformed-input handling in `resolve-feature.js` improved incrementally as new edge cases were discovered in review, rather than from one complete review pass.

### Evidence for AC7 (installer coverage gap)

Two BLOCKING findings across separate features stemmed from the same root cause — new files added to the source tree but never copied by `init.sh`:
- ISS-009: `resolve-feature.js` added to commands but `init.sh` doesn't copy it; fresh installs have broken phases 2-7
- Dogfood: `CHANGELOG.md` and `release-notes/` required by Phase 7 but never bootstrapped by installer

A contract test that asserts "every source file has a copy line in init.sh" would have caught both.
