# ISS-029: Add pipeline tier routing, ticket fidelity, and clarification checkpoints

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `workflow`, `pipeline`, `documentation`
- **Created:** 2026-04-11
- **Updated:** 2026-04-13
- **Reporter:** Linoy
- **Feature area:** Command UX, PRD generation, architecture review loop, pipeline routing

## Description

The current pipeline has two problems:

1. **All changes run the same 7-phase pipeline.** A one-line convention fix gets the same PRD → architecture → test design → security → implement → review → document treatment as a new cross-cutting skill. This wastes tokens and operator time on small changes while providing no additional safety.

2. **`/specify` and `/architect` are one-shot artifact generators.** They don't ask clarification questions when the request is ambiguous, don't verify PRD fidelity against the source ticket, and don't pause for human review before finalizing.

This ticket addresses both:

### Pipeline tier routing (infer-with-override)

The agent infers a pipeline tier from context and states it at the start. The operator can override with `/pipeline full` (or similar) if the inference is wrong.

| Tier | Example | Phases |
|------|---------|--------|
| **Convention/config** | Line limit change, settings tweak, ticket update | Direct commit, no pipeline |
| **Bugfix** | Fix a broken regex, update a stale reference | 5 (implement) → 6 (review) |
| **Small feature** | Add one rule to a skill, update one command | 1 → 3 → 5 → 6 |
| **Full feature** | New skill, cross-cutting change, multi-ticket | 1 → 2 → 3 → 4 → 5 → 6 → 7 |

Inference heuristics:
- Ticket type = Bug → Bugfix tier
- Ticket type = Architecture, touches only docs/CLAUDE.md → Convention tier
- No ticket reference, change described in one sentence → Convention tier
- Ticket type = Feature, single skill/command → Small feature tier
- Ticket type = Feature, multiple files/skills/commands → Full feature tier
- Any mention of auth, security, user input, API keys → Escalate to full

### Clarification checkpoints and ticket fidelity

- `/specify` asks clarification questions when the request is ambiguous
- `/specify` transcribes ticket ACs rather than paraphrasing them
- `/architect` presents proposed architecture for review before finalizing

## Acceptance Criteria

- [ ] **AC0 (`/specify` ticket fidelity check):** When `/specify` receives a ticket reference (e.g., ISS-NNN), the product-owner agent must read the ticket's acceptance criteria and transcribe them into Given/When/Then format — not paraphrase, broaden, or weaken them. If the PRD AC diverges from the ticket AC in scope, severity, or specificity, the agent must flag the divergence explicitly as an assumption rather than silently drifting. This prevents the recurring pattern where PRDs soften ticket intent (observed in ISS-022, ISS-013, ISS-026). Additionally:
  - When citing a project convention (e.g., line budgets, naming rules), the agent must verify the cited value against the current CLAUDE.md — do not rely on remembered or stale values. (Evidence: review-hardening PRD cited "~300 lines" for code-review budget when ISS-013 changed the convention to 150 prose / 250 total.)
  - After writing all ACs, the agent must check for internal contradictions — no two ACs should make mutually exclusive claims about the same field or behavior. (Evidence: review-hardening AC14 required `source_spec` in handoff while AC15 allowed review to proceed when it's missing, undermining the invariant.)
  - When a ticket uses open-ended scope ("and any other relevant X"), the agent must either enumerate the candidates or ask the user which ones apply — do not silently drop the open-ended clause. (Evidence: review-hardening named only 2 roles when ISS-014 said "and any other relevant gate-review roles.")
- [ ] **AC1 (`/specify` clarification gate):** `/specify` is updated so that when the feature request is ambiguous, the command explicitly asks the user the minimum necessary clarification questions before finalizing `prd.md`.
- [ ] **AC2 (Question discipline):** The specify-phase clarification step is constrained to only ask questions that materially affect the PRD or downstream implementation, not generic brainstorming prompts.
- [ ] **AC3 (Clarification outcome handling):** After the user answers, `/specify` incorporates the answers into the PRD and proceeds normally with the feature slug and handoff flow.
- [ ] **AC4 (`/architect` review checkpoint):** `/architect` is updated so it produces or summarizes a proposed architecture and then requests user review/feedback before finalizing `architecture.md`, committing, and writing the next-phase handoff.
- [ ] **AC5 (Feedback incorporation):** `/architect` can revise the proposed architecture based on user feedback before the final artifact is written.
- [ ] **AC6 (No hidden auto-advance):** If the user has not yet reviewed/responded at the required checkpoint, the command does not silently finalize the artifact or advance the pipeline.
- [ ] **AC7 (Workflow compatibility):** The updated behavior is documented clearly enough that users understand when a command is waiting for clarification/review versus when a phase is complete.
- [ ] **AC8 (Verification):** Deterministic tests or contract checks cover the new clarification/review checkpoint language in the command files and any supporting workflow instructions.

### Pipeline Tier Routing

- [ ] **AC9 (Tier inference):** When the agent receives a task (ticket reference, free-text request, or `/specify` invocation), it infers a pipeline tier (convention, bugfix, small feature, full feature) from context signals (ticket type, file count, change scope, security surface) and states the inferred tier at the start of the response before proceeding.
- [ ] **AC10 (Tier override):** The operator can override the inferred tier at any time (e.g., "run the full pipeline" or "this is just a convention fix"). The agent adjusts the phase set accordingly.
- [ ] **AC11 (Security escalation):** If the change touches auth, security, user input handling, API keys, or prompt injection surfaces, the agent must escalate to the full feature tier regardless of other signals — never infer convention or bugfix for security-sensitive changes.
- [ ] **AC12 (Tier documentation):** The tier definitions, inference heuristics, phase sets, and override mechanism are documented in CLAUDE.md or the relevant pipeline documentation so operators understand the routing.
- [ ] **AC13 (Tier in handoff):** The inferred (or overridden) tier is recorded in `handoff.json` so downstream phases know which phase set applies and don't accidentally run skipped phases.
- [ ] **AC14 (Tier verification):** Deterministic tests verify that the tier routing table and heuristics are present in the relevant command/skill files and that `handoff.json` schema accepts the tier field.

## Notes

### Why this matters

- Ambiguity caught during `/specify` is cheaper to resolve than reworking tests, security review, implementation, or review findings later.
- Architecture quality improves when the user can sanity-check the proposed design before the pipeline hardens it into downstream test and implementation work.
- This makes the human collaboration model explicit instead of relying on ad hoc interruption.
- Running 7 phases on a 5-minute convention change wastes tokens and operator time. Tier routing matches pipeline rigor to change risk — small changes get fast paths, security-sensitive changes always get full scrutiny.

### Evidence for AC0 (PRD-vs-ticket drift)

Across 3 features, Codex PRD reviews found the product-owner agent consistently weakening or broadening ticket ACs:
- ISS-022: AC7 dropped "flag as gap" behavior; AC5 weakened from "asserts visible effect" to "imports entry point"
- ISS-013: AC4 locked pilot to "preserve all content" — opposite of ticket intent; AC5 hardcoded "12 skills" when repo has 9
- ISS-026: AC2 broadened specific wildcard `review-codex-*.md` to generic "Codex review files"

Root cause: `/specify` reads the ticket but paraphrases ACs instead of transcribing them, and no phase currently verifies PRD fidelity against the source ticket.

### Relationship to other tickets

- ISS-028 improves feature selection and backlog-state UX; ISS-029 improves the human feedback loop inside the early pipeline phases themselves.
- ISS-022 and ISS-023 both benefit from stronger early-phase artifacts, since better clarified PRDs and reviewed architectures should improve downstream test design and architecture precision.
- ISS-033 adds source-spec verification at *review* time; AC0 here prevents drift at *authoring* time — complementary, not overlapping.
