# ISS-029: Add clarification checkpoints to `/specify` and review checkpoints to `/architect`

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `workflow`, `pipeline`, `documentation`
- **Created:** 2026-04-11
- **Reporter:** Linoy
- **Feature area:** Command UX, PRD generation, architecture review loop

## Description

The current pipeline treats `/specify` and `/architect` as mostly one-shot artifact generators. That keeps the flow simple, but it pushes too much responsibility onto the user to front-load all ambiguity before the command starts and to manually interrupt the flow later if the generated output needs adjustment.

This ticket adds explicit human review checkpoints to the front of the pipeline:

1. **`/specify` should ask necessary clarification questions before finalizing the PRD** when the request is ambiguous or underspecified.
2. **`/architect` should present the proposed architecture for review and feedback before the architecture artifact is finalized and the handoff advances.**

The goal is to improve correctness and reduce avoidable rework by making the human feedback loop a first-class part of the command contract rather than an informal side channel.

## Acceptance Criteria

- [ ] **AC1 (`/specify` clarification gate):** `/specify` is updated so that when the feature request is ambiguous, the command explicitly asks the user the minimum necessary clarification questions before finalizing `prd.md`.
- [ ] **AC2 (Question discipline):** The specify-phase clarification step is constrained to only ask questions that materially affect the PRD or downstream implementation, not generic brainstorming prompts.
- [ ] **AC3 (Clarification outcome handling):** After the user answers, `/specify` incorporates the answers into the PRD and proceeds normally with the feature slug and handoff flow.
- [ ] **AC4 (`/architect` review checkpoint):** `/architect` is updated so it produces or summarizes a proposed architecture and then requests user review/feedback before finalizing `architecture.md`, committing, and writing the next-phase handoff.
- [ ] **AC5 (Feedback incorporation):** `/architect` can revise the proposed architecture based on user feedback before the final artifact is written.
- [ ] **AC6 (No hidden auto-advance):** If the user has not yet reviewed/responded at the required checkpoint, the command does not silently finalize the artifact or advance the pipeline.
- [ ] **AC7 (Workflow compatibility):** The updated behavior is documented clearly enough that users understand when a command is waiting for clarification/review versus when a phase is complete.
- [ ] **AC8 (Verification):** Deterministic tests or contract checks cover the new clarification/review checkpoint language in the command files and any supporting workflow instructions.

## Notes

### Why this matters

- Ambiguity caught during `/specify` is cheaper to resolve than reworking tests, security review, implementation, or review findings later.
- Architecture quality improves when the user can sanity-check the proposed design before the pipeline hardens it into downstream test and implementation work.
- This makes the human collaboration model explicit instead of relying on ad hoc interruption.

### Relationship to other tickets

- ISS-028 improves feature selection and backlog-state UX; ISS-029 improves the human feedback loop inside the early pipeline phases themselves.
- ISS-022 and ISS-023 both benefit from stronger early-phase artifacts, since better clarified PRDs and reviewed architectures should improve downstream test design and architecture precision.
