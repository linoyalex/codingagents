# ISS-015: Add first-class post-implementation QA verification stage

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `pipeline`, `qa`, `verification`
- **Created:** 2026-04-07
- **Reporter:** Linoy
- **Feature area:** Pipeline stage design, QA workflow, release confidence

## Description

The current pipeline has a strong test-design phase and a code-review phase, but QA's
post-implementation verification role is not operationalized as a first-class stage or
command. `ROLE_QA.md` already references a secondary "Phase 5b — VERIFY" mode, but the
pipeline does not clearly expose that as a repeatable, enforceable workflow step.

We should add an explicit post-implementation QA verification stage so the pipeline can
challenge the implemented behavior separately from test design and separately from code
review. This stage should act as an independent adversarial validation pass focused on
acceptance criteria, release readiness, and likely user-facing or operational failures.

## Acceptance Criteria

- [ ] The pipeline defines a first-class post-implementation QA verification stage or
      command with a clear place in the workflow
- [ ] The QA verification stage runs in separate context from the implementation authoring
      phase
- [ ] The stage has clear inputs, outputs, and pass/fail expectations
- [ ] QA verification artifacts are written to predictable feature-local locations under
      `docs/features/<feature>/`
- [ ] The stage distinguishes between:
  - test-design responsibilities
  - implementation responsibilities
  - release/readiness verification responsibilities
- [ ] The new QA verification stage integrates cleanly with handoff and pipeline
      advancement rules
- [ ] Commands, roles, and verification docs are updated to reflect the new stage
- [ ] Deterministic contract tests cover the new stage behavior where practical

## Notes

This feature may result in either:
- a new explicit pipeline phase, or
- a well-defined sub-phase / gate with equivalent workflow clarity

Key design question:
- whether QA verification should happen before code review, after code review, or as a
  dedicated release-readiness gate after implementation fixes land

Likely files to update:
- `ROLE_QA.md`
- pipeline commands
- `PIPELINE_GUIDE.md`
- handoff conventions
- feature artifact conventions for QA verification output
