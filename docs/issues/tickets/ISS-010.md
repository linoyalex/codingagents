# ISS-010: Refresh all skills with current best practices and stronger robustness guards

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `skills`, `quality`, `developer-experience`
- **Created:** 2026-04-07
- **Reporter:** Linoy
- **Feature area:** `skills/`, prompt quality, workflow resilience

## Description

The skill library is now a core part of how the framework guides specification, test
design, implementation, security review, verification, and documentation. As the system
has grown, some skills reflect older prompt assumptions, weaker guardrails, or less
robust workflows than we now want to encourage.

We should refresh the skill set so the skills consistently reflect current best
practices, are more robust under edge cases, and provide clearer guidance for failure
handling, verification, and context discipline. This should be treated as a framework
quality upgrade, not a one-off wording pass.

This work should explicitly optimize for **agent reliability**, not just style or
completeness. The recurring problem is that agents miss critical context, improvise when
outputs are under-specified, or follow happy-path instructions while failing on ambiguity,
stale state, or malformed artifacts.

The goal is therefore:

- improve reliability per token
- reduce avoidable context and output-shape mistakes
- strengthen failure handling and determinism
- avoid bloating the context window
- avoid confusing agents with too many overlapping rules

The preferred approach is **fewer, sharper, higher-leverage instructions**, not longer
skills by default.

## Acceptance Criteria

- [ ] Audit every first-party skill under `skills/` for outdated guidance, weak
      instructions, and missing robustness checks
- [ ] The first implementation pass focuses on the core skills first:
  - `prd-writing`
  - `architecture-decision`
  - `tdd`
  - `verification-gate`
- [ ] Update skills to reflect current best practices for:
  - correctness-first execution
  - failure-path handling
  - verification and regression protection
  - context discipline and scoped reading
  - security and safety expectations where relevant
- [ ] Include companion command changes where skill improvements require command-level
      alignment for reliability or clarity
- [ ] Remove or tighten instructions that are vague, contradictory, or likely to produce
      brittle behavior
- [ ] Add stronger guidance where skills should explicitly handle:
  - missing inputs
  - malformed artifacts
  - stale state / stale handoff assumptions
  - ambiguous or conflicting signals
  - deterministic verification expectations
- [ ] Preserve the intended role boundaries between commands, roles, and skills rather
      than moving command logic into skills
- [ ] Identify which skill changes require matching updates in commands, reviewer prompts,
      tests, or docs, and make those companion changes where needed
- [ ] Add or update deterministic coverage for any skills whose behavior is enforced by
      tests, templates, or verification logic
- [ ] Changes materially improve reliability without meaningfully bloating prompt size or
      adding low-value rule sprawl
- [ ] New instructions are prioritized and front-loaded so the highest-leverage guidance
      is easier for agents to follow under context pressure
- [ ] Output expectations are more explicit where under-specification currently causes
      agents to improvise or miss required structure
- [ ] Document the major skill-library improvements in a way maintainers can review
      easily, such as a short summary or review artifact

## Notes

This ticket is about improving the quality and resilience of the skill layer as a
system. It should include both content quality and operational robustness.

Product direction captured for this ticket:

- Reliability matters more than adding more advice.
- The fix is not "more rules"; it is clearer, more compact, more deterministic guidance.
- Critical instructions should be front-loaded and easy to scan.
- Skills should better handle ambiguity, stale state, and malformed inputs.
- Companion command changes are in scope when needed to keep the workflow aligned.
- The work should materially improve outcomes while staying disciplined about context
  window size and instruction density.

Examples of likely improvement areas:
- stronger failure-mode guidance in verification-oriented skills
- clearer review heuristics in audit/review skills
- better TDD and implementation guardrails around ambiguity and stale context
- more explicit output-shape expectations where downstream tooling depends on them

The goal is not to make skills longer by default. Prefer sharper, clearer, more reliable
instructions over adding generic text. A small template addition is acceptable when it
materially improves reliability, but the default should be to increase signal, not volume.
