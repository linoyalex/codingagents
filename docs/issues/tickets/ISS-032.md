## Description

The pipeline convention requires running `/status` at the start of each session to detect the current pipeline phase, active feature, and next action. In practice this is easy to forget, and the agent starts working without knowing where the pipeline left off â€” leading to wrong-phase execution, stale handoff reads, or redundant work.

This ticket adds automatic `/status` execution (or equivalent status output) when a fresh context window starts, so the agent and user are immediately oriented without a manual step.

## Acceptance Criteria

- [ ] **AC1 (Automatic status on session start):** When a new session begins, the agent automatically receives pipeline status context (active feature, current phase, next action, handoff state) without the user needing to type `/status`.
- [ ] **AC2 (Hook-based implementation):** Status context is injected via the `SessionStart` hook (`restore-context.js`) as additional context alongside the existing handoff loading. This avoids consuming a user turn.
- [ ] **AC3 (Lightweight output):** The injected status is compact (~10-20 lines) â€” feature name, current phase, next recommended command, and any warnings (stale handoff, missing artifacts). Not a full `/status` dump.
- [ ] **AC4 (No duplication):** If the user explicitly runs `/status` after session start, it does not produce confusing duplicate output. The hook output and the command output should be complementary, not redundant.
- [ ] **AC5 (Opt-out):** The behavior can be disabled via a setting or environment variable for sessions that intentionally start without pipeline context (e.g., ad-hoc debugging).
- [ ] **AC6 (No regression):** Existing `restore-context.js` behavior (handoff loading, context archive restoration) continues to work unchanged.

## Notes

### Current behavior

`restore-context.js` already runs on `SessionStart` and loads `handoff.json` if present. It outputs the handoff context but does not run phase detection or print the next recommended action. The `/status` command does both â€” it reads `pipeline-checkpoint.json` and `handoff.json`, detects the phase from artifacts, and prints a status summary.

### Implementation approach

The simplest path is to extend `restore-context.js` to include a compact status summary when a valid handoff exists. This avoids adding a second hook and keeps the SessionStart output unified. The status logic already exists in `checkpoint.js` (`detectPhase`, `phaseFromHandoff`) â€” it can be imported.

### Why not just tell the agent to run /status?

The `/status` command consumes a turn and tokens. A hook-based approach injects the context before the first turn, so the agent starts oriented without any user interaction.
