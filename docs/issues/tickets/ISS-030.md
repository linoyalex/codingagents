# ISS-030: Introduce semantic versioning and tie backlog planning to major/minor/patch releases

- **Type:** Architecture
- **Priority:** P2 — Medium
- **Label:** `framework`, `workflow`, `documentation`, `deployment`
- **Created:** 2026-04-11
- **Reporter:** Linoy
- **Feature area:** Release/versioning policy, backlog planning, changelog/release workflow

## Description

The `codingagents` repo currently has versions like `v5`, but it does not have a clearly defined semantic versioning model that ties backlog work to the type of release it should land in. That makes it harder to answer questions like:

- is this issue a breaking change, a feature, or a patch?
- which backlog items are intended for the next major vs minor vs patch release?
- how should multiple open tickets roll up into a versioned roadmap?
- when should changelog, release notes, and version bumps occur?

This ticket introduces semantic versioning for the repo and connects backlog planning to release intent so tickets can be associated with a target release type such as major, minor, or patch.

It also establishes the canonical baseline that the `version5-codex+token-governance` line defines major version `5`. For historical cleanup, the repo should treat that branch line and its merge into `main` as the `5.0.0` baseline, then map the later transitional `0.x` release artifacts into the canonical `5.x` series.

## Notes

### Why this matters

- Semantic versioning makes the framework easier to reason about for both maintainers and consumers.
- Backlog tickets are currently prioritized by urgency and sequencing, but not by release impact.
- Several open tickets affect framework contracts and upgrade behavior; they should be classifiable as major/minor/patch work rather than handled ad hoc.

### Canonical `5.x` mapping

- Treat the `version5-codex+token-governance` line as the `5.0.0` generation baseline.
- Canonical mapping of the current published release artifacts:
  - former `0.5.0` -> `5.1.0` (`ISS-010`, reliability refresh)
  - former `0.6.0` -> `5.1.1` (`ISS-009`, fail-closed feature resolution bug fix)
  - former `0.7.0` -> `5.2.0` (`ISS-026`, artifact timestamps)
  - former `0.8.0` -> `5.3.0` (`ISS-013`, skill size convention / progressive disclosure)
  - former `0.9.0` -> `5.4.0` (`ISS-022`, integration-test coverage hardening)
- During the transition period, some release dates reflect when work was authored on the v5 branch line rather than when the history was later normalized under `5.x`. The policy should document that explicitly instead of pretending the transition never happened.
- If historical filenames are not renamed immediately, the release policy must at minimum define an explicit compatibility/mapping table so operators can translate old `0.x` labels into canonical `5.x` versions.

### Proposed semver heuristic

- **Patch:** restores intended behavior without changing the expected operator workflow or required artifact set. Typical cases: bug fixes, installer omissions, regression fixes, sync-drift fixes.
- **Minor:** adds a backward-compatible framework capability, new optional artifact, stronger guidance, new review surface, or a new required workflow step that stays within the same overall framework generation.
- **Major:** changes the mental model or contract of the framework in a way that can invalidate existing in-flight work or require coordinated migration across commands, handoff schema, install paths, or backlog/release conventions.

Suggested `/specify` heuristic:

1. Detect whether the ticket changes command contracts, handoff schema, install/upgrade behavior, required artifacts, or phase/gate behavior.
2. Propose `patch`, `minor`, or `major` with a short rationale:
   - `patch` if behavior is being corrected to match existing expectation
   - `minor` if a new capability or gating factor is introduced without resetting the framework generation
   - `major` if migration or coordinated rollout is required
3. Ask the operator to confirm or override the proposal before finalizing the PRD.

### Current backlog release proposal

- Group tightly related reliability tickets into shared `5.x` minor releases rather than issuing one minor per ticket when the operator-facing story is one coherent upgrade.
- `5.5.0` — review-hardening bundle (`ISS-024`, `ISS-014`, `ISS-033`)
- `5.6.0` — command/skill correctness bundle (`ISS-036`, `ISS-027`, `ISS-029`)
- `5.7.0` — invariants and review-traceability bundle (`ISS-001`, `ISS-006`, `ISS-037`)
- `5.8.0` — workflow ergonomics bundle (`ISS-028`, `ISS-032`, optionally `ISS-031`)
- `5.9.0` — release/install management bundle (`ISS-007`, `ISS-008`, `ISS-030`)
- Patch releases should be reserved for post-release regressions that do not add new workflow concepts. Example: if `5.4.0` ships with a broken install copy path, fix as `5.4.1`.

### Upgrade warning policy

- When a release introduces a new required gate, artifact, schema field, or command invocation rule, the release notes and `README.md` must prominently warn that upgrading agents in the middle of an active feature cycle is strongly discouraged.
- The warning should explain the consequence in plain language, for example:
  - existing feature artifacts may not satisfy new gates
  - in-flight handoffs may fail new schema validation
  - earlier test-design output may be missing newly required artifact types
- For these releases, the documented recovery path should include one of:
  - finish the current feature before upgrade
  - restart from the last stable phase after upgrade
  - run a documented migration/repair checklist before continuing

### Phased implementation plan

1. **Phase 1 — Policy and history normalization**
   - Create `RELEASE.md` as the canonical release-process guide.
   - Document the `5.0.0` baseline and the transitional `0.x` -> `5.x` mapping.
   - Update `CHANGELOG.md` and the existing `release-notes/` artifacts to canonical `5.x` numbering.
2. **Phase 2 — Operator-facing release guidance**
   - Update `README.md` and `QUICKSTART.md` with the current `5.x` line, upgrade-safety guidance, and explicit warnings for releases that add new gates or required artifacts.
   - Ensure the current published release note calls out new gating factors prominently.
3. **Phase 3 — Backlog and planning integration**
   - Add release-impact metadata conventions for new tickets and grouped release planning.
   - Decide where target release and semver impact live (ticket fields, backlog columns, PRD, or handoff).
4. **Phase 4 — Pipeline assistance**
   - Extend `/specify` to propose semver impact with rationale and operator confirmation.
   - Carry the chosen release intent into the appropriate downstream artifacts.
5. **Phase 5 — Verification and automation**
   - Add deterministic checks for semver metadata shape and any machine-checked release-planning conventions.
   - Update release-generation/documentation workflows once the policy is stable.

### Release process guide note

- `RELEASE.md` should be maintained as the canonical release-process guide for the `5.x` line and kept in sync with `CHANGELOG.md`, release notes, and any future semver automation.
