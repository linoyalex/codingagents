# ISS-030: Introduce semantic versioning and tie backlog planning to major/minor/patch releases

- **Type:** Architecture
- **Priority:** P2 — Medium
- **Label:** `framework`, `workflow`, `documentation`, `deployment`
- **Created:** 2026-04-11
- **Reporter:** Linoy
- **Feature area:** Release/versioning policy, backlog planning, changelog/release workflow

## Description

The `codingagents` repo currently has versions like `v5`, but it does not have a clearly defined semantic versioning model that ties backlog work to the type of release it should land in. That makes it harder to answer questions like:

- is this issue a breaking change, a feature, or a patch?
- which backlog items are intended for the next major vs minor vs patch release?
- how should multiple open tickets roll up into a versioned roadmap?
- when should changelog, release notes, and version bumps occur?

This ticket introduces semantic versioning for the repo and connects backlog planning to release intent so tickets can be associated with a target release type such as major, minor, or patch.

## Acceptance Criteria

- [ ] **AC1 (Semver policy):** A documented semantic versioning policy is added for `codingagents`, defining what counts as a major, minor, and patch change in this framework.
- [ ] **AC2 (Backlog metadata):** The backlog system is extended so tickets can be tagged or categorized by intended release impact/type (major, minor, patch).
- [ ] **AC3 (Ticket guidance):** New-ticket conventions document when and how to assign release intent to a ticket.
- [ ] **AC4 (Planning visibility):** `docs/issues/backlog.md` or a related planning document surfaces release intent clearly enough to support roadmap and batching decisions.
- [ ] **AC5 (Workflow integration):** The versioning policy is connected to changelog/release-note/version-bump workflow guidance so release planning is not just a label with no downstream effect.
- [ ] **AC6 (Backward compatibility guidance):** The policy explicitly covers framework-breaking changes such as command contract changes, artifact path changes, or install/upgrade behavior changes.
- [ ] **AC7 (Verification):** Deterministic tests or contract checks protect the chosen backlog/version metadata format and any release-planning rules that become machine-checked.

## Notes

### Why this matters

- Semantic versioning makes the framework easier to reason about for both maintainers and consumers.
- Backlog tickets are currently prioritized by urgency and sequencing, but not by release impact.
- Several open tickets affect framework contracts and upgrade behavior; they should be classifiable as major/minor/patch work rather than handled ad hoc.

### Relationship to other tickets

- ISS-026, ISS-027, ISS-028, and ISS-029 improve process quality and workflow ergonomics; ISS-030 adds release-planning structure on top of that work.
- This should align with changelog and release-note generation expectations already used in Phase 7 documentation work.
