# ISS-047: Configure default effort level and plan mode per pipeline command

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `commands`, `developer-experience`, `pipeline`
- **Created:** 2026-04-15
- **Reporter:** Linoy
- **Feature area:** `commands/*.md`, role frontmatter, `hooks/`

## Description

Different pipeline commands have different cognitive demands. `/specify` benefits from plan mode (alignment before writing) and high effort (comprehensive thinking). `/document` is mechanical template-filling that needs neither. Currently, effort level and plan mode are session-level settings that the operator must toggle manually before each command — there's no way to declare per-command defaults.

This creates two problems:
1. **Operator friction** — the operator must remember to set `/effort high` and `/plan` before `/specify`, then switch back for other commands
2. **Quality variance** — when the operator forgets, the agent produces lower-quality output on commands that need deep reasoning (specify, architect, review) or skips the alignment step (plan mode) on commands that benefit from it

### Desired behavior

Each command declares its preferred effort level and plan mode in frontmatter or a convention section. When the command is invoked, the runtime (or a hook) applies the defaults before execution and restores the previous settings after.

### Current Claude Code limitations

As of the current Claude Code runtime, effort level and plan mode are operator-controlled — there is no programmatic API for agents or hooks to toggle them. This ticket should be implemented in the most effective way available:

1. **Frontmatter declaration** — add `effort` and `plan-mode` fields to command YAML frontmatter as the canonical source of truth
2. **Hook-based enforcement** — if Claude Code adds hook support for mode switching, use a `PreToolUse` hook on the Skill tool to read frontmatter and apply settings
3. **Operator reminder (interim)** — until programmatic enforcement exists, a `PreToolUse` hook prints a reminder when the current effort/plan settings don't match the command's declared defaults

The interim reminder approach ensures the convention is established and documented even before the runtime supports automatic switching.

## Acceptance Criteria

- [ ] **AC1 (Frontmatter convention):** All pipeline commands include `effort` and `plan-mode` fields in their YAML frontmatter. Values: `effort: low | medium | high`, `plan-mode: true | false`. These are the canonical declaration of each command's preferred settings.
- [ ] **AC2 (Default assignments):** Each command's frontmatter reflects its cognitive demands. The specific assignments should be validated during implementation (see Design Considerations for initial recommendations).
- [ ] **AC3 (Interim hook reminder):** A `PreToolUse` hook on the Skill tool detects when a slash command is invoked and prints a reminder if the current session settings don't match the command's declared defaults. Example: "⚠ /specify recommends: effort=high, plan-mode=on. Current: effort=medium, plan-mode=off."
- [ ] **AC4 (Programmatic enforcement path):** When Claude Code's runtime adds support for programmatic effort/plan-mode switching (via hooks or agent API), the frontmatter declarations can be consumed automatically without changing command files. The design must not create a pattern that conflicts with future runtime enforcement.
- [ ] **AC5 (Ticket-type variation):** The `/specify` and `/architect` commands support ticket-type-aware effort defaults. Bug-fix specs and small features may use medium effort, while new features and architecture changes use high. The mechanism for detecting ticket type should use the ticket's `Type` field from the backlog or the feature request context.
- [ ] **AC6 (Restore after command):** The design ensures that command-specific settings don't leak into subsequent commands in the same session. Whether via hook restore or operator guidance, the previous effort/plan settings are restored after the command completes.
- [ ] **AC7 (No regression):** Existing command behavior is unchanged. The frontmatter fields are additive — commands work the same with or without the new fields until enforcement is available.
- [ ] **AC8 (Documentation):** CLAUDE.md or the relevant skill documents the frontmatter convention and explains how operators should use it (manual compliance now, automatic enforcement later).

## Notes

### Design considerations

Initial recommended defaults (to be validated during implementation):

| Command | Effort | Plan Mode | Rationale |
|---------|--------|-----------|-----------|
| `/specify` | high | true | PRD defines the entire feature scope — a wrong interpretation cascades through 6 phases. Plan mode catches misalignment before wasted work. |
| `/architect` | high | true | Structural decisions are hard to reverse. Always high+plan regardless of ticket type. |
| `/test-design` | medium | false | Follows patterns from PRD + architecture. Spec is already defined; low ambiguity. |
| `/security-gate` | high | false | Needs thorough analysis but is read-only (gate phase) — no decisions to pre-approve. |
| `/implement` | medium | false | TDD is self-correcting via RED/GREEN/REFACTOR. Plan mode would slow each micro-iteration. |
| `/review` | medium | false | Read-only analysis against known rules. No decisions to pre-approve. |
| `/document` | low | false | Mechanical template-filling. |
| `/status` | low | false | Read-only diagnostic. |

### Ticket-type variation (AC5)

Not all `/specify` runs need the same effort:

| Ticket type | `/specify` effort | `/specify` plan | Rationale |
|------------|-------------------|-----------------|-----------|
| New feature | high | true | Ambiguous scope, multiple valid interpretations, high cost of misalignment |
| Architecture change | high | true | Irreversible decisions need calibration |
| Bug fix | medium | false | Scope defined by the bug report — the "what" is clear |
| Refactor | medium | false | Scope typically well-defined (rename X, extract Y) |

`/architect` should always be high + plan regardless of ticket type.

The command could read the ticket type from:
- The backlog ticket file (if ISS-028 lands and provides ticket-aware selection)
- The feature request text (heuristic: "bug", "fix" → medium; "feature", "add" → high)
- An explicit flag (e.g., `/specify --effort medium Fix login timeout`)

The simplest interim approach: default to high, let the operator override with `/effort medium` before running. The frontmatter documents the default; the operator overrides when appropriate.
