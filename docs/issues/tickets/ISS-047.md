# ISS-047: Implement the first portability slice: configurable paths, outputs, diff/test commands, and work-type profiles

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `portability`, `configurability`, `workflow`, `developer-experience`
- **Created:** 2026-04-14
- **Reporter:** Linoy
- **Feature area:** Project config implementation, command behavior, portability slice

## Description

ISS-046 defines the shared architecture for project portability. This ticket is the first
implementation slice that should deliver the biggest practical improvement without waiting
for every adapter and every provider-neutral concern to land.

The goal is to make the framework meaningfully easier to use in non-`codingagents`
projects by addressing the top five portability gaps first:

1. a shared project-config resolution path instead of one-off assumptions
2. configurable artifact and documentation paths
3. configurable review diff base and automated test command selection
4. the ability to narrow or disable documentation-specific outputs where appropriate
5. configurable work-type/review profiles so a bugfix, a small ticket, and a large feature
   do not all have to feel identical

This ticket is intentionally a bounded slice. It should move the framework forward
significantly without trying to solve the full backlog-system adapter problem or the full
multi-agent routing problem in one branch.

## Acceptance Criteria

- [ ] **AC1 (Shared config entry point):** Introduce the first implemented shared
      project-config entry point and resolution helper described by ISS-046, with clear
      precedence between framework defaults and project overrides.

- [ ] **AC2 (Configurable project paths):** At least the following currently hardcoded
      locations become configurable through the shared project config while preserving the
      current defaults when omitted:
  - feature artifact root
  - ticket/backlog root
  - release-note root
  - changelog path
  - project guidance path(s)

- [ ] **AC3 (Configurable diff/test behavior):** Review and implementation flows can read
      project-configured defaults for:
  - review diff base / default branch
  - automated test command selection or discovery
  - any closely related path assumptions needed to make those commands meaningful

- [ ] **AC4 (Optional documentation outputs):** A project or profile can disable or narrow
      documentation-specific outputs such as changelog/release-note/README update work
      without breaking non-documentation phases or leaving command behavior ambiguous.

- [ ] **AC5 (Work-type profiles):** Implement at least a small set of bounded work-type
      profiles, such as:
  - `bugfix`
  - `small-ticket`
  - `standard-feature`
  - `epic`

  The profile behavior must define what changes in artifact expectations, documentation
  expectations, and review strictness, while preserving core safety invariants.

- [ ] **AC6 (Review strictness):** The first slice includes a concrete way to express
      stricter vs lighter review behavior within the supported profiles or config surface,
      and commands/skills that consume it behave deterministically.

- [ ] **AC7 (No silent fallback to codingagents-only paths):** The implemented commands and
      skills in this slice must fail clearly or use resolved config when a project changes
      the default locations, rather than silently drifting back to `docs/features/`,
      `docs/issues/`, `CHANGELOG.md`, or `main...HEAD`.

- [ ] **AC8 (Regression protection):** Deterministic tests cover the new config resolution,
      path overrides, documentation toggle behavior, diff/test-command overrides, and at
      least one profile-specific behavior difference.

- [ ] **AC9 (Scope boundary):** The implementation notes and docs make it explicit that:
  - backlog-system adapters beyond the default markdown convention remain in ISS-034
  - first-class multi-agent/provider routing remains in ISS-038
  - this ticket provides the shared portability slice they should build on

## Notes

### Why this is the right first slice

These changes remove the highest-friction assumptions when trying to use the framework in
another repository:

- fixed artifact paths
- fixed documentation outputs
- fixed `main...HEAD` review assumptions
- fixed test-command expectations
- one-size-fits-all workflow intensity

Addressing these first moves the framework meaningfully toward portability even before the
broader backlog-system and multi-agent work is complete.
