# ISS-034: Make backlog management skill configurable for different backlog systems

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `skills`, `backlog`, `developer-experience`
- **Created:** 2026-04-12
- **Reporter:** Linoy
- **Feature area:** `skills/backlog-management/SKILL.md`, `CLAUDE.md`

## Description

The current backlog management skill hardcodes a specific markdown-file-based backlog system (`docs/issues/backlog.md`, `in-progress.md`, `closed.md`, `tickets/ISS-NNN.md`). This works well for the codingagents repo itself, but when the framework is installed into a target project, that project may use a different backlog system — GitHub Issues, Linear, Jira, a different markdown layout, or no file-based backlog at all.

Both Claude and Codex need to know how to read, update, and triage the project's backlog as part of the development workflow. Currently this knowledge is baked into the skill with no extension point. When an agent moves a ticket to in-progress but forgets to remove it from the backlog (as happened with ISS-026), the root cause is partly that the lifecycle rules are implicit rather than enforced by a clear, project-configured convention.

This ticket makes the backlog management skill configurable so it can adapt to the project's backlog system while preserving the default markdown-based convention for projects that don't configure an alternative.

## Acceptance Criteria

- [ ] **AC1 (Skill-level backlog type registry):** `skills/backlog-management/SKILL.md` includes a `## Backlog Type Registry` section with a table listing supported backlog types. Each entry specifies: type name, and a pointer to a sibling reference file containing lifecycle rules, format rules, and agent commands for that type. Default entries: `markdown-files` (current behavior), `github-issues`, `custom`. The registry is loaded on demand when backlog operations are invoked — not always-loaded in CLAUDE.md. Follows the skill registry convention established by ISS-031.

- [ ] **AC2 (Default behavior preserved):** If no backlog configuration is specified, the skill uses the current markdown-file-based system exactly as it works today. No existing behavior changes for projects that don't opt in.

- [ ] **AC3 (Lifecycle completeness):** The skill clearly documents the full ticket lifecycle for each supported backlog type: create → triage → start (move to in-progress) → close. Each transition specifies which files/systems to update and which to remove from, preventing partial-move bugs like ISS-026.

- [ ] **AC4 (Agent-readable convention):** Each sibling reference file (e.g., `markdown-files.md`, `github-issues.md`) is structured so that both Claude and Codex can read it and know: where tickets live, how to read them, how to move them through states, and what format to use when creating new ones. The reference file is loaded on demand when the skill is invoked.

- [ ] **AC5 (GitHub Issues support):** For `github-issues` mode, the skill documents how agents should use `gh` CLI commands to create, label, close, and query issues. The markdown index files become optional or are replaced by issue queries.

- [ ] **AC6 (Custom backlog pointer):** For `custom` mode, a `custom.md` sibling reference file in `skills/backlog-management/` points to a project-defined conventions file (e.g., `docs/backlog-conventions.md`) that the agent reads for backlog operations. The skill provides the lifecycle framework; the project file provides the format details.

- [ ] **AC7 (No regression):** Existing tests and the current markdown-file backlog continue to work unchanged.

- [ ] **AC8 (Extensibility documentation):** The `## Backlog Type Registry` section includes a brief guide on how to add a new backlog type: (1) create a sibling reference file with lifecycle and format rules, (2) add an entry to the registry table, (3) the skill automatically picks it up. References ISS-031 as the established convention.

- [ ] **AC9 (Wave/batch placement on ticket creation):** When a new ticket is created, the skill optionally prompts the agent to place the ticket in an execution wave and/or batch. The agent should: (a) identify which wave the ticket belongs to by theme, (b) update the wave's description and remaining-tickets list, (c) if a batch execution plan exists, suggest where the ticket fits based on file overlap and dependencies. This step is optional — the agent may skip it if the operator defers placement — but the skill must surface the prompt so placement isn't silently forgotten. For markdown-files backlog type, this means updating the Execution Waves and Batch execution plan sections of `backlog.md`.

## Notes

### Current pain points

- ISS-026 was moved to `in-progress.md` but not removed from `backlog.md` — a partial-move bug. The lifecycle rules exist in the skill but aren't explicit enough about "remove from source when adding to destination."
- Target projects that use GitHub Issues have no guidance for how agents should interact with their backlog. The skill is silent on external backlog systems.
- Codex sessions need backlog context too. The skill should be readable by both Claude and Codex without assuming either agent's toolset.

### Design considerations

- **Skill registry pattern (ISS-031 convention):** The backlog type registry lives in `skills/backlog-management/SKILL.md`, loaded on demand. Each backlog type gets a sibling reference file (e.g., `markdown-files.md`, `github-issues.md`) containing lifecycle rules, format rules, and agent commands. This avoids CLAUDE.md budget pressure and keeps config colocated with the skill that uses it.
- **Active backlog type selection:** A project's active backlog type is specified in a single line in the registry table (e.g., `active: markdown-files`). The skill reads this on invocation and loads only the relevant reference file.
- For GitHub Issues mode, the `github-issues.md` reference file should document `gh issue list`, `gh issue create`, `gh issue close` etc. Labels could map to priority and type fields.
- The skill should separate the **lifecycle rules** (which are universal and live in SKILL.md) from the **format rules** (which are per-backlog-type and live in sibling reference files). This keeps the skill lean while allowing rich per-type guidance.
- Once ISS-046 lands, backlog-system selection should live in the shared project configuration model rather than becoming a one-off backlog-only config surface.

### Wave/batch placement gap (AC9 rationale)

During the QA test quality batch (Batch 2.75), multiple new tickets (ISS-052, ISS-053, ISS-054) were filed and added to the Open Tickets table but their wave descriptions were not consistently updated. ISS-054 was missed entirely until the user noticed. The backlog skill's lifecycle covers create → triage → start → close, but says nothing about wave/batch placement. This is a separate concern from ticket creation but happens at the same time and is easy to forget.

AC9 makes wave placement an explicit optional step in the ticket creation flow rather than relying on the agent to remember.

### Closed-but-referenced tickets convention (to formalize in AC3)

The current skill says closed tickets should be removed from `backlog.md` and added to `closed.md`. But in practice, closed tickets were left inline with strikethrough (`~~ISS-NNN~~`) because the batch execution plan references them for dependency context. This created drift from the defined lifecycle.

AC3 should formalize the convention:
- **Open Tickets table:** Closed tickets must be removed — no strikethrough rows. If an open ticket depends on a closed one, annotate the dependency cell with `(closed)` instead.
- **Batch execution plan:** Closed branches use `✅ MERGED` / `Closed <date>` annotations — this is fine and should be the documented convention. The batch plan is a historical record, not a live index.
- **Lifecycle rule to add:** "When closing a ticket that appears in both the Open Tickets table and a batch plan, remove it from the table but leave the batch plan annotation intact."
