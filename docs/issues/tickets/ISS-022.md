# ISS-022: Harden Phase 3 test design with integration-level coverage requirements

- **Type:** Feature
- **Priority:** P1 — High
- **Label:** `framework`, `skills`, `testing`, `pipeline-quality`
- **Created:** 2026-04-11
- **Reporter:** Linoy (via global memory audit of cross-project learnings)
- **Feature area:** `skills/tdd/SKILL.md`, `commands/test-design.md`, `PIPELINE_GUIDE.md`

## Description

Phase 3 (test design) has been producing unit/contract tests and E2E shells but skipping the integration level — tests that call the production entry point and assert the feature's effect is visible in the output. This gap caused the worst class of escaped defect observed: 275 passing unit tests with zero production wiring. The feature was "green" but not shipped.

Three related sub-problems are grouped here because they all modify what Phase 3 produces and what the TDD skill teaches:

1. **Missing integration tests.** No skill explicitly requires tests that call the production entry point. The TDD skill mentions "production-wiring test" in passing but doesn't define it or require it.

2. **Test fixtures not validated against production schemas.** Test fixtures have used stand-in data types (e.g., `type: 'shelving'` to model an accessory), causing tests to pass via the wrong rule. No skill requires reading the production schema before writing fixtures.

3. **Degenerate inputs not tested when loosening validation.** Widening a constraint (e.g., `z.enum` → `z.string`) admits empty strings, whitespace, and other degenerate values. No skill triggers additional tests when validation is loosened.

## Acceptance Criteria

- [ ] **AC1 (Three-level coverage):** `skills/tdd/SKILL.md` adds a "Three-level test coverage" section requiring: unit/contract tests (existing), at least one integration test per feature that calls the production entry point, and E2E shells. Integration tests use the naming convention `[feature].integration.test.*`.
- [ ] **AC2 (Integration test definition):** The skill clearly defines: "An integration test calls the production entry point (not the module directly) and asserts the feature's effect is visible in the output. A test that calls the module directly is a unit test, regardless of how many modules it touches."
- [ ] **AC3 (Fixture validation rule):** The TDD skill adds: "For any AC that references a real-world data type, read the production schema/type/enum definition before writing fixtures. Confirm fixture values match exactly. Do not invent stand-in shapes."
- [ ] **AC4 (Degenerate input rule):** The TDD skill adds: "When widening a validation constraint, list the degenerate values now admitted (empty, whitespace, max-length) and add a test for each."
- [ ] **AC5 (Phase 3 command update):** `commands/test-design.md` adds verification: "at least one test imports from a production entry point module."
- [ ] **AC6 (Pipeline guide update):** `PIPELINE_GUIDE.md` Phase 3 "What these tests cover" list includes integration tests.
- [ ] **AC7 (Architecture dependency):** The skill documents that integration tests require the architecture doc to include a "Call Chain" or "Integration Points" section (see ISS-023). If the arch doc doesn't include it, the QA agent should flag it as a gap.
- [ ] **AC8 (No regression):** Existing tests continue to pass. Skill stays within size budget (see ISS-013).

## Notes

### Evidence from closet-designer project

- 275 passing unit tests + 0 integration tests = validators that work perfectly in isolation and are never called in production. The test suite was green but the feature was not shipped.
- Test fixtures used `type: 'shelving'` to model an accessory — test passed via the wrong rule. Caught by Codex, not by the pipeline.
- `toTitleCase('')` returned `''` producing a blank label after widening `z.enum` to `z.string`. The empty string was newly admitted and should have been tested.

### Relationship to other tickets

- ISS-013 (skill size convention) should land first so the new content doesn't immediately violate the line budget.
- ISS-023 (architecture wiring diagrams) is a soft dependency — integration tests are most effective when the arch doc includes the call chain.
