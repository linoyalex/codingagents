# ISS-061: Add architecture review gate between Phase 2 and Phase 3

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `pipeline`, `review-quality`
- **Created:** 2026-04-16
- **Reporter:** RCA from ISS-001 invariants-audit Phase 1 Codex review
- **Feature area:** Pipeline gates, `commands/architect.md`, `codex/reviewers/review-architecture.md`

## Description

Phase 2 (Architect) output feeds directly into Phase 3 (Test Design) and Phase 5 (Implement). Phase 4 (Security Gate) reviews the architecture but only for security concerns — structural correctness, failure modes, hidden coupling, over-engineering, and boundary clarity are not in its scope.

The Codex architecture reviewer (`codex/reviewers/review-architecture.md`) already defines a comprehensive non-security review methodology covering:
- Structural correctness
- Failure modes and recovery
- Boundary and dependency clarity
- Simplicity vs over-engineering
- Data and scaling risks

This review is currently manual and out-of-pipeline. Wiring it as a gate would catch architecture-level issues before they propagate to test design and implementation.

### Evidence from prior features

- **ISS-036 RCA:** The architect agent committed fail-open defaults without self-challenge. This was caught later, not at architecture time. (Referenced in ISS-025 AC7.)
- **ISS-029 RCA:** Architecture silently tightened AC8 without updating the PRD — a PRD↔architecture delta that would have been caught by a review comparing architecture against PRD.
- **ISS-001 (current):** The architecture phase consumed a PRD with gaps (caught by Codex PRD reviewer). An architecture reviewer would have surfaced the same gaps when checking PRD↔architecture alignment.

### Asymmetry with Phase 4

Phase 4 (Security Gate) is the only formal review between Phase 2 and Phase 5. Its scope is explicitly security: OWASP Top 10, auth/authz, dependency scanning, secrets. It does not check:
- Whether the architecture correctly addresses all PRD ACs
- Whether failure modes are handled (non-security failures)
- Whether the design is unnecessarily complex
- Whether boundaries and ownership are clear
- Whether alternatives were adequately considered

These are the checks the Codex architecture reviewer already performs.

## Acceptance Criteria

- [ ] **AC1 (Review gate command or integration):** An architecture review step can be triggered after Phase 2 completes and before Phase 3 starts. This may be a standalone command, a hook in `/architect`, or an integration with `/review --phase 2` (see ISS-057). The mechanism is an architecture decision.

- [ ] **AC2 (Separate context):** The architecture review runs in separate context from the architect agent, consistent with the separate-context principle enforced for Phase 4 and Phase 6 gates.

- [ ] **AC3 (Review methodology):** The reviewer applies an architecture-quality methodology covering:
  - PRD alignment: does the architecture address every AC in the PRD?
  - Structural correctness: are boundaries, ownership, and data flows clear?
  - Failure modes: are non-security failure modes (rollback, retry, partial failure, degraded state) addressed?
  - Simplicity: is the design appropriate for the problem scope, or over-engineered?
  - Dependency clarity: are new dependencies justified? Are existing utilities reused?
  - Alternatives: were rejected approaches documented with sufficient reasoning?

- [ ] **AC4 (Review artifact):** The review produces a structured artifact at `docs/features/<feature>/review-architecture-<feature>.md` (or per ISS-058 naming convention if landed) with findings, open questions, and a proceed/proceed-with-changes/rework verdict.

- [ ] **AC5 (Blocking on rework verdict):** A "rework before implementation" verdict blocks progression to Phase 3. "Proceed with changes" findings are surfaced but non-blocking.

- [ ] **AC6 (Complements Phase 4):** The architecture review explicitly does not duplicate Phase 4 security gate concerns. Security findings are deferred to Phase 4. The gate focuses on structural, behavioral, and design quality.

- [ ] **AC7 (Leverages existing reviewer):** The review methodology aligns with or directly loads `codex/reviewers/review-architecture.md`, which already covers the required checks. No duplicate methodology.

### Model recommendation

Sonnet is recommended — architecture review requires reasoning about structural correctness, failure modes, and design trade-offs, which is more complex than PRD pattern-matching.

### Ordering relative to Phase 4

The architecture review gate should run **before** Phase 4 (Security Gate). Proposed flow:
```
Phase 2 (Architect) → Architecture Review Gate → Phase 4 (Security Gate) → Phase 3 (Test Design)
```
This ensures the architecture is structurally sound before the security reviewer examines it, reducing noise in security findings caused by architectural gaps.
