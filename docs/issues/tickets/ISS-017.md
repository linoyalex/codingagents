# ISS-017: Add stop-conditions footer to high-stakes skills

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `skills`, `quality`
- **Created:** 2026-04-07
- **Reporter:** Linoy (via Claude research on ISS-010)
- **Feature area:** `skills/verification-gate/`, `skills/tdd/`, `skills/security-audit/`

## Description

Recent prompt-engineering research (2026 playbooks, Lakera prompt engineering guide, multi-agent reliability literature) consistently finds that **models attend most reliably to the start and end of long prompts** and least reliably to the middle. Critical constraints — stop conditions, prohibited actions, output format requirements — should appear at **both** ends of a prompt.

ISS-010's refresh added a "Stop Conditions" section near the top of each refreshed skill, which is the right idea but only half the implementation. For high-stakes skills where misuse has real cost (verification, security audit, TDD), the stop conditions should also appear at the bottom of the file so they survive context-window pressure and are visible to an agent that has read past the initial preamble.

This ticket adds a footer reinforcement pattern, separate from the size convention revision (ISS-013) so the two changes are reviewed on their own merits.

## Acceptance Criteria

- [ ] **AC1:** `skills/verification-gate/SKILL.md` has a footer block (after the existing No-Go Criteria section if present) titled "Final Stop Check" that restates the 3-5 highest-leverage stop conditions in one block
- [ ] **AC2:** `skills/tdd/SKILL.md` has a footer block restating: "If RED fails for the wrong reason, stop. Do not write GREEN code on top of a false-positive failure."
- [ ] **AC3:** `skills/security-audit/SKILL.md` has a footer block restating: "If any BLOCKING finding exists, do not write a forward handoff. The pipeline halts here."
- [ ] **AC4:** Footer wording is intentionally **shorter** than the top-of-file Stop Conditions section — it is reinforcement, not duplication. Aim for 3-7 lines per footer.
- [ ] **AC5:** A contract test asserts the presence of a "Final Stop Check" or equivalent footer marker in each high-stakes skill, using **structural** assertions (heading presence) not literal phrases (so future wording refinement is allowed)
- [ ] **AC6:** The line cost of adding footers is documented and counted against ISS-013's revised budget

## Notes

This is a small, surgical change (~30 lines total across 3 skills) but it matters because the highest-cost agent failures we have seen so far happen exactly where stop conditions live in the middle of a long file. A reinforcement footer is a cheap way to align with how models actually attend to long prompts.

This ticket should land **after** ISS-013 (size convention revision) so the footer additions are scored against the new budget, not the old one. Sequencing: ISS-010 rework → ISS-013 → ISS-017.

### Research grounding

- "The most critical constraints — stopping conditions, prohibited actions, and required output format — should appear both at the beginning and end of the system prompt, since models attend to the start and end of context more reliably than the middle." — Lakera prompt engineering guide (2026)
- Anthropic's published attention-pattern guidance for long contexts agrees on the U-shaped attention curve for sufficiently long prompts.
