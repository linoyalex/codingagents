# ISS-060: Add PRD review gate between Phase 1 and Phase 2

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `pipeline`, `review-quality`
- **Created:** 2026-04-16
- **Reporter:** RCA from ISS-001 invariants-audit Phase 1 Codex review
- **Feature area:** Pipeline gates, `commands/specify.md`, `codex/reviewers/review-prd.md`

## Description

Phase 1 (Specify) is the only authoring phase without a review gate before its output is consumed downstream. Phase 5 has Phase 6 (code review). Phase 2 has Phase 4 (security gate). Phase 1 has nothing — the PRD goes directly to the architect.

During the ISS-001 invariants-audit feature, the Codex PRD reviewer (`codex/reviewers/review-prd.md`) caught 1 HIGH and 3 MEDIUM findings plus 3 missing states — all legitimate issues that would have propagated through architecture, test design, and implementation. The findings were:

- **[HIGH] AC6 too weak:** "At least one deterministic test" for a feature spanning 8 consumers — a single test could leave most integrations unverified.
- **[MEDIUM] AC5 vague:** "Incorporated into" Codex reviewers gave no testable structural marker.
- **[MEDIUM] AC1 gap:** Sibling file discoverability not formally required despite being a documented convention.
- **[MEDIUM] Screen states mismatch:** Output contract in screen states table not backed by any AC.

These findings were caught only because Codex review was manually invoked. Without it, the architect would have designed around a vague AC5, the QA agent would have written one test for AC6, and the developer would have interpreted "incorporated" differently — causing at least one full rework cycle.

### Root cause

The product-owner agent correctly follows the Ticket Fidelity Procedure — it transcribes ticket ACs, it doesn't adversarially challenge them. The Phase 1 verification gate checks structure (file exists, Given/When/Then present, Dependencies section present, line count), not semantics (AC coverage vs consumer counts, screen states backed by ACs, testability). No separate-context review exists between Phase 1 and Phase 2.

### Why this matters

The PRD is the foundation for every downstream phase. A vague or weak AC in the PRD propagates through:
- Phase 2: architect designs around the vague requirement
- Phase 3: QA writes tests to the weak bar
- Phase 5: developer implements ambiguously
- Phase 6: reviewer has no clear spec to review against

The ISS-008 RCA showed ~50% rework cost from Phase 3 QA gaps. PRD gaps are upstream of Phase 3 — they compound.

## Acceptance Criteria

- [ ] **AC1 (Review gate command or integration):** A PRD review step can be triggered after Phase 1 completes and before Phase 2 starts. This may be a standalone command, a hook in `/specify`, or an integration with `/review --phase 1` (see ISS-057). The mechanism is an architecture decision.

- [ ] **AC2 (Separate context):** The PRD review runs in separate context from the product-owner agent that authored the PRD, consistent with the separate-context principle enforced for Phase 4 and Phase 6 gates.

- [ ] **AC3 (Review methodology):** The reviewer applies a spec-quality methodology (not implementation-quality). At minimum:
  - AC coverage: do ACs cover all declared consumers, dependencies, and integration points?
  - AC testability: can each AC be verified by a deterministic test?
  - Screen states alignment: are screen states backed by ACs?
  - Missing states: permission denied, partial failure, drift, and other edge states
  - Scope control: are open-ended terms enumerated or flagged?

- [ ] **AC4 (Review artifact):** The review produces a structured artifact at `docs/features/<feature>/review-prd-<feature>.md` (or per ISS-058 naming convention if landed) with findings, missing states, and a ready/needs-clarification verdict.

- [ ] **AC5 (Blocking on HIGH):** HIGH findings or a "needs clarification" verdict block progression to Phase 2. MEDIUM findings are surfaced but non-blocking.

- [ ] **AC6 (Trigger conditions):** The review gate documents when it should be triggered vs skipped. Suggested triggers:
  - Consumer count > 3 (cross-cutting features)
  - P1 priority
  - Features touching the review layer, pipeline gates, or conventions
  - Features with cross-layer integration (skills + commands + hooks + tests)
  Optional: always-on for all features (architecture decision).

- [ ] **AC7 (Leverages existing reviewer):** The review methodology aligns with or directly loads `codex/reviewers/review-prd.md`, which already covers the required checks. No duplicate methodology.
