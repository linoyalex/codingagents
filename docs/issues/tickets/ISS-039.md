# ISS-039: Add downstream-impact, drift-check, and reproduction steps to code-review skill

- **Type:** Feature
- **Priority:** P1 — High
- **Label:** `framework`, `review-quality`
- **Created:** 2026-04-13
- **Reporter:** code-reviewer agent (RCA from review-hardening Phase 6)
- **Feature area:** `skills/code-review/SKILL.md`, `commands/review.md`

## Description

RCA of the review-hardening Phase 6 review revealed that Claude's code-review skill missed 4 findings that the Codex code review caught. The root causes are systemic gaps in the review methodology, not one-off oversights:

1. **No downstream consumer check** — When a schema gains a required field, the reviewer should verify all producers populate it. Claude missed that 6 other command templates don't populate `source_spec` after it became required in `handoff.schema.json`.

2. **No source/installed drift check** — `docs/CLAUDE.md` Known Gotchas (ISS-009) explicitly warns about source vs installed copy drift. The code-review skill has no step that triggers this check. Claude missed that `.claude/commands/review.md` and `.claude/commands/security-gate.md` are stale copies of the updated `commands/` source files.

3. **No existing-test-suite run** — The skill's "Quick Automated Checks" are grep-only (console.log, .skip, as any). Codex ran the actual test suite and found 2 existing test failures from fixture migration gaps. Claude never ran `node --test`.

4. **No reproduction step** — Claude stated findings analytically; Codex reproduced them with actual commands, leading to more accurate severity calibration. Claude rated `source_spec` file-existence as LOW; Codex reproduced it as BLOCKING.

## Acceptance Criteria

- [ ] **AC1 (Schema impact check):** The code-review skill's Review Methodology or a new "Impact Analysis" section requires: when a diff adds/removes a required field in a schema, grep for all producers and consumers of that schema and verify they handle the change.
- [ ] **AC2 (Source/installed drift check):** The Quick Automated Checks section includes a command that compares source files (`commands/`, `skills/`, `hooks/`) against their installed copies (`.claude/commands/`, `.claude/skills/`, `.claude/helpers/`) for any file touched in the diff.
- [ ] **AC3 (Run existing tests):** The Quick Automated Checks section includes running the project's test suite (not just grep checks) and verifying no regressions in existing tests.
- [ ] **AC4 (Reproduction requirement):** The Review Methodology includes a step requiring the reviewer to reproduce at least BLOCKING and HIGH findings with actual commands before finalizing severity.
- [ ] **AC5 (Regression test):** Contract tests verify the presence of the new methodology sections using structural anchors (not phrase-binding).
- [ ] **AC4a (Implementer reproduction requirement):** The `/implement` command (or TDD skill) must instruct the developer: "When implementing against review feedback, reproduce each reported finding before declaring it fixed. If the finding describes a code defect (not a test failure), write a failing test that demonstrates the defect before writing the fix." Evidence: ISS-036 Codex reported blank-cell parser defects in `lib/wiring-check.js`. Claude ran existing tests (which all passed because no test covered blank cells), declared all findings resolved, and repeated this 7 times. The defect was only fixed after the user explicitly explained what "blank-cell parser defect" meant.
- [ ] **AC6 (Symmetric gate enforcement):** When the Review Methodology or Reviewer Independence section specifies a check for one gate role/command (e.g., produced_by in review.md), it must explicitly instruct the reviewer to verify the same check exists in all enumerated gate roles/commands (e.g., security-gate.md). The code-review skill must include a "verify symmetric enforcement across all gate components" instruction.

## Technical Analysis

The code-review skill (`skills/code-review/SKILL.md`) currently has:
- A 5-step methodology (understand → big picture → tests → impl → integration)
- "Quick Automated Checks" limited to 3 grep patterns
- "Reviewer Independence" section for spec-first reading

Missing:
- No step for tracing schema changes to all producers/consumers
- No step for verifying source/installed parity
- No step for running the actual test suite
- No step requiring reproduction of findings
