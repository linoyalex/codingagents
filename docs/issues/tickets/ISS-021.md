# ISS-021: Add first-class command entry points for Codex reviewer prompts

- **Type:** Feature
- **Priority:** P3 — Low
- **Label:** `developer-experience`, `commands`, `codex`, `review`
- **Created:** 2026-04-10
- **Reporter:** Linoy
- **Feature area:** `commands/`, `codex/reviewers/`, reviewer invocation workflow

## Description

The repository already includes Codex reviewer prompts under `codex/reviewers/`, but
invoking them still requires the user to remember the right reviewer file, reconstruct
the recommended prompt, decide on the output filename, and keep the invocation aligned
with the repo's current artifact conventions.

That friction is especially noticeable because the surrounding pipeline already has
first-class slash-style command entry points under `commands/` for the Claude-driven
phases. The Codex review lane feels comparatively manual even though the reviewer
prompts themselves are already checked in and documented.

We should add first-class command entry points for the Codex reviewers so a user can
invoke the existing reviewer prompts with a short, discoverable command-like workflow.
This work should focus on command ergonomics and repo consistency, not on changing the
review policy or adding a parallel gated pipeline.

## Acceptance Criteria

- [ ] Add first-class command files or equivalent command entry points for the current
      Codex reviewers:
  - `review-code`
  - `review-test-design`
  - `review-architecture`
  - `review-prd`
- [ ] Each command points to the existing reviewer prompt file under `codex/reviewers/`
      instead of duplicating review logic in multiple places
- [ ] The command guidance clearly tells Codex what to review for the current branch or
      active feature and where to write the output artifact
- [ ] Output paths are standardized and aligned with the repo's actual downstream
      conventions, especially the feature-local review files consumed by later phases
- [ ] If a feature slug is omitted, the command has a deterministic rule for resolving
      the active feature or clearly fails with an actionable message
- [ ] The command guidance preserves the current Codex review design rules:
  - advisory only
  - one artifact per run
  - diff-first for code review
  - `.claude/handoff.json` used only for AC/risk context when relevant
- [ ] Repository docs are updated so the preferred Codex invocation path is obvious and
      does not conflict with older examples
- [ ] Any overlap with shell-wrapper work is documented so this command-based workflow and
      `ISS-011` do not drift into redundant solutions without an explicit decision

## Notes

This ticket is intentionally separate from `ISS-011`, which is about shell script
wrappers. The goal here is first-class command ergonomics inside the repository's
existing command conventions.

A likely implementation shape is:

- add `commands/review-code.md`
- add `commands/review-test-design.md`
- add `commands/review-architecture.md`
- add `commands/review-prd.md`
- keep the reviewer logic in `codex/reviewers/*.md`
- update docs so users know when to use the Claude pipeline review flow vs. the Codex
  reviewer commands

The main risk is creating two competing invocation patterns. The final design should
pick a clear preferred path and make the artifact locations consistent with what
`commands/document.md` and existing feature review files already expect.
