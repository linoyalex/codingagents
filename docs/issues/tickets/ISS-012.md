# ISS-012: Add stage-matched Codex reviews and iterative feedback resolution through the pipeline

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `codex`, `review-workflow`, `pipeline`
- **Created:** 2026-04-07
- **Reporter:** Linoy
- **Feature area:** `codex/`, commands, review loop, pipeline quality gates

## Description

The repository already supports standalone Codex review prompts for targeted second-pass
reviews, but that workflow is still loosely coupled to the main development pipeline.
There is no first-class mechanism to invoke the appropriate Codex review for each stage
of the feature lifecycle, capture the result, and have the responsible agent iterate on
the feedback until the work reaches a passing Codex review state.

We should add structured support for Codex reviews that map to the major pipeline stages
such as PRD, architecture, test design, code review, and other relevant review
checkpoints. The workflow should make it easy to run the right Codex review at the right
time, record the output in the feature artifacts, and explicitly loop on actionable
feedback until the review passes or is intentionally deferred.

## Acceptance Criteria

- [ ] The system supports invoking Codex reviews that align with the major pipeline
      stages, including at minimum:
  - PRD review
  - architecture review
  - test-design review
  - code review
- [ ] The stage-to-review mapping is documented and reflected in the relevant commands,
      prompts, wrappers, or workflow docs
- [ ] Review output is written to predictable feature-local artifacts under
      `docs/features/<feature>/`
- [ ] Agents are instructed to address actionable Codex findings and re-run the relevant
      Codex review until it reaches a passing state or an explicit defer/override outcome
- [ ] The iteration loop distinguishes between:
  - blocking findings that must be resolved
  - non-blocking improvements that may be deferred
  - intentionally accepted risk with recorded justification
- [ ] The workflow records review rounds clearly so maintainers can see:
  - which review was run
  - what feedback was given
  - what changed in response
  - whether the latest review passed
- [ ] The system includes safeguards against infinite or low-signal review churn, such as
      clear stop conditions, escalation rules, or defer paths
- [ ] Companion updates are made to commands, review prompts, scripts, or docs as needed
      so the behavior is operational rather than aspirational
- [ ] Codex review findings must distinguish finding type — at minimum: `test-failure` (reproducible by running a test command) vs `code-inspection` (latent defect identified by reading code, not caught by existing tests). The implementer workflow must handle each type differently: test-failures are reproduced by running the cited command; code-inspection findings require writing a new failing test before fixing. Evidence: ISS-036 Codex identified blank-cell parser defects through code inspection. Claude's resolution workflow was "run tests → they pass → finding resolved," which cannot catch inspection-type findings.
- [ ] Deterministic checks or contract tests cover the new stage-matched Codex review
      workflow where practical

## Notes

This request is about integrating Codex review into the pipeline more deliberately, not
about replacing the main pipeline agents with Codex.

Key design concerns to resolve during implementation:
- whether Codex review is advisory-by-default or required before advancing certain stages
- how "passes Codex review" is represented in artifacts and commands
- how to preserve forward progress when findings are partially subjective
- how to keep the loop high-signal and avoid repeated review cycles that do not converge

Likely companion areas:
- `codex/reviewers/`
- `codex/README.md`
- any review wrapper scripts
- stage commands that should trigger or reference Codex review
- feature-local review history artifacts such as `review-history.md`
