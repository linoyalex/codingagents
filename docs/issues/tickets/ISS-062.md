# ISS-062: In-flight phase-progress notes for interrupt recovery

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `pipeline`, `workflow`
- **Created:** 2026-04-16
- **Reporter:** Linoy
- **Feature area:** `commands/implement.md`, `commands/test-design.md`, `hooks/restore-context.cjs`, new `.claude/phase-progress.json`

## Description

The pipeline currently has three persistence layers: `handoff.json` (phase-boundary contract), `pipeline-checkpoint.json` (Stop-hook snapshot), and the narrow `checkpoint_pending` flag for "wait for user review" cases. None of them track progress *within* a single command's execution.

When a long-running command (especially `/implement` and `/test-design`) is interrupted by a context compact, an unrelated user request, or a session crash partway through its work, the next session has no structured way to know which steps already completed. Recovery today is "guess from file existence" — which loses ordering, intent, and any partial decisions made along the way.

The proposal: a mid-command progress note that the command updates as it works, so a fresh session can resume from the last completed step instead of redoing work.

## Acceptance Criteria

- [ ] **AC1 (Storage):** Progress is written to `.claude/phase-progress.json` (separate from `handoff.json` so it cannot break checkpoint.cjs schema validation). Schema published at `schemas/phase-progress.schema.json` (draft-07, `additionalProperties: false`).
- [ ] **AC2 (Lifecycle):** When a phase completes and writes a new `handoff.json`, the same step clears `.claude/phase-progress.json` (or marks it complete and dated). Stale progress notes from prior phases must not leak into new sessions.
- [ ] **AC3 (Opt-in scope):** Initial adopters are `commands/implement.md` and `commands/test-design.md` only. Other pipeline commands are unchanged. The ticket explicitly documents this as opt-in: short or already-checkpointed commands (`/specify`, `/document`, `/status`, `/security-gate`, `/architect`, `/review`) do not pay the overhead. Future commands may opt in.
- [ ] **AC4 (Command-instruction-driven, not hook-driven):** Progress notes are written by explicit steps in the opted-in command's instructions (e.g., "after committing the RED suite, write `{step: 'red-committed'}`"). Hooks do not fabricate progress on their own — only the command knows what counts as a meaningful step boundary.
- [ ] **AC5 (Resume surface):** `restore-context.cjs` (SessionStart hook) reads `.claude/phase-progress.json` if present and surfaces the last completed step in its output, alongside the existing handoff context. The agent sees "Resuming /implement at step `green-test-3-of-5`" before deciding next action.
- [ ] **AC6 (Test coverage):** Contract tests verify that the schema validates well-formed progress notes, rejects malformed ones, and that lifecycle transitions (write → clear on phase complete) work end-to-end. At least one integration test simulates interrupt + resume.
- [ ] **AC7 (Documentation):** `docs/CLAUDE.md` Architecture Notes documents the three-tier persistence model (handoff = boundary contract, pipeline-checkpoint = Stop snapshot, phase-progress = mid-command journal). Known Gotchas notes the opt-in nature so contributors don't expect every command to surface resume hints.
- [ ] **AC8 (`/status` integration):** `commands/status.md` reads `.claude/phase-progress.json` if present and reports a clear, structured resume directive: which command to re-invoke (e.g., `/implement <feature>`), the last completed step name, and the next step the command should pick up at. Output must distinguish three states unambiguously: (a) no progress note → normal phase-status output, (b) progress note present and in-phase → "Resume `/<command> <feature>` at step `<step-id>` (next: `<next-step>`)", (c) progress note present but stale (handoff phase has advanced past the progress phase) → flag the inconsistency and recommend deletion. Contract test verifies all three branches.

## Notes

### Why opt-in (known limitation)

A user who interrupts `/specify` will not get a resume hint, and might wonder why `/implement` does. This is an accepted tradeoff: applying progress notes universally would bloat instruction sets in commands that don't benefit. The asymmetry should be called out in CLAUDE.md.

### Why command-instruction-driven, not hook-driven

Hooks (`PreCompact`, `Stop`) fire on harness events but cannot know what step the command considers "done." Only the command's own instructions can mark progress points meaningfully. A hook could append a generic "compacted at turn N" note, but that's not the same as "completed step 3 of 5."

### Why a separate file, not a `progress` field on `handoff.json`

`handoff.json` is the phase-boundary contract validated by `checkpoint.cjs` (Stop hook, blocking gate). Adding mutable mid-phase state would muddy that schema and risk a malformed-during-write progress field tripping the validation gate. Separate file = separate concern.

### Prior art in this repo

- `checkpoint_pending: "architecture-review"` ([commands/architect.md:32-33](../../../commands/architect.md#L32-L33)) — narrow case of the same idea, just for "waiting on user review." This ticket generalizes the pattern.
- ISS-023 AC10 — established the meta-rule "what persisted state marks this checkpoint as pending, and how does session resumption recover it?" This ticket operationalizes that rule for in-flight progress.
- `/session-note` skill — terminal end-of-session note, manual. Complementary, not a substitute.

### Out of scope

- Multi-user / concurrent-session conflict handling (single-user assumption holds for codingagents pipeline).
- Auto-detection of "should I resume?" — that's a judgment call left to the agent reading the surfaced state.
- Retrofitting progress notes into already-running commands. Adopters must add the explicit checkpoint-write steps as part of their next normal revision.
