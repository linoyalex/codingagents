# ISS-038: Support Codex, Gemini, and other LLMs as first-class coding agents

- **Type:** Architecture
- **Priority:** P2 — Medium
- **Label:** `framework`, `multi-agent`, `provider-agnostic`, `architecture`, `workflow`
- **Created:** 2026-04-13
- **Reporter:** Linoy
- **Feature area:** Core pipeline contract, install/upgrade, agent-facing docs, provider adapters

## Description

The current framework is still centered on Claude as the primary authoring agent, with
Codex mostly treated as an additive review lane. That is useful, but it is not the same as
supporting multiple LLMs as **first-class coding agents** that can participate in the main
development pipeline with equal clarity and predictable setup.

We should evolve the framework so Codex, Gemini, and future LLM-based coding agents can be
treated as primary pipeline participants rather than provider-specific exceptions. "First
class" here means:

- the core pipeline contract is defined independently from any one vendor
- install/upgrade/setup flows can provision the necessary agent-specific surfaces
- docs make it clear how each supported agent participates in the pipeline
- agent-specific files/directories are adapters around a shared framework contract, not the
  framework itself

This ticket is intentionally broader than "add more wrappers" or "add a Gemini prompt
template." It is about making the pipeline architecture provider-aware without forking the
whole framework per model/vendor.

The intended operating model is also more specific than "support multiple agents somehow."
Projects should be able to choose:

- a default **authoring agent** for feature-building phases
- a default **review agent** for gate/review phases
- specific agent overrides for particular commands or phases
- a way to record, evaluate, and improve which agents are strongest at which jobs over time

## Acceptance Criteria

- [ ] **AC1 (Provider-neutral core contract):** Document the provider-neutral core pipeline
      contract: phases, handoff schema, artifact expectations, review outputs, and shared
      workflow rules that should hold regardless of whether Claude, Codex, Gemini, or
      another supported agent is running the phase.
- [ ] **AC2 (First-class agent definition):** Define what "first-class coding agent" means
      operationally in this repo, including the minimum capabilities/entry points an agent
      must support to participate in the main pipeline as an authoring agent rather than
      only as a reviewer.
- [ ] **AC3 (Adapter model):** Define an explicit adapter layer for agent-specific surfaces
      such as root instruction files, runtime directories, slash-command equivalents,
      invocation patterns, and install targets, so new agents can be added without copying
      the whole framework.
- [ ] **AC4 (Supported-agent matrix):** Add a documented capability matrix for at least
      Claude, Codex, and Gemini showing which phases they can run, what setup they need,
      and what remains provider-specific versus shared.
- [ ] **AC5 (Project defaults):** A project can define its default authoring agent and
      default review agent in a machine-readable configuration that later commands and
      workflows can read deterministically.
- [ ] **AC6 (Command/phase routing):** A project can override the default agent for
      specific commands or phases (for example: `specify` uses Claude, `review` uses
      Codex, `security-gate` uses Gemini). The configuration format must define clear
      precedence between:
  - global defaults
  - authoring vs review defaults
  - command-specific overrides
  - explicit per-run user override
- [ ] **AC7 (Install/upgrade strategy):** `init.sh`, `upgrade.sh`, or their replacement
      contract define how a target project installs support for one or more first-class
      agents in a predictable way instead of assuming Claude-first structure only.
- [ ] **AC8 (Artifact and path strategy):** Clarify how agent-specific files/directories
      such as `CLAUDE.md`, `.claude/`, Codex-specific artifacts, and future Gemini-specific
      artifacts relate to the shared framework contract. The design must avoid treating one
      vendor's file layout as the implicit definition of the framework.
- [ ] **AC9 (Authoring vs review role clarity):** The docs distinguish clearly between:
  - an agent acting as a primary phase author/reviewer in the main pipeline
  - an additive cross-model review lane
  - an unsupported or partial adapter
- [ ] **AC10 (Agent specialization tracking):** The framework defines a project-readable way
      to track agent strengths, weaknesses, or specialization notes over time, such as:
  - strongest phases/commands for an agent
  - known failure modes or review blind spots
  - recommended usage patterns
  - evidence or history that led to those conclusions
- [ ] **AC11 (Improvement loop):** The design defines how projects update agent
      specializations over time based on real usage, review outcomes, regressions, or
      benchmark signals, without turning the system into an unstructured notes dump.
- [ ] **AC12 (Selection explainability):** Docs or command output make it clear why a given
      agent was selected for a phase/command: inherited default, review default,
      command-specific override, or explicit user choice.
- [ ] **AC13 (Extensibility path):** The architecture defines how a future agent/provider is
      added without reworking every command, skill, and document manually.
- [ ] **AC14 (Regression protection):** Deterministic tests or contract checks cover the
      chosen provider/adapter model enough to catch silent drift in install shape, shared
      artifacts, supported-agent declarations, and agent-selection precedence.
- [ ] **AC15 (Docs update):** Top-level docs explain how a project chooses supported agents,
      what "first-class" support means, and what is currently fully supported versus
      experimental. They also explain how to configure:
  - the default authoring agent
  - the default review agent
  - command-specific agent routing
  - the specialization-tracking location and update process

## Notes

### Why this is separate from ISS-005

- **ISS-005** is a dogfood execution ticket: use the current framework to develop the
  framework itself and log what breaks.
- **ISS-038** is a platform-architecture ticket: make multiple LLMs first-class authoring
  agents in the framework design.

ISS-005 may surface evidence for this work, but it should not absorb the broader
multi-agent/provider-neutral architecture scope.

### Relationship to other tickets

- **ISS-005** is a likely evidence source, not a substitute.
- **ISS-021** and **ISS-011** touch Codex invocation ergonomics, but they do not make
  Codex or other agents first-class authoring participants.
- **ISS-034** helps if backlog conventions need to be readable by multiple agent types.
- **ISS-008** may intersect if project guidance files become provider-specific or
  multi-provider.

### Requested outcome shape

This ticket should explicitly support:

- selecting the default authoring agent per project
- selecting the default reviewer/gate agent per project
- choosing specific agents for specific commands/phases
- preserving a history of which agents perform best at which responsibilities so projects
  can refine routing decisions over time
