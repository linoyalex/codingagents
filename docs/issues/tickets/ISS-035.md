# ISS-035: Capture the backlog ticket ID in generated PRDs

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `artifacts`, `traceability`, `documentation`, `workflow`
- **Created:** 2026-04-12
- **Reporter:** Linoy
- **Feature area:** `commands/specify.md`, PRD template/convention, feature traceability

## Description

When a feature starts from a backlog ticket, the generated PRD should carry the originating
ticket identifier directly in the document. Today, the feature folder and PRD can drift away
from the original ticket context, which makes it harder for reviewers and future operators to
answer basic traceability questions:

- which backlog item this PRD came from
- whether the feature still matches the original ticket intent
- which ticket a reviewer should cross-check when validating correctness
- how to reconnect a feature artifact to the backlog when multiple similar features exist

This ticket adds a first-class backlog/ticket identifier field to generated PRDs so the source
request remains visible in the artifact itself.

## Acceptance Criteria

- [ ] **AC1 (PRD field defined):** The PRD convention defines a consistent field for the
      originating backlog/ticket ID, including exact field name and placement.
- [ ] **AC2 (Specify command update):** `commands/specify.md` is updated so that when a
      feature originates from a backlog ticket, the generated `prd.md` includes the exact
      backlog/ticket identifier (for example `ISS-026`).
- [ ] **AC3 (No-ticket path defined):** The convention explicitly defines what happens when
      a PRD is created without a backlog ticket, so agents do not improvise or silently omit
      the field in inconsistent ways.
- [ ] **AC4 (Review usefulness):** The documented PRD field is sufficient for a later
      reviewer to locate the originating ticket without relying on branch name or chat
      history.
- [ ] **AC5 (Docs updated):** Relevant PRD-writing or pipeline guidance is updated so
      contributors know the field is required for ticket-backed features.
- [ ] **AC6 (Regression protection):** At least one deterministic check verifies that the
      PRD-producing command/template requires the backlog/ticket ID field.

## Notes

### Why this matters

- It preserves source traceability inside the feature artifact itself.
- It complements reviewer-correctness work by making the original ticket easier to verify
  against during later phases.
- It reduces ambiguity when multiple PRDs or similarly named features exist over time.

### Relationship to other tickets

- ISS-033 makes reviewers verify against the source specification or ticket. ISS-035 makes
  that source easier to locate by embedding the ticket ID directly in the PRD.
- ISS-028 may later improve ticket-aware feature selection, but this ticket is specifically
  about artifact traceability in the PRD itself.
