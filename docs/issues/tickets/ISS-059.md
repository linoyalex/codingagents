# ISS-059: Separate the CLAUDE.md template from the codingagents repo's own CLAUDE.md

- **Type:** Architecture
- **Priority:** P1 — High
- **Label:** `framework`, `deployment`, `architecture`
- **Created:** 2026-04-16
- **Reporter:** Linoy
- **Feature area:** `CLAUDE.md`, `init.sh`, `upgrade.sh`, `lib/sync-claude-md.sh`

## Description

The root `CLAUDE.md` currently serves a dual role: it's both the active instructions for the codingagents repo AND the template that `init.sh` copies to consumer projects. This creates two problems:

1. **The root file can't be optimized for this repo** without degrading the template for consumer projects. Template placeholders (`<!-- FILL IN -->`, `<!-- e.g. ... -->`) clutter the active instructions. Sections like Phase Handoff Protocol and Context & Token Management are redundant with installed skills/commands but needed in the template for onboarding.

2. **Consumer projects get framework-development content** they don't need (backlog system gotchas, feature slug naming issues, upgrade.sh idempotency gaps) when `--sync-claude-md` runs, because the sync reads from `docs/CLAUDE.md` which contains codingagents-repo-specific content.

The root CLAUDE.md has been optimized for this repo (314 → 144 lines). The template needs to be a separate file.

## Acceptance Criteria

- [ ] **AC1 (Template file):** A dedicated template file (e.g., `templates/CLAUDE.md` or `CLAUDE.template.md`) contains the consumer-facing template with placeholders, onboarding sections (Phase Handoff Protocol, Memory Governance, Context Management), and guided structure.

- [ ] **AC2 (init.sh updated):** `init.sh` copies the template file (not the root CLAUDE.md) to consumer projects.

- [ ] **AC3 (upgrade.sh updated):** `upgrade.sh` references the template for CLAUDE.md operations.

- [ ] **AC4 (sync-claude-md updated):** `--sync-claude-md` reads managed sections from a source that contains only universal framework conventions — not codingagents-repo-specific gotchas, ADR entries, or development instructions. This may mean syncing from the template or from a filtered subset of `docs/CLAUDE.md`.

- [ ] **AC5 (Root CLAUDE.md independent):** The root `CLAUDE.md` is free to contain codingagents-repo-specific content without affecting consumer projects. It can reference `docs/CLAUDE.md` for framework details without being constrained by the template structure.

- [ ] **AC6 (No regression):** Existing installer tests pass. Consumer projects that upgrade continue to work.

- [ ] **AC7 (Documentation updated):** README, QUICKSTART, and docs/CLAUDE.md updated to reflect the new template location and the separation of concerns.

## Notes

### Root cause

The dual-role design was expedient when the framework was small. As the repo grew (58 tickets, 9 completed feature cycles), the root CLAUDE.md accumulated both template placeholders and repo-specific content, reaching 314 lines. The managed section sync (`--sync-claude-md`) compounded the problem by injecting framework-dev-specific gotchas into consumer projects.

### Relationship to other tickets

- **ISS-008** (closed) — established `--sync-claude-md`. AC4 here refines what the sync reads from.
- **ISS-007** — force upgrade. Template separation may affect how `--force` handles CLAUDE.md.
- **ISS-055** — .cjs rename. Both touch init.sh/upgrade.sh; can share a batch if sequenced.

### Design consideration

The template should remain a complete, standalone file that makes sense when copied into a fresh project — not a file that requires reading `docs/CLAUDE.md` to understand. The root CLAUDE.md for this repo can be lean because agents have `docs/CLAUDE.md` auto-loaded alongside it; consumer projects don't have that luxury.
