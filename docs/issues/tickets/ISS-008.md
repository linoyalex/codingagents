# ISS-008: Sync project CLAUDE.md with reference docs/CLAUDE.md on init/upgrade

- **Type:** Feature
- **Priority:** P1 — High
- **Label:** `deployment`, `documentation`, `framework`
- **Created:** 2026-04-06
- **Reporter:** Linoy
- **Feature area:** init.sh, upgrade.sh, CLAUDE.md lifecycle

## Description

When `init.sh` or `upgrade.sh` runs, offer an option to update the project's root `CLAUDE.md` with relevant sections from the framework's reference `docs/CLAUDE.md`. Currently, the root `CLAUDE.md` is a static template copied once on init and never updated — meaning projects miss new conventions, architecture notes, known gotchas, and other framework improvements added to `docs/CLAUDE.md` over time.

This should be opt-in to avoid overwriting project-specific customizations. The mechanism should:

1. **On `init.sh`** — offer `--sync-claude-md` flag to populate the root `CLAUDE.md` template with relevant sections from `docs/CLAUDE.md` (code conventions, naming, folder structure, architecture notes, known gotchas) instead of leaving placeholder comments.

2. **On `upgrade.sh`** — offer `--sync-claude-md` flag to merge new or updated sections from `docs/CLAUDE.md` into the existing root `CLAUDE.md`, preserving project-specific content the user has added.

## Acceptance Criteria

- [ ] `init.sh --sync-claude-md` populates the root `CLAUDE.md` template sections (conventions, naming, folder structure, architecture notes, known gotchas) with content from `docs/CLAUDE.md`
- [ ] `init.sh` without the flag behaves as today — copies the template with placeholder comments intact
- [ ] `upgrade.sh --sync-claude-md` merges new sections from `docs/CLAUDE.md` into the existing root `CLAUDE.md` without overwriting user-customized content
- [ ] Sections owned by the user (e.g. project-specific gotchas, custom conventions) are preserved during sync
- [ ] A diff or summary is printed showing what was added or updated in root `CLAUDE.md`
- [ ] If root `CLAUDE.md` already contains a section that matches the reference, it is skipped (no duplicates)
- [ ] Existing tests in `test-install-scripts.sh` continue to pass
- [ ] New test cases cover: init with sync, upgrade with sync, preservation of user content, no-op when already in sync

## Notes

- The sync should be section-level, not full-file replacement. Use markdown heading markers (e.g. `## Code Conventions`, `## Known Gotchas`) as merge boundaries.
- Consider adding a marker comment (e.g. `<!-- managed by codingagents -->`) to synced sections so the tool can distinguish framework-managed content from user-added content on subsequent upgrades.
- `docs/CLAUDE.md` is the framework-development reference; the root template is the consumer-facing file. Not all sections from `docs/CLAUDE.md` belong in the consumer template — the sync should be selective.
