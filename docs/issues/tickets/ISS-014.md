# ISS-014: Make gate reviewers adversarial and require separate context for review phases

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `pipeline`, `review-workflow`, `reliability`
- **Created:** 2026-04-07
- **Reporter:** Linoy
- **Feature area:** Roles, commands, review discipline, phase-gate reliability

## Description

Security review, code review, and related gate phases are most valuable when they act as
independent challenge functions rather than extensions of the authoring agent. Today the
pipeline partially encodes this idea, especially for code review, but the stance is not
fully consistent across reviewer roles and commands.

We should make gate reviewers explicitly adversarial-but-constructive and require them to
run in separate context from the agent or session that produced the artifact being
reviewed. The goal is to reduce self-justifying review behavior, improve defect
discovery, and make review outcomes more credible.

## Acceptance Criteria

- [ ] `ROLE_SECURITY.md`, `ROLE_CODE_REVIEWER.md`, and any other relevant gate-review roles
      explicitly adopt an adversarial-but-constructive stance
- [ ] Gate review commands require separate context from the authoring phase, not just a
      generic fresh session
- [ ] The workflow clearly distinguishes authoring phases from gate/review phases
- [ ] Reviewer commands instruct the agent to challenge hidden assumptions, bypass paths,
      stale state, trust boundary failures, and contradictory artifacts
- [ ] Gate reviewers remain read-only on production code and do not repair the artifacts
      they are reviewing
- [ ] Review artifacts capture reviewer identity and context independence clearly enough to
      audit later
- [ ] Deterministic tests or contract checks cover the new separate-context and adversarial
      review requirements where practical

## Notes

This is not a request to make every phase adversarial. The adversarial stance should be
strongest in gate/check/review phases, where independence matters most.

Likely areas to update:
- `ROLE_SECURITY.md`
- `ROLE_CODE_REVIEWER.md`
- `commands/security-gate.md`
- `commands/review.md`
- supporting docs about review process and artifact conventions
