# ISS-063: Unattended end-to-end pipeline mode with bounded review-feedback loop

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `pipeline`, `workflow`
- **Created:** 2026-04-16
- **Reporter:** Linoy
- **Feature area:** new `commands/auto-pipeline.md` (or equivalent), `commands/architect.md`, `commands/review.md`, `commands/security-gate.md`, `hooks/checkpoint.cjs`, `schemas/handoff.schema.json`

## Description

Today every pipeline transition requires a human in the loop: the user invokes each `/<phase>` slash command in turn, reads the output, decides whether to proceed, and re-invokes the next phase. For small, well-scoped tickets — bug fixes, single-AC features, no new dependencies — this hand-holding is pure overhead. The user wants an opt-in mode that runs the pipeline end-to-end, addresses review feedback automatically within bounded retries, and stops in a state ready for human review (branch pushed, PR opened, or branch left clean depending on configuration).

This is **not** "auto-merge to main." The terminal state is always human-reviewable. Auto-mode does the mechanical work between checkpoints; merge and deploy decisions remain with the user.

## Acceptance Criteria

- [ ] **AC1 (Opt-in entry point):** A new command (working name `/auto-pipeline <feature>`) drives the full pipeline. Auto-mode is never the default — a human invokes it explicitly. The command refuses to run unless the eligibility gate (AC3) passes or `--force` is supplied.
- [ ] **AC2 (Pipeline scope — Phase 2 through 6):** Auto-mode covers `/architect` → `/test-design` → `/security-gate` → `/implement` → `/review`. It does **not** auto-write the PRD (Phase 1) or auto-document (Phase 7). The PRD is the highest-leverage human input; the documentation phase touches consumer-facing CHANGELOG and CLAUDE.md and warrants a human pass. Auto-mode requires the PRD already exists at `docs/features/<feature>/prd.md` before starting.
- [ ] **AC3 (Eligibility gate — fail closed):** Auto-mode refuses to run unless the ticket / PRD meets the small-scope heuristic. Default heuristic (rejects if any of the following): more than N acceptance criteria (default N=3), introduces a new top-level dependency, modifies `schemas/`, modifies anything under `auth/` or `security/`, modifies more than M files in the prior phase's plan (default M=10). The heuristic is documented and tunable. `--force` bypasses with a printed warning that lists which checks were skipped.
- [ ] **AC4 (Bounded review-feedback loop):** When `/review` returns REQUEST_CHANGES, auto-mode re-invokes `/implement` with the review findings as input, then re-runs `/review`. This loop is bounded by `--max-review-cycles` (default 2). When the limit is reached, auto-mode halts and surfaces the unresolved findings for human attention. The same bounded-retry pattern applies to `/security-gate` BLOCKING findings → re-architect or re-implement → re-audit (default `--max-security-cycles=1` because security re-design usually warrants human input).
- [ ] **AC5 (Hard stop conditions — never auto-retry):** Auto-mode halts immediately and waits for a human on any of: BLOCKING security finding that the loop limit can't resolve; verification gate failure (Phase 3, 4, or 6); test suite regression in unrelated test files; any phase exceeding a per-phase timeout (default 30 min wall-clock); any uncaught error from a phase command. On halt, the current `handoff.json`, the latest review/security artifact, and a clear "Halted because: <reason>. Resume with: /<command> <feature>" message are written to a new `.claude/auto-pipeline-status.json`.
- [ ] **AC6 (Architecture user-review checkpoint behavior):** `/architect`'s `checkpoint_pending: "architecture-review"` is preserved in auto-mode by default — the pipeline pauses there for human approval. Auto-mode includes a `--skip-arch-review` flag, but this flag is only honored when the architecture document is under N lines (default 100), introduces no new dependencies, and the eligibility gate (AC3) passed without `--force`. Otherwise the flag is ignored and the checkpoint pauses regardless.
- [ ] **AC7 (Terminal state — leave branch ready for human review):** When the full pipeline completes successfully, auto-mode leaves the work in one of two configurable terminal states: (a) **branch-only** — commits pushed to the feature branch, no PR opened, console output prints the suggested PR title/body and the `gh pr create` command; (b) **pr-opened** — branch pushed and PR opened with a structured body listing each phase's artifact and the auto-pipeline run summary. Default is **branch-only** because PR creation is irreversible and discoverable in CI/notifications.
- [ ] **AC8 (Run summary):** Auto-mode writes `.claude/auto-pipeline-status.json` with: feature, started_at, finished_at, terminal_state, phases_run (with per-phase duration and exit), review_cycles_used / limit, security_cycles_used / limit, halt_reason (if applicable), and the resume directive (next command to invoke if halted, or the suggested PR command if completed). `/status` reads this file and surfaces the current state ([cross-reference: ISS-062 AC8 for `/status` resume-directive contract](ISS-062.md)).
- [ ] **AC9 (Auto-pipeline never bypasses gate fail-closed semantics):** Existing fail-closed checks (separate-context enforcement via `produced_by`, `source_spec` requirement, schema validation in `checkpoint.cjs`) MUST still apply in auto-mode. Auto-mode runs each phase in a separate sub-session so `produced_by` checks behave correctly — the auto driver is not allowed to short-circuit gate validation by reusing context.
- [ ] **AC10 (Test coverage):** Contract tests verify the eligibility heuristic (AC3 — passes for known-good fixtures, fails for known-bad), the retry-loop budget (AC4 — synthetic failing review halts after N cycles, not N+1), the hard-stop conditions (AC5 — each condition surfaces in `auto-pipeline-status.json`), and the architecture-checkpoint behavior (AC6 — pause-by-default, skip honored only when conditions met). Integration test runs a synthetic small-feature pipeline end-to-end with a mocked review verdict alternating REQUEST_CHANGES / APPROVE.
- [ ] **AC11 (Documentation):** `docs/CLAUDE.md` Architecture Notes documents the auto-pipeline mode, the eligibility heuristic, the retry budgets, the hard-stop list, and explicit guidance on when to use it vs. driving phases manually. Known Gotchas notes the asymmetry: auto-mode is opt-in *and* refuses without explicit override on tickets outside the small-scope heuristic.

## Open Design Decisions

These were raised in the proposal conversation. Capturing them here as decisions to make during architecture (Phase 2 of this ticket, ironically not in auto-mode):

### D1 — Opt-in surface

Three candidates:
- **(a) New command `/auto-pipeline <feature>`** — explicit, discoverable, easy to add flags. Drawback: another command to maintain.
- **(b) Field on the ticket / PRD (`auto: true`)** — sticky across sessions, shows intent in the ticket itself. Drawback: requires reading the ticket file at every command boundary to know whether to chain.
- **(c) Both** — command for one-off invocation, ticket field as a shortcut for "always auto-run this ticket when picked up."

Default in this ticket: **(a) only**, with (c) deferred to a follow-up if demand emerges. Single command keeps blast radius contained.

### D2 — Eligibility heuristic owner

Three candidates:
- **(a) Triage-time field (`auto-eligible: true` set when ticket is filed)** — human judgment up-front, but requires triage discipline.
- **(b) Framework auto-detection from PRD content (AC count, dependency mentions, file scope)** — no human field needed, but heuristic can be wrong both ways.
- **(c) Hybrid: auto-detect proposes, human confirms via `--force` or `--auto-ok`** — user is informed, gate is fail-closed.

Default in this ticket: **(b) with `--force` override**. AC3 above codifies this; the heuristic is tunable as the framework learns from real runs.

### D3 — Pipeline scope

Two candidates:
- **(a) Phase 1 → 7 (full)** — true unattended end-to-end, but auto-PRD is risky for cost amplification and auto-docs touches consumer-facing files.
- **(b) Phase 2 → 6** — requires human PRD as input, leaves docs for a human pass.

Default in this ticket: **(b)** (codified in AC2). PRD remains the highest-leverage human input. If demand emerges for Phase 1 or Phase 7 in auto-mode, file as a separate ticket so that decision gets its own scrutiny.

## Notes

### Why bounded retries (not unbounded)

Unbounded auto-retry is a cost-amplification trap: a misframed PRD could burn through tokens for hours producing increasingly desperate implementations. Bounded retries (default: 2 review cycles, 1 security cycle) cap the damage and force human attention before continuing. The defaults are conservative on purpose — raise them per-run with `--max-review-cycles=N` once the framework has track record on a project.

### Why "branch-only" is the default terminal state

Opening a PR triggers CI runs, notifies reviewers, and shows up in dashboards. If an auto-pipeline run finishes at 3am on a marginal feature, the user wants to inspect the artifacts before that PR is visible. Branch-only is reversible (delete the branch); PR-opened is observable and harder to undo cleanly. `--pr-opened` is opt-in.

### Relationship to ISS-062 (in-flight progress notes)

ISS-062 lets a single command resume after interrupt. ISS-063 lets a sequence of commands run without user prompting. They compose: if `/auto-pipeline` is interrupted mid-`/implement`, ISS-062's progress note tells the next session where `/implement` was, and ISS-063's `auto-pipeline-status.json` tells it that it was the auto-driver who was running it.

### Out of scope

- Auto-merging to main, auto-deploying, auto-tagging releases.
- Auto-PRD generation (Phase 1).
- Auto-documentation (Phase 7).
- Multi-feature batching ("auto-run all P2 tickets in the backlog").
- Auto-mode for tickets that touch `auth/`, `security/`, or `schemas/` — these are categorically excluded by AC3 and the exclusion is not user-overridable even with `--force` (separate ticket if that proves too strict in practice).

### Prior art

- `commands/architect.md` `checkpoint_pending` — narrow precedent for "pipeline pauses, waits for human, resumes from persisted state."
- ISS-008 RCA documented ~50% rework in Phase 3 — auto-mode amplifies the cost of bad framing, hence AC3's strict eligibility gate.
- ISS-039 found that asymmetric gate enforcement caused false-positive HIGH findings — auto-mode's review-cycle loop must not trigger on cosmetic differences in framing between gates. Worth a contract test.
