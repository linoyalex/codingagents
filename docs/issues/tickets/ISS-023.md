# ISS-023: Strengthen architecture decision skill with wiring diagrams, evidence rules, and observability naming

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `skills`, `architecture`, `documentation`
- **Created:** 2026-04-11
- **Reporter:** Linoy (via global memory audit of cross-project learnings)
- **Feature area:** `skills/architecture-decision/SKILL.md`, `PIPELINE_GUIDE.md`

## Description

The architecture-decision skill defines module boundaries, API contracts, and failure modes but is missing four patterns that have caused architectural gaps to slip through the pipeline:

1. **No wiring diagrams.** The arch doc describes boundaries but not the call chain. Modules get built and tested in isolation but never wired into the production path.

2. **No fitness functions for doc/code sync.** When a spec doc and a code constant both define the same data, there's no mechanism to catch drift.

3. **"No changes" claims accepted without evidence.** Writing "No data model changes" without citing the schema line hides optional-field dependencies.

4. **Observability paths not named.** Architecture says "log a warning" without stating where that log goes or who sees it.

## Acceptance Criteria

- [ ] **AC1 (Wiring diagrams):** The architecture.md template in the skill adds a required "Integration Points" section showing: `entry point → module → output → consumer`.
- [ ] **AC2 (Fitness functions for sync):** The fitness functions guidance adds: "If the architecture references constants, enums, or labels that also appear in a spec doc, require a fitness function that parses both and asserts exact match."
- [ ] **AC3 (Evidence for "no changes"):** The skill adds: "Every 'No changes' statement must include a parenthetical citing the specific schema line or field that proves the claim."
- [ ] **AC4 (Observability naming):** The template adds: "For any logging/monitoring mentioned, state: log destination, searchability, and alerting path."
- [ ] **AC5 (Consumer enumeration for shared contracts):** When the architecture modifies a shared schema or config (e.g., `handoff.schema.json`, `settings.json`), the skill requires a "Producers & Consumers" subsection enumerating every file that reads or writes the contract — not just the ones in the feature's scope. The architect must state what each consumer needs to do to adopt the change (or confirm no change needed).
- [ ] **AC6 (Violation detection prompt):** For every behavioral constraint in the architecture (e.g., "reviewers must use separate context", "gate roles must not write src/"), the skill requires the architect to state how a violation is detected. If the only detection is "role prose instructs against it," the architect must flag that as weak enforcement and either propose a stronger mechanism or record the gap as a known risk. When a detection mechanism is proposed, the architect must also state what it does NOT catch (residual risk) — partial mitigations must not be presented as complete solutions.
- [ ] **AC7 (Role-command symmetry check):** When the architecture touches a role file (`ROLE_*.md`), the skill requires verifying the corresponding command file is also updated (and vice versa). The Files Changed table must show symmetric entries or explicitly justify asymmetry.
- [ ] **AC8 (Pipeline guide update):** `PIPELINE_GUIDE.md` Phase 2 architecture.md format includes the new "Integration Points" section.
- [ ] **AC9 (No regression):** Existing tests continue to pass. Skill stays within size budget.
- [ ] **AC10 (Session durability prompt):** When the architecture introduces a human-in-the-loop checkpoint or a "stop and wait" behavior, the skill requires the architect to answer: "What persisted state marks this checkpoint as pending, and how does session resumption recover it?" If the answer is "conversational context only," the architect must record that as a known risk.
- [ ] **AC11 (Installed-copy sync prompt):** When the Files Modified table includes a file that has a known installed copy (e.g., `commands/*.md` → `.claude/commands/*.md`, `skills/*/SKILL.md` → `.claude/skills/*/SKILL.md`), the skill requires the architect to list both the source and installed path in the boundary table and include a sync test in the test surface. This extends AC5 (consumer enumeration) to the source/installed copy deployment boundary.
- [ ] **AC12 (Failure mode invariant stress-test):** For each entry in the Failure Modes table, the skill requires the architect to state whether the proposed fallback preserves or violates the feature's core invariant. If a fallback bypasses the feature's primary safeguard (e.g., "skip fidelity check" for a fidelity feature), it must be flagged as a design contradiction and either strengthened or explicitly approved by the user.
- [ ] **AC13 (Negative coverage analysis):** The test design section must include a "What these tests cannot verify" subsection listing behavioral properties that structural or unit tests cannot enforce (e.g., multi-turn interaction, session resumption). This prevents architects from overclaiming test coverage.

## Notes

### Evidence

- Modules built and tested in isolation but never wired: validators had 253 passing tests but were never called in the live generation flow (closet-designer ISS-007).
- `component-reference.md` and `SECTION_LABEL_MAP` had no sync mechanism — caught by Codex.
- `itemsNotFitted` declared as "no changes" but had a hidden optional-field dependency — caught by Codex.
- `console.warn` vs Vercel function logs is a meaningful distinction that was glossed over.

### Evidence for AC5–AC7 (review-hardening architecture misses)

Three gaps found by Codex review of the review-hardening architecture doc:
- **Consumer enumeration (AC5):** `source_spec` was added as required in `handoff.schema.json` but the architecture only scoped updates to `commands/review.md` — missed that `checkpoint.js` validates all 7 phase handoffs, making this a pipeline-wide breaking change.
- **Violation detection (AC6):** "Separate context" enforcement was defined as a behavioral constraint but the only detection was role prose and artifact headers. No mechanism to detect same-agent continuity. Fixed post-review by adding `produced_by` mismatch check.
- **Role-command symmetry (AC7):** Both gate roles (ROLE_SECURITY, ROLE_CODE_REVIEWER) were updated but only `commands/review.md` got command-level source_spec treatment — `commands/security-gate.md` was missed. Fixed post-review.
- **Residual risk disclosure (AC6 extension):** Round 2 review found the `produced_by` role mismatch check was presented as catching "same-agent continuity" when it only catches same-role continuity — a weaker guarantee. The architecture overclaimed the mitigation's coverage without stating what it fails to catch. Fixed by adding Known limitation and Residual risk subsections.

### Evidence for AC10–AC13 (clarification-checkpoints architecture misses)

Four gaps found by Codex review of the clarification-checkpoints architecture doc (ISS-029):
- **Session durability (AC10):** Architecture said checkpoints are "instructional, not programmatic" and explicitly avoided hooks or persisted state — but the feature's core contract is "do not silently finalize." A session interruption mid-checkpoint had no recovery mechanism. Fixed by adding `checkpoint_pending` handoff field.
- **Installed-copy sync (AC11):** Module boundary table listed only source files (`commands/*.md`, `skills/prd-writing/SKILL.md`) but the runtime uses `.claude/` installed copies. The sync dependency was in Implementation Notes — not in the boundary table or test surface. Fixed by promoting to first-class boundary.
- **Failure mode invariant violation (AC12):** "Ticket file not found → skip fidelity check, proceed" directly bypassed the feature's primary safeguard. The architect treated this as a graceful degradation rather than recognizing it contradicts the feature's core guarantee. Fixed by changing to block/ask-user.
- **Negative coverage gap (AC13):** Structural tests were claimed to verify "no hidden auto-advance" and "handoff not written before checkpoint," but they only check instruction ordering — they cannot verify multi-turn behavior. The gap was undisclosed until Codex review.
