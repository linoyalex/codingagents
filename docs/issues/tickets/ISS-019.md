# ISS-019: Add `allowed-tools` frontmatter to read-only skills

- **Type:** Architecture
- **Priority:** P2 — Medium
- **Label:** `framework`, `skills`, `security`
- **Created:** 2026-04-07
- **Reporter:** Linoy (via Claude research on ISS-010)
- **Feature area:** `skills/`, tool scoping

## Description

Anthropic's skill frontmatter spec supports an optional `allowed-tools` field that restricts which tools an agent can use while operating under that skill. We currently scope tool access at the agent/role level only (`ROLE_*.md`), which means an agent loaded with multiple skills inherits the union of all skill expectations. This is too permissive for skills whose entire purpose is read-only review.

Skills like `verification-gate`, `code-review`, `security-audit`, and `architecture-decision` should never write to `src/`. Today this is enforced by prose ("read-only") and by the agent role definition. Adding `allowed-tools` to the skill frontmatter makes the constraint enforceable at the harness level — Claude Code will refuse to invoke a forbidden tool while the skill is active, regardless of what the role permits.

## Acceptance Criteria

- [ ] Audit each skill in `skills/` and classify as: read-only, read-write, or unconstrained
- [ ] Add `allowed-tools` frontmatter to read-only skills with the minimum tool set:
  - `verification-gate`: `Read`, `Bash`, `Glob`, `Grep` (no `Write`/`Edit`)
  - `code-review`: `Read`, `Glob`, `Grep`, `Bash` (read-only via `git diff`); allow `Write` only for the `review.md` artifact
  - `security-audit`: same as code-review
  - `architecture-decision`: `Read`, `Glob`, `Grep`, `Write` (writes only to `docs/features/<feature>/architecture.md`)
- [ ] Document the read-only/read-write classification in `docs/CLAUDE.md` Skills section
- [ ] Verify Claude Code respects `allowed-tools` at skill load time (manual smoke test)
- [ ] Add a contract test asserting that read-only skills declare `allowed-tools` and that the list does not include `Write`/`Edit` for the protected paths
- [ ] If `allowed-tools` is not granular enough to distinguish "Write to docs/features/" from "Write to src/", document the gap and propose a workaround (e.g. agent-level path allowlist)

## Notes

This complements but does not replace agent-level tool scoping. The two layers serve different purposes:
- **Agent-level** (`ROLE_*.md` `tools:` field): "What can this agent ever do?"
- **Skill-level** (`SKILL.md` `allowed-tools` frontmatter): "What is appropriate while doing this specific kind of work?"

Defense in depth. The skill-level scoping protects against mistakes in agent definitions, and vice versa.

This is P2 because every framework whose review layer can write to source code has a real attack surface — a confused agent that writes a "fix" while in review mode is exactly the kind of incident this prevents.
