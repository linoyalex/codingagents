# ISS-049: QA must default to fixture-driven behavioral tests for executable code, not structural checks

- **Type:** Feature
- **Priority:** P1 — High
- **Label:** `framework`, `testing`, `qa-convention`
- **Created:** 2026-04-14
- **Reporter:** RCA from ISS-008 Codex test design review
- **Feature area:** `skills/tdd/SKILL.md`, `commands/test-design.md`

## Description

During ISS-008 (CLAUDE.md sync), the QA agent wrote contract and E2E tests that checked whether shell source files *contained* certain strings (e.g. `/migrat/i`, `/managed:end/`, `/pre-sync/`), rather than invoking the code on fixture inputs and asserting outputs. This passed Codex review as 2 HIGH findings — the tests could go green while the actual behavior remained broken.

The root cause: the QA agent applied the test pattern from prior features (ISS-029, ISS-027, ISS-036), which tested markdown command/skill files. Those features are declarative — the "implementation" IS the markdown text, so structural checks are behavioral checks. But ISS-008's implementation is a shell library (`lib/sync-claude-md.sh`) with executable logic. String-presence in shell source code does not prove runtime behavior.

The TDD skill (`skills/tdd/SKILL.md`) already says "Testing implementation details instead of outcomes" is an anti-pattern, but it doesn't distinguish between declarative artifacts (where structure = behavior) and executable code (where structure ≠ behavior). The QA agent needs explicit guidance on when to use which test strategy.

### The distinction

| Production artifact | Test strategy | Why |
|---|---|---|
| Markdown command/skill file | Structural checks (heading exists, section contains pattern) | The text IS the instruction — if the heading exists, the agent sees it |
| Shell script / JS module / executable code | Fixture-driven tests (invoke function, assert I/O) | String presence doesn't prove the function works correctly |
| Config file (JSON schema, settings) | Schema validation + fixture loading | Structure must parse correctly AND produce expected behavior |

### What happened on ISS-008

1. QA wrote 25 contract tests — all string-presence checks against `lib/sync-claude-md.sh`
2. QA wrote 9 E2E tests — same static wiring checks, duplicating the contract layer
3. Codex flagged both as HIGH: "can all pass while marker insertion, outside-marker preservation, no-op handling, or backup-abort behavior is wrong at runtime"
4. Fix: rewrote contracts with fixture-driven Layer 2 (invoke sync function on sample CLAUDE.md, assert output files + CLI), rewrote E2E to execute init.sh/upgrade.sh in real temp directories

### Why this matters for the pipeline

If this convention isn't codified, every future feature with executable code will face the same Codex review cycle: QA writes structural tests → Codex catches the gap → rework. The rework cost on ISS-008 was ~50% of the Phase 3 test design effort.

## Acceptance Criteria

- [ ] `skills/tdd/SKILL.md` includes guidance distinguishing structural vs. fixture-driven test strategies based on artifact type (declarative vs. executable)
- [ ] `commands/test-design.md` includes a decision point: "Is the production artifact executable code or declarative text?" that routes QA to the appropriate test pattern
- [ ] The guidance includes the artifact-type → test-strategy table (or equivalent) so QA agents can pattern-match
- [ ] Existing tests in the test suite continue to pass
- [ ] The guidance is stack-agnostic (not specific to shell/Node/Python)

## Notes

- This is additive guidance, not a change to existing test patterns. Structural checks on markdown files remain correct for that artifact type.
- The fix should NOT invalidate existing passing tests — it only changes how QA approaches NEW test design for executable artifacts.
