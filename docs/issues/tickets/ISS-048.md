# ISS-048: Add release-manager skill and /release-plan command for ongoing release planning

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `skills`, `commands`, `release-planning`, `workflow`
- **Created:** 2026-04-15
- **Reporter:** Linoy
- **Feature area:** `skills/release-manager/SKILL.md`, `commands/release-plan.md`, `RELEASE.md`

## Description

The RELEASE.md roadmap drifted completely from reality between v5.5 and v5.9 — all four planned versions shipped with different ticket bundles than originally projected. The root cause: the roadmap is a static document updated manually by the documentation-specialist during Phase 7. When tickets are reprioritized, new tickets are filed, or batches are reorganized mid-cycle, nobody updates the release plan until the next Phase 7 run — by which point the drift has accumulated.

This ticket introduces a `release-manager` skill and `/release-plan` command that treats the release roadmap as a living artifact. The command can be invoked:
- **On-demand** — when tickets are added, reprioritized, or closed
- **During Phase 7** — as part of the documentation update (extends ISS-031's artifact registry)
- **As a standalone check** — to audit roadmap consistency against the backlog

### What this is NOT

This is not a replacement for ISS-030 (semver policy). ISS-030 establishes the rules for choosing major/minor/patch and the backlog metadata conventions. This ticket builds the operational tooling that applies those rules to keep RELEASE.md current.

## Acceptance Criteria

- [ ] **AC1 (Release-manager skill):** A new `skills/release-manager/SKILL.md` defines the release planning methodology: how to read the backlog's batch execution plan, map batches to version numbers, determine what's shipped vs planned, and update the roadmap table in RELEASE.md.
- [ ] **AC2 (Sibling reference files):** The skill uses the registry pattern (ISS-031 convention) with sibling reference files for discrete operations:
  - `roadmap-update.md` — rules for updating the roadmap table when tickets ship, are added, or are reprioritized
  - `consistency-check.md` — rules for auditing RELEASE.md against backlog.md, CHANGELOG.md, and release-notes/ for version mismatches, missing entries, or stale projections
- [ ] **AC3 (/release-plan command):** A new `commands/release-plan.md` invokes the release-manager skill. Accepts subcommands:
  - `/release-plan update` — reads the current backlog batch plan and updates RELEASE.md's roadmap table (shipped versions from CHANGELOG.md dates, planned versions from backlog batches)
  - `/release-plan check` — audits consistency between RELEASE.md, backlog.md, CHANGELOG.md, and release-notes/ without modifying files; reports discrepancies
- [ ] **AC4 (Backlog-driven version projection):** The skill reads the backlog's batch execution plan and execution waves to project forward versions. The mapping is: one minor version per batch (Batch 2.75 → next minor, Batch 3 → minor after that, etc.). Wave-only tickets without a batch assignment get a `future` projection.
- [ ] **AC5 (Shipped version detection):** The skill reads CHANGELOG.md to determine which versions have shipped (have dated entries) and marks them in the roadmap. It reads release-notes/ filenames to cross-check version numbers match.
- [ ] **AC6 (New ticket integration):** When a new ticket is filed (via backlog management), the skill can be invoked to determine which planned version the ticket falls into based on its batch assignment and update the roadmap accordingly.
- [ ] **AC7 (Reprioritization handling):** When tickets are moved between batches or waves, `/release-plan update` adjusts the roadmap: tickets moved earlier may pull into a sooner version, tickets deferred may shift to a later one.
- [ ] **AC8 (Phase 7 integration):** The `/document` command's artifact registry (ISS-031) includes RELEASE.md with a pointer to the release-manager skill's `roadmap-update.md` reference file, so Phase 7 automatically invokes the release plan update.
- [ ] **AC9 (ISS-030 dependency):** If ISS-030 (semver policy) has landed, the skill uses its semver metadata from tickets to determine version numbers. If ISS-030 has not landed, the skill uses the manual heuristic already documented in RELEASE.md (one minor per batch).
- [ ] **AC10 (Consistency report format):** `/release-plan check` outputs a structured report:
  - Versions in CHANGELOG.md but not in RELEASE.md roadmap (missing shipped entries)
  - Versions in RELEASE.md roadmap marked "Shipped" but not in CHANGELOG.md (phantom entries)
  - Tickets in a roadmap version that are not in the corresponding backlog batch
  - Release note files whose version doesn't match their content header
  - Roadmap "Planned" versions whose tickets have all shipped (stale projections)
- [ ] **AC11 (No regression):** Existing Phase 7 behavior unchanged. The release-plan command is additive.
- [ ] **AC12 (Model tier):** The skill runs on Haiku — this is mechanical reading and table manipulation, not complex reasoning.

## Notes

### Evidence: the v5.5-v5.9 drift

The RELEASE.md roadmap planned at v5.5:
- 5.6.0 = ISS-036, ISS-027, ISS-029 → actual: ISS-027, ISS-029, ISS-042
- 5.7.0 = ISS-001, ISS-006, ISS-037 → actual: ISS-036
- 5.8.0 = ISS-028, ISS-032, ISS-031 → actual: ISS-039
- 5.9.0 = ISS-007, ISS-008, ISS-030 → actual: ISS-008

Every single version shipped different tickets than planned. The roadmap was only corrected manually during a documentation catch-up session, not as part of the pipeline.

### Design considerations

- **Skill registry pattern:** Follows ISS-031 convention — registry table in SKILL.md + sibling reference files. The release-manager skill is loaded on demand by `/release-plan` or by `/document` via the artifact registry.
- **Read-only by default:** `/release-plan check` is read-only. `/release-plan update` writes to RELEASE.md only. Neither modifies backlog.md, CHANGELOG.md, or release-notes/ — those are owned by their respective skills/commands.
- **Incremental adoption:** The skill works with or without ISS-030's semver metadata. Without it, it uses the manual "one minor per batch" heuristic. With it, it reads ticket-level semver tags for more precise version assignment.

### Relationship to other tickets

- **ISS-030** — establishes semver policy and backlog metadata. ISS-048 is the operational tooling that applies those rules. ISS-048 works without ISS-030 (using manual heuristics) but is better with it.
- **ISS-031** — establishes the doc artifact registry pattern. ISS-048's Phase 7 integration (AC8) uses ISS-031's registry to trigger roadmap updates.
- **ISS-035** — captures ticket IDs in PRDs. Relevant because ticket IDs are the join key between backlog batches and roadmap versions.
- **ISS-028** — ticket-aware feature selection. If ISS-028 lands, `/release-plan` can use its ticket resolution to auto-detect which batch a new ticket belongs to.
