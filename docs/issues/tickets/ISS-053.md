# ISS-053: Adversarial review council with iterative resolution for authoring phases

- **Type:** Feature
- **Priority:** P1 — High
- **Label:** `framework`, `multi-agent`, `review-workflow`, `pipeline`, `council`
- **Created:** 2026-04-15
- **Reporter:** Linoy
- **Feature area:** Commands (specify, architect, test-design, implement), skills, project config
- **Supersedes:** ISS-012

## Description

The pipeline currently treats adversarial review (Codex) as a manual, loosely-coupled step — the operator runs a Codex review, reads the feedback, re-runs the authoring command, and hopes the feedback was fully addressed. There is no structured loop, no multi-reviewer consensus, no auto-trigger capability, and no per-command control over whether review is blocking or advisory.

This ticket introduces an **adversarial review council** pattern inspired by the LLM Council approach: multiple adversarial agents review the same artifact, the authoring agent addresses all feedback, and the adversarial agents re-review until the artifact passes or a configured stop condition is reached.

### Key capabilities

1. **Council composition** — a project defines a default set of adversarial reviewers (e.g., Codex + Gemini) at the project level, with per-command override for phases that need specialized reviewers.

2. **Iterative resolution loop** — authoring commands (Phases 1-3, 5) accept a review configuration that triggers adversarial review after the phase produces its artifact. The author addresses feedback, the reviewer re-reviews, and this loops until clean or a max iteration cap is reached.

3. **Auto-trigger with manual fallback** — if the adversarial agent is accessible via API (Claude instances, Gemini, etc.), the command auto-triggers the review. For agents that require separate tooling (Codex CLI), the command formalizes a structured manual handoff loop with iteration tracking.

4. **Configurable blocking** — each command can be configured as blocking (won't write handoff until review passes or cap reached), advisory (surfaces feedback but operator decides), or off. Configuration lives in project config with per-command override.

5. **Multi-model consensus** — when multiple reviewers participate, the command aggregates findings and requires consensus (configurable: unanimous, majority, or any-one-blocks) before advancing.

### Workflow example

```
/specify "Add user auth flow" --council

# Command writes PRD, then:
# 1. Triggers Codex PRD review (auto or manual depending on config)
# 2. Triggers Gemini PRD review (if configured)
# 3. Aggregates findings
# 4. If blocking findings exist:
#    - Surfaces findings to operator/author
#    - Author addresses findings
#    - Re-triggers reviews on updated artifact
#    - Loops until clean or max iterations (e.g., 3)
# 5. If max iterations reached: operator must explicitly override or defer
# 6. Records review rounds in feature artifact
# 7. Writes handoff only after review passes (if blocking) or with findings noted (if advisory)
```

## Acceptance Criteria

### Council configuration

- [ ] **AC1 (Project-level council):** A project defines its default adversarial review council in a structured section of CLAUDE.md or project config. The council specifies: reviewer names, invocation method (API auto-trigger vs manual), and default blocking mode.

- [ ] **AC2 (Per-command override):** Individual commands can override the project-level council — adding, removing, or replacing reviewers, or changing the blocking mode for that phase.

- [ ] **AC3 (Reviewer registry):** Each adversarial reviewer is defined with: name, invocation method (API endpoint, CLI command, or manual), review prompt/template path, expected output format, and capability profile (which artifact types it can review).

### Iterative resolution loop

- [ ] **AC4 (Loop mechanics):** Authoring commands (specify, architect, test-design, implement) support a review loop that: (a) triggers configured reviewers after artifact production, (b) surfaces findings, (c) allows the author to address findings, (d) re-triggers review on the updated artifact, (e) repeats until clean or max iterations.

- [ ] **AC5 (Iteration tracking):** Each review round records: reviewer name, timestamp, findings (with severity), and author's response (addressed, deferred, disputed). This history is written to a predictable feature-local artifact under `docs/features/<feature>/`.

- [ ] **AC6 (Finding classification):** Findings must distinguish at minimum: `test-failure` (reproducible by running a test command) vs `code-inspection` (latent defect identified by reading code). The author's resolution workflow handles each type differently: test-failures are reproduced by running the cited command; code-inspection findings require writing a new failing test before fixing. (Carried forward from ISS-012.)

- [ ] **AC7 (Stop conditions):** The loop has configurable stop conditions: max iteration cap (default: 3), convergence detection (no new findings), and explicit operator override. At cap, the command surfaces remaining findings and requires the operator to explicitly continue, defer, or override.

### Auto-trigger and manual fallback

- [ ] **AC8 (Auto-trigger):** For reviewers with API access (Claude instances via Anthropic API, Gemini via Google API, or any model accessible via a configured endpoint), the command auto-triggers the review, waits for results, and ingests findings without operator intervention.

- [ ] **AC9 (Manual fallback):** For reviewers that require separate tooling (e.g., Codex CLI, manual prompt-paste), the command: (a) writes a structured review request to a known location, (b) pauses and instructs the operator to run the review externally, (c) waits for the review output file to appear at the expected path, (d) ingests findings and continues the loop.

- [ ] **AC10 (Invocation abstraction):** The difference between auto-trigger and manual is abstracted behind a reviewer interface. Commands don't know or care whether a reviewer is API-driven or manual — they call the same interface and get findings back.

### Blocking and advisory modes

- [ ] **AC11 (Blocking mode):** When a command is configured as blocking, it will not write the phase handoff until all reviewers report clean or the operator explicitly overrides. Blocking findings that remain unresolved at max iterations require explicit operator action.

- [ ] **AC12 (Advisory mode):** When configured as advisory, the command surfaces findings and records them in the review history, but writes the handoff and advances the pipeline. Unresolved findings are carried in the handoff's `known_risks` field.

- [ ] **AC13 (Off mode):** When configured as off, no adversarial review is triggered. The command behaves as it does today.

### Multi-model consensus

- [ ] **AC14 (Consensus rules):** When multiple reviewers participate, the command supports configurable consensus: (a) unanimous — all reviewers must be clean, (b) majority — more than half must be clean, (c) any-one-blocks — any reviewer with blocking findings prevents advancement. Default: any-one-blocks.

- [ ] **AC15 (Finding aggregation):** When multiple reviewers report findings on the same artifact, findings are aggregated with reviewer attribution. Duplicate findings (same issue from multiple reviewers) are grouped and noted as corroborated.

### Pipeline documentation and tests

- [ ] **AC16 (Command arguments):** Authoring commands accept `--council` (use project default), `--review-with <reviewer>` (single reviewer override), or `--no-review` (skip). The argument syntax is documented in each command's usage section.

- [ ] **AC17 (Pipeline docs updated):** CLAUDE.md, README, and QUICKSTART reflect the new review council capability, including configuration format, invocation syntax, and the blocking/advisory/off modes.

- [ ] **AC18 (Contract tests):** Contract tests verify: (a) council configuration section exists in the expected location, (b) authoring commands reference the review council skill, (c) review history artifacts follow the expected format.

- [ ] **AC19 (No regression):** Existing pipeline behavior is unchanged when council is not configured or `--no-review` is passed. All existing tests pass.

### Design considerations

- **LLM Council inspiration:** The multi-model consensus pattern draws from the LLM Council approach — multiple models independently review the same artifact, findings are aggregated, and consensus determines passage. This is more robust than single-reviewer loops because different models have different blindspots (as demonstrated by the Codex-catches-what-Claude-misses pattern throughout this project).
- **Token budget impact:** Auto-triggered reviews add token cost. The ticket should define how review tokens are tracked (extend the existing `.claude/token-usage.jsonl` tracking) and whether review tokens count against the phase budget or are tracked separately.
- **Scope control:** The iterative loop must not become a scope expansion vector. The author should address *existing* findings, not add new features. This complements ISS-044 (rework scope lock).
- **Gradual rollout:** Projects can start with `--council` on just `/implement` and expand to other phases as confidence grows. The off-by-default design supports this.

  artifact

That pattern captures a small piece of AC5's iteration tracking ("author's response")

modes. It is a valid incremental workflow improvement, but it should be treated as
