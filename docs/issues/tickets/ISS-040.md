# ISS-040: checkpoint.js detectPhase() should recognize .js and .mjs test files

- **Type:** Bug
- **Priority:** P2 — Medium
- **Label:** `pipeline`, `testing`
- **Created:** 2026-04-13
- **Reporter:** code-reviewer agent (RCA from review-hardening Phase 6)
- **Feature area:** `hooks/checkpoint.js`

## Description

`detectPhase()` in `checkpoint.js` hard-codes `.test.ts` and `.spec.ts` as the patterns for detecting Phase 3 (test design) completion. In a project using `.test.js` files (including this repo's newer feature suites), `detectPhase()` misclassifies Phase 3 as Phase 2 (`ARCHITECT complete`).

Codex reproduced this during the review-hardening code review: in a temp project containing `tests/contracts/review-hardening.test.js`, `detectPhase()` returned phase 2 instead of phase 3.

## Acceptance Criteria

- [ ] **AC1:** `detectPhase()` recognizes `.test.js`, `.spec.js`, `.test.mjs`, and `.spec.mjs` in addition to `.test.ts` and `.spec.ts`.
- [ ] **AC2:** A test case with a `.test.js` file in the fixture confirms correct phase detection.
- [ ] **AC3:** The pattern uses a stack-agnostic approach (e.g., glob `*.{test,spec}.{ts,js,mjs}`) consistent with the "adapt to your stack" convention.

## Technical Analysis

The hard-coded pattern in `checkpoint.js` `detectPhase()` likely uses a glob or regex matching only `.test.ts`/`.spec.ts`. Extending it to include `.js` and `.mjs` extensions is a minimal change. The existing tests should be updated to cover the `.js` case.
