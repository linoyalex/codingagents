# ISS-057: Allow /review to target specific phase outputs without advancing the pipeline

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `pipeline`, `review-workflow`
- **Created:** 2026-04-16
- **Reporter:** Linoy
- **Feature area:** `commands/review.md`, `skills/code-review/SKILL.md`, `hooks/checkpoint.js`

## Description

The `/review` command is currently hardcoded as Phase 6. It writes a Phase 6 handoff, reads `git diff`, and expects to review implementation output. But there are legitimate cases where an operator wants a formal Claude code-review-quality review of intermediate phase outputs — most commonly Phase 3 (test design) — without advancing the pipeline past Phase 6.

During the qa-test-quality feature (Batch 2.75), the test design output was reviewed using an ad-hoc process (`review-claude-test-design-qa-test-quality.md`) because running `/review` formally would have:
1. Written a Phase 6 handoff, overwriting the Phase 3 handoff
2. Confused the pipeline about which phase was active
3. Applied implementation-review methodology (diff-based) instead of test-design-review methodology (spec-based)

The command needs a way to specify which phase's output to review, applying the appropriate review methodology for that phase, without clobbering the pipeline state.

## Acceptance Criteria

- [ ] **AC1 (Phase argument):** `/review <feature> --phase <N>` reviews the output of phase N using methodology appropriate for that phase. Without `--phase`, defaults to Phase 6 (current behavior).

- [ ] **AC2 (Phase-appropriate methodology):** Each reviewable phase has a defined review approach:
  - Phase 1 (specify): Review PRD against ticket ACs, check completeness
  - Phase 2 (architect): Review architecture against PRD, check decision confidence and rollback
  - Phase 3 (test-design): Review tests against PRD + architecture, check AC coverage and anchor quality
  - Phase 5 (implement): Review diff against spec (current Phase 6 behavior)

- [ ] **AC3 (No pipeline state clobbering):** When reviewing a phase other than Phase 5/6, the command does NOT write a Phase 6 handoff. Instead, it either: (a) writes the review artifact without modifying handoff.json, or (b) writes a temporary review checkpoint that restores the previous handoff state after review completes.

- [ ] **AC4 (Review artifact naming):** Review artifacts for non-Phase-6 reviews use a phase-qualified naming convention (e.g., `review-claude-test-design-<feature>.md`) so they don't overwrite the Phase 6 `review.md`.

- [ ] **AC5 (Diff target for non-implementation phases):** For Phase 3 reviews, the diff target is the test files committed in the RED state, not the full branch diff. The command determines the appropriate diff scope based on the phase.

- [ ] **AC6 (Backward compatible):** `/review <feature>` without `--phase` behaves identically to today (Phase 6, writes handoff, reviews implementation diff). No existing behavior changes.

- [ ] **AC7 (Contract tests):** Contract tests verify: (a) the `--phase` argument is accepted, (b) review artifact naming follows the phase-qualified convention, (c) handoff.json is not clobbered when reviewing non-Phase-6 output.

## Notes

### Root cause

The pipeline assumes reviews only happen at Phase 6. But the actual workflow has reviews at multiple points:
- Codex reviews at Phases 1, 2, 3, 5 (already operational, but ad-hoc)
- Claude self-review at Phase 3 (happened during qa-test-quality, ad-hoc)
- The `/review` command only supports Phase 6

### Relationship to other tickets

- **ISS-053** — adversarial review council. ISS-053 automates multi-agent review loops at all authoring phases. ISS-057 gives the formal `/review` command the ability to target any phase, which ISS-053 would use as the underlying mechanism for Claude-side reviews in the council.
- **ISS-006** — review-history.md. Phase-qualified review artifacts (AC4) feed naturally into a review history that tracks reviews across phases.
- **ISS-037** — additive review artifacts. Multiple phase-qualified reviews for the same feature should be additive, not overwriting.

### Phase 3 methodology detail (from Codex test-design reviewer)

AC2 defines Phase 3 methodology as "Review tests against PRD + architecture, check AC coverage and anchor quality." The Codex test-design reviewer (`codex/reviewers/review-test-design.md`) provides a more specific checklist that should inform the Phase 3 review methodology:

**Review priorities** (in order):
1. Acceptance criteria coverage — every AC has a corresponding test
2. Boundary and negative case coverage — not just happy path
3. Permission and authorization coverage — unauthorized/forbidden paths
4. Test falsifiability — tests that can't fail are worthless
5. Determinism and maintainability — no flaky or implementation-coupled tests

**Specific anti-patterns to check:**
- ACs with no corresponding test
- Happy-path-only coverage (missing empty, loading, error, edge states)
- Tests that assert implementation details rather than behavior
- Tests that can pass trivially (e.g., asserting a string exists in a file without checking its meaning)
- Missing contract tests at API boundaries
- Missing unauthorized or forbidden-path checks

**Output should include a coverage map** showing which ACs are well-covered vs uncovered/weakly-covered, plus highest-value next tests to add. This aligns with the structured output in `review-test-design.md`.

The Phase 3 review should load or align with the existing Codex reviewer methodology rather than inventing a parallel one. ISS-060 and ISS-061 follow the same pattern for Phases 1 and 2 respectively.

### Design consideration

The simplest implementation may be to have `/review --phase 3` load the test-design review methodology (a subset of the code-review skill focused on spec-coverage rather than diff-correctness) rather than trying to make one review methodology work for all phases. The architect should decide whether this is one polymorphic command or separate review commands per phase.
