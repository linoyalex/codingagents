# ISS-033: Require reviewers to verify against the source specification or ticket

- **Type:** Feature
- **Priority:** P2 — Medium
- **Label:** `framework`, `review-quality`, `pipeline`, `workflow`, `documentation`
- **Created:** 2026-04-12
- **Reporter:** Linoy
- **Feature area:** Reviewer prompts/skills, handoff contract, review workflow

## Description

Review quality is weaker when reviewers only read the latest artifact or developer framing
instead of checking the work against the original product intent. In practice, that creates
two failure modes:

1. **Spec drift goes unnoticed.** A PRD or review artifact can drift from the original
   ticket, and the review still looks internally consistent because the reviewer never
   checks the source requirement.
2. **Bugfixes are reviewed against implementation claims, not the bug report.** When
   there is no PRD, reviewers can still miss whether the fix actually answers the ticket
   because the original issue file is not carried forward as a first-class review input.

We should make the source specification explicit and reviewable across the pipeline:

- when a feature has a PRD, reviewers should verify against that PRD and, when relevant,
  the originating ticket
- when a feature is ticket-driven without a PRD, reviewers should verify against the
  ticket or another declared source specification
- the handoff contract should carry a machine-readable pointer to the source
  specification/ticket so later phases do not have to guess

## Acceptance Criteria

- [x] **AC1 (Claude review contract):** Claude reviewer instructions are updated so the
      reviewer must read and verify against the source specification before relying on
      developer handoff claims or downstream artifacts. For code review, this means PRD
      first when it exists; otherwise the originating ticket or declared source spec.
- [x] **AC2 (Codex review contract):** Codex reviewer prompts are updated so each review
      type checks the work against the source specification or ticket, not just the latest
      generated artifact. This should cover at least code review and PRD review, and any
      other review prompts whose current scope can drift from original intent.
- [x] **AC3 (Handoff source-spec metadata):** The handoff contract carries an explicit
      pointer to the source requirement, such as a PRD path, ticket path, or equivalent
      source-spec field, so later phases can locate the authoritative product intent
      without guessing.
- [x] **AC4 (Bugfix/no-PRD path):** The review workflow defines what reviewers use when
      no PRD exists, including ticket-driven bugfixes. The fallback source must be
      explicit rather than inferred from branch name or recent chat history.
- [x] **AC5 (Ambiguity handling):** If the source specification/ticket is missing,
      ambiguous, or conflicts with downstream artifacts, reviewer instructions tell the
      agent to stop and surface the mismatch instead of approving based only on local
      consistency.
- [x] **AC6 (Regression protection):** Deterministic tests or contract checks protect the
      chosen handoff/source-spec format and the updated reviewer instructions from
      regressing silently.
- [x] **AC7 (Process docs):** Shared review-process documentation is updated so human
      operators understand that reviewers are expected to verify against the original
      intent, not just the latest generated artifact.

## Notes

### Evidence

- The `artifact-timestamps` PRD drifted from `ISS-026` in ways a PRD-only review could
  miss unless the reviewer also checked the original ticket scope.
- Several review-quality tickets already address adjacent problems such as reviewer
  anchoring, adversarial review, and Codex review-method hardening. This ticket closes a
  separate gap: preserving the original product intent as a first-class review input.

### Relationship to other tickets

- ISS-024 improves reviewer independence and boundary tracing. ISS-033 complements it by
  making the original requirement a mandatory review input.
- ISS-012 becomes more credible when stage-matched Codex reviews are anchored to a
  declared source specification rather than only the latest artifact.
- ISS-028 may eventually make ticket selection and active-feature context easier to carry,
  but this ticket is specifically about review correctness and handoff contract fidelity.
