# ISS-011: Add shell script wrappers for Codex reviewer workflows

- **Type:** Feature
- **Priority:** P3 — Low
- **Label:** `developer-experience`, `tooling`, `review`
- **Created:** 2026-04-06
- **Reporter:** Human
- **Feature area:** `codex/`, `scripts/`, review workflow

## Description

The repository already has a lightweight Codex review lane documented in `codex/README.md`
and `codex/fresh-context-playbook.md`, with reviewer prompts in `codex/reviewers/`. That
workflow is useful, but it is still manual: the user has to remember which reviewer file
to use, reconstruct the recommended prompt, pick an output filename, and optionally log
usage afterward.

That friction makes Codex reviews harder to run consistently, especially for quick
second-pass reviews where the goal is to keep context fresh and token usage low. A small
set of shell script wrappers would turn the documented process into a one-command
workflow without changing the underlying review model.

## Proposed Outcome

Provide convenient wrapper scripts for the supported Codex reviewer flows, such as:

- Code review against `main...HEAD`
- Architecture review for `docs/features/<feature>/architecture.md`
- Test-design review for `tests/contracts/` and `tests/e2e/`
- PRD review for `docs/features/<feature>/prd.md`

The wrappers should encode the existing recommended prompts, derive predictable output
paths under `codex/reviews/`, and make it easy to optionally log usage with
`codex/log-usage.sh`.

## Acceptance Criteria

- A developer can run a Codex reviewer flow with a short shell command instead of
  manually reconstructing the prompt from `codex/README.md`
- Supported reviewer types are available through wrappers or a generic wrapper with
  clear subcommands
- Code review defaults to the repository's documented diff scope (`main...HEAD`) with
  an option to override the base branch when needed
- The wrapper derives or accepts a feature slug and writes output to a predictable file
  in `codex/reviews/`
- The wrapper includes the existing fresh-context guidance in the generated prompt,
  including use of `.claude/handoff.json` only for acceptance criteria and risk context
  when present
- The wrapper can optionally call `codex/log-usage.sh` after the review metadata is
  provided
- Errors are actionable when required inputs are missing (for example, feature slug,
  expected docs, or a missing diff target branch)
- `codex/README.md` is updated to document the wrapper commands as the preferred way to
  kick off Codex reviews

## Technical Considerations

- Decide whether to implement one generic entrypoint such as
  `codex/run-review.sh <type> <feature>` with thin aliases, or separate scripts such as
  `codex/review-code.sh`
- Keep the implementation shell-friendly and consistent with existing repository
  scripts, which already use Bash
- Preserve the current design rule that Codex reviews are advisory only and do not
  become a parallel gated pipeline
- If the wrapper launches Codex directly, it should still make the exact prompt and
  target output path visible so the workflow remains transparent and debuggable
- Usage logging may need a prompt or flag-based flow because token counts and duration
  are not always known until the review finishes

## Notes

This request is about convenience and consistency, not about changing review policy. The
goal is to make the existing Codex reviewer playbook easy enough to use that it becomes
a normal part of high-value reviews, especially `review-code`, which `codex/README.md`
already identifies as the best first checkpoint.
